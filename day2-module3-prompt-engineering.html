Prompt Engineering for Code

Outcome: Write prompts that produce correct, idiomatic, production-ready code on the first or second attempt — by applying structured anatomy, XML context boundaries, context window discipline, iterative patterns, and role specifications. Build a team prompt library that compounds this skill across your entire engineering organization.

01

Anatomy of a High-Quality Code Prompt

Most developers underperform with Claude Code not because Claude is limited, but because their prompts are. A vague prompt forces Claude to make assumptions — about your stack, your style conventions, your security posture, your error handling philosophy. Every assumption Claude makes is a potential mismatch with your codebase. The fix is not trial-and-error iteration; it is a structured prompt that eliminates ambiguity before Claude writes a single line of code.

A high-quality code prompt has five elements. None of them are optional. Together they give Claude everything it needs to produce code you can actually use.

Element 01
Role
Who Claude is for this task. Determines vocabulary, reasoning depth, and tradeoff priorities. Without a role, Claude defaults to a generalist stance that rarely matches your team's standards.
Element 02
Context
What exists in the codebase that Claude must respect. Without context, Claude generates generic code that won't fit your patterns, naming conventions, or dependency choices.
Element 03
Task
The precise, bounded thing Claude must do. Tasks scoped too broadly produce sprawling output. Tasks scoped too narrowly miss integration concerns. One task per prompt.
Element 04
Constraints
What Claude must not do. Without explicit constraints, Claude adds features you didn't ask for, picks libraries you don't use, and introduces patterns that violate your architecture rules.
Element 05
Output Format
How Claude should structure its response. A full file? A diff? A function only? With tests included? Inline comments? If you don't specify, Claude will guess — and the guess won't match your review workflow.

Side-by-Side: Weak vs Strong Prompt

The task is identical in both prompts: implement a rate limiter. The outputs Claude produces are not.

✗ Weak Prompt 
Implement a rate limiter in Python.
✓ Strong Prompt 
You are a senior Python backend engineer at a
B2B SaaS company building on FastAPI.

Context: Our API uses FastAPI 0.111 with
dependency injection. Redis (via redis-py asyncio
client) is already in scope as our cache layer.
Rate limit state must survive pod restarts.
We target 99.9% uptime so the limiter must fail
open (allow requests) if Redis is unavailable.

Task: Implement a sliding-window rate limiter as
a FastAPI dependency. It should accept requests
per window and window_seconds as parameters.

Constraints:
- Use ONLY redis-py asyncio — no aioredis
- No global state; pass the Redis client via DI
- Do NOT add a middleware; it must be a Depends()
- Raise HTTP 429 with Retry-After header
- No external libraries beyond fastapi, redis-py

Output: A single self-contained Python module
`app/dependencies/rate_limiter.py` with the
class, the dependency factory, and inline type
hints. Include a usage example as a docstring.
⚠️
The Hidden Cost of Weak Prompts
A weak prompt doesn't just produce worse code on the first attempt — it forces you into a correction loop. Each correction turn costs context tokens, time, and the mental overhead of reviewing half-wrong code. A well-structured prompt that takes 90 seconds to write routinely saves 10–20 minutes of back-and-forth. At team scale, this is a meaningful engineering productivity multiplier.

What Claude Does Without Each Element

  • Without Role: Claude assumes a generic software engineer persona. It may choose Flask over FastAPI, synchronous code over async, or suggest libraries your team has explicitly excluded. The code will be "correct Python" but not your Python.
  • Without Context: Claude invents its own data models, import paths, and error handling conventions. Integrating this output requires significant rework because it doesn't connect to anything real in your codebase.
  • Without Task Boundary: Claude may implement the rate limiter and add a caching decorator, refactor adjacent code it notices in context, or suggest a follow-up architecture change. All unsolicited and all requiring review.
  • Without Constraints: Claude defaults to whatever it considers best practice. If your team has decided not to use a pattern Claude considers idiomatic (for reasons of operational simplicity, team familiarity, or vendor lock-in), Claude will use it anyway unless you say so.
  • Without Output Format: Claude returns whatever it deems most helpful — sometimes a full explanation plus code, sometimes code only, sometimes a refactored version of existing code it finds in context. Your review workflow deserves predictability.

XML Prompting for Structured Code Tasks

For simple tasks — "explain this function", "add a docstring" — a well-written plain-text prompt is sufficient. But as tasks grow in complexity, the boundaries between instruction, your existing code, reference examples, and desired output become ambiguous. Claude is a language model: it parses structure from text, and ambiguous structure produces ambiguous output.

XML tags solve this. They create unambiguous, machine-readable context boundaries. Claude has been specifically trained to respect them. This is not a workaround or a hack — it is the intended authoring surface for complex, multi-constraint tasks.

The Core Tag Vocabulary

  • <context> — Background information about your system, architecture, or conventions that Claude needs to reason correctly. Not the task itself.
  • <task> — The precise action Claude must take. Single, bounded, actionable.
  • <constraints> — Explicit limits. What Claude must not do, what libraries it must not introduce, what patterns it must follow.
  • <existing_code> — Code that already exists in your codebase that Claude must read, respect, or modify. Using this tag makes it unambiguous that this is yours, not a reference.
  • <examples> — Reference examples showing the style, pattern, or convention you want Claude to follow. Distinct from existing code.
  • <output_format> — The exact format Claude should return: file path, language, structure, whether to include tests or explanations.

Complete Example: Add Validation to a FastAPI Endpoint

The following is a production-quality XML prompt for adding input validation to an existing endpoint. Notice how each tag serves a distinct purpose and eliminates a whole class of assumption.

XML prompt — FastAPI input validation 
# Send this as the user turn in your Claude Code session

<context>
We run a multi-tenant B2B API built on FastAPI 0.111, Pydantic v2,
and Python 3.12. Authentication is handled upstream via a gateway
that injects X-Tenant-ID and X-User-ID headers — we do NOT
re-authenticate inside endpoint handlers.

Our validation convention is Pydantic model validators with
model_validator(mode="before") for cross-field checks and
field_validator for single-field checks. We raise HTTPException
directly from validators rather than using a custom exception handler.

We use snake_case for all Python identifiers and kebab-case for
HTTP header names. Responses always include a "request_id" field
sourced from the request state.
</context>

<existing_code file="app/routers/payments.py">
from fastapi import APIRouter, Depends, Request
from app.dependencies.auth import require_tenant
from app.models.payments import PaymentRequest, PaymentResponse
from app.services.payment_service import PaymentService

router = APIRouter(prefix="/payments", tags=["payments"])

@router.post("/initiate", response_model=PaymentResponse)
async def initiate_payment(
    request: Request,
    body: dict,                       # TODO: replace with validated model
    tenant_id: str = Depends(require_tenant),
    svc: PaymentService = Depends(),
) -> PaymentResponse:
    result = await svc.initiate(tenant_id, body)
    return PaymentResponse(**result, request_id=request.state.request_id)
</existing_code>

<task>
Replace the untyped `body: dict` parameter with a Pydantic v2 model
`InitiatePaymentRequest` defined in `app/models/payments.py`.
The model must validate:
1. `amount` — positive Decimal, max 2 decimal places, max value 999_999.99
2. `currency` — ISO 4217 three-letter uppercase string (e.g. "USD", "EUR")
3. `recipient_account_id` — non-empty string, max 64 chars, alphanumeric + hyphens only
4. `idempotency_key` — UUID4 string; if absent, generate one server-side (do not require client to send it)
5. Cross-field: if currency is "USD", amount must be >= 1.00 (USD has a minimum charge floor)
</task>

<constraints>
  <style>
    - Use Pydantic v2 syntax only (model_validator, field_validator, not @validator)
    - Do not import or use any third-party validation library (e.g. no voluptuous, marshmallow)
    - Keep the router file changes minimal — only swap body: dict for the new model type
  </style>
  <security>
    - Do not log or expose the full amount or recipient_account_id in error messages
    - Validation error responses must not leak internal field names beyond what Pydantic provides by default
  </security>
  <behaviour>
    - Raise HTTP 422 for validation failures (FastAPI default — do not override)
    - Raise HTTP 400 explicitly only for the USD minimum floor violation with message:
      "USD payments require a minimum amount of 1.00"
  </behaviour>
</constraints>

<output_format>
Return two code blocks clearly labelled:
1. `app/models/payments.py` — the full updated models file with InitiatePaymentRequest added
2. `app/routers/payments.py` — the updated router with only the type annotation changed

No explanatory prose. Inline comments only where the logic is non-obvious.
</output_format>

Plain-Text vs XML: Same Task, Different Output Quality

✗ Plain-Text Prompt 
Add validation to the initiate_payment
endpoint in our FastAPI app. Validate the
amount, currency, recipient, and add an
idempotency key.
Likely output problems:
  • Uses Pydantic v1 syntax (@validator) if Claude trained on older examples
  • Invents currency codes as an Enum, which breaks with new currencies
  • Adds a global exception handler you didn't ask for
  • Missing the USD minimum floor logic
  • May modify the router signature in ways that break the DI chain
✓ XML Prompt (above)
What XML boundaries give you:
  • Claude reads your existing code as constraint, not example to rewrite
  • Style constraints eliminate the Pydantic v1/v2 ambiguity entirely
  • Security constraints prevent over-verbose error messages in the model
  • Output format ensures you get two labelled blocks, not a narrative
  • Nested constraint tags let you group security vs style vs behaviour concerns

When XML Helps Most

  • Multi-step tasks: When you need Claude to do X, then Y, then Z in sequence, XML tags let each step be independently scoped without bleeding into each other.
  • Tasks with many constraints: When you have more than 3 explicit constraints, plain-text lists get lost. Nested XML groups related constraints so Claude can process them coherently.
  • Distinguishing your code from reference code: Use <existing_code> for what Claude must preserve and <examples> for patterns Claude should emulate. Without this distinction, Claude may modify your reference examples or treat existing code as optional.
  • Long context sessions: In a session with many files in context, XML tags in your prompt give Claude clear anchoring points to return to when it reasons about constraints.
💡
Nested Tags for Multi-Dimension Constraints
Group constraints by dimension rather than listing them flat. <constraints><security>...</security><style>...</style><behaviour>...</behaviour></constraints> lets Claude process security-related constraints as a coherent block before shifting to style. This mirrors how a human engineer mentally separates security requirements from coding style — and produces correspondingly more coherent output.

Context Window Management

Claude Code's 200K-token context window is large enough that most developers stop thinking about it — and that is a mistake. Context is not free storage. Every token you add to the context is a token Claude must attend over when generating its response. The relationship is not linear: adding irrelevant context actively degrades output quality because Claude's attention is diluted across content that doesn't bear on the task.

┌─────────────────────────────────────────────────────────────────────┐ │ CONTEXT WINDOW ECONOMICS │ │ │ │ 200K tokens total │ │ ┌────────────────────────────────────────────────────────────┐ │ │ │ Focused session │ │ │ │ ▓▓▓▓▓ Task-relevant files (~15K) │ │ │ │ ▒▒▒▒▒ CLAUDE.md conventions (~2K) │ │ │ │ ░░░░░ Conversation so far (~8K) │ │ │ │ │ │ │ │ Remaining for reasoning: ~175K tokens ✓ │ │ │ └────────────────────────────────────────────────────────────┘ │ │ │ │ ┌────────────────────────────────────────────────────────────┐ │ │ │ Bloated session │ │ │ │ ████████████████████ All src/ files (~80K) │ │ │ │ ████████████ All test/ files (~40K) │ │ │ │ ██████ Long uncleared conversation (~30K) │ │ │ │ ████ Config, lockfiles, migrations (~20K) │ │ │ │ │ │ │ │ Remaining for reasoning: ~30K tokens ✗ │ │ │ └────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────┘

The Context Tax

The context tax is the performance penalty you pay for every irrelevant token in your session. It manifests as: responses that ignore constraints buried late in a long context, code that doesn't match conventions because the relevant convention was 80K tokens earlier, and hallucinated function signatures because Claude is attending across too much noise to reliably locate the correct one.

The rule is simple: add the minimum context needed to answer the question correctly, and no more.

Practical Context Management Strategies

Strategy 1: Add Files Surgically, Not Broadly
Use /add for the 2–4 files most likely to contain the answer — not all files in the module. If you're fixing a bug in payment_service.py, add that file and its direct dependencies (payment_repository.py, models/payment.py). Do not add the entire services/ directory.
Strategy 2: Clear and Restart for New Sub-Tasks
Long sessions accumulate noise: early hypotheses Claude generated but discarded, files you added for a previous sub-task, partial implementations that were superseded. When you shift to a new, distinct sub-task, run /clear and start a fresh session with a focused prompt. The overhead of re-explaining context is almost always less than the quality penalty of continuing in a polluted window.
Strategy 3: Use CLAUDE.md to Replace Repeated Context
A well-written CLAUDE.md file at your project root is loaded automatically by Claude Code on every session. It replaces the need to add architecture files, explain naming conventions, or specify framework choices at the start of each prompt. Every convention you encode in CLAUDE.md is removed from the per-prompt context budget. Think of it as a persistent, zero-cost system prompt for your project.
Strategy 4: Rough Token Counting
Before a large session, estimate your context spend. A rough proxy: cat your_file.py | wc -c gives bytes; divide by 4 for an approximate token count. For a 200K window, you want to keep task-relevant file context under 40K tokens to leave headroom for the conversation and Claude's output budget. Python source files average 3–8K tokens per 100 lines depending on verbosity.
CLAUDE.md — project conventions replacing per-prompt context 
# CLAUDE.md — loaded automatically by Claude Code on every session

## Stack
- Python 3.12, FastAPI 0.111, Pydantic v2, SQLAlchemy 2.0 (async)
- PostgreSQL 16 via asyncpg driver
- Redis 7.x via redis-py asyncio client (never aioredis)
- pytest + pytest-asyncio for all tests; no unittest

## Conventions
- snake_case for Python identifiers; kebab-case for HTTP headers
- All service methods are async; no sync DB calls in request handlers
- Use Pydantic v2 model_validator / field_validator — never @validator
- Raise HTTPException directly; no custom exception handler middleware

## Architecture Rules
- No global state. All shared resources (DB pool, Redis, config) via DI
- Repository pattern: DB access only in `repositories/`; never in routers
- 3-layer: router → service → repository. Services never import from routers
- All external API calls go through `integrations/` with a circuit breaker

## Output Preferences
- Return complete files, not diffs, unless I ask for a diff
- Include type hints on all function signatures
- Do not add print() debug statements; use structlog for logging
- Tests use fixtures from `tests/conftest.py` — do not redefine them
✅
The CLAUDE.md Effect
A single well-maintained CLAUDE.md in your repository root is worth more than any individual prompt optimization. It encodes your team's hard-won conventions once, permanently, and applies them to every Claude Code session on the project without using any per-session context budget. Treat it as a first-class engineering artifact — review it in PRs, version it, and update it when your conventions evolve.

Iterative Prompting Patterns

The one-shot prompt — write the perfect prompt, get perfect code — is a tempting model but rarely the right one for complex tasks. For anything beyond a self-contained function, iterative prompting produces consistently better results: you validate assumptions at each step, catch integration issues early, and give Claude focused, narrowly-scoped tasks that are each individually easy to do correctly.

Iteration is not failure. It is the intended workflow for complex code generation. The question is not whether to iterate, but how to structure the iterations so each one moves you forward without polluting the context.

The Four Core Iterative Patterns

The Scaffolding Pattern
Generate in layers. Step 1: data models and interfaces only. Step 2: core business logic. Step 3: error handling and edge cases. Step 4: tests. Each step is small enough to review confidently before proceeding. You catch architecture mistakes after Step 1, not after Claude has implemented 400 lines of logic on a flawed foundation.
The Correction Pattern
When Claude's output is technically correct but architecturally wrong, name the specific violation rather than asking for a generic rewrite. "This implementation is correct but violates our no-global-state rule. Refactor to pass the Redis client explicitly via the function signature rather than accessing the module-level client." Claude responds to specific, actionable corrections far better than vague feedback.
The Elaboration Pattern
Start with the happy path, then expand incrementally. "Expand the error handling in the payment module to cover these 3 additional cases: (1) upstream timeout from the payment gateway, (2) duplicate idempotency key from a different tenant, (3) currency not supported by the recipient's bank." Elaboration prompts are precise, additive, and easy for Claude to scope correctly.
The Undo Pattern
When Claude has gone in a direction you don't want, say so explicitly and reason about the tradeoff. "Revert to the version before you added distributed caching — the operational complexity of cache invalidation isn't worth it for the read volume of this endpoint. Implement the simpler in-process LRU cache from Step 2." Claude understands product reasoning, not just technical corrections.

Full Example: 4-Step JWT Authentication Middleware

The following session builds a production JWT authentication middleware for FastAPI through four focused iterations. Each prompt is short, scoped, and builds on verified output from the previous step.

Step 1 — Scaffold: data models and interface 
# PROMPT STEP 1
# "Define the Pydantic models and the interface only — no implementation yet."

# Claude generates:
from pydantic import BaseModel, Field
from typing import Optional
import uuid

class TokenClaims(BaseModel):
    """Validated claims extracted from a verified JWT."""
    sub: str = Field(..., description="User ID (UUID4 string)")
    tenant_id: str = Field(..., description="Tenant ID")
    roles: list[str] = Field(default_factory=list)
    jti: str = Field(..., description="JWT ID — used for revocation checks")
    exp: int = Field(..., description="Expiry as Unix timestamp")

class AuthContext(BaseModel):
    """Injected into route handlers via Depends(require_auth)."""
    user_id: str
    tenant_id: str
    roles: list[str]
    token_id: str  # for audit logging

class JWTConfig(BaseModel):
    """Settings sourced from environment — injected via DI."""
    secret_key: str
    algorithm: str = "HS256"
    leeway_seconds: int = 30  # clock skew tolerance
Step 2 — Core logic: token verification 
# PROMPT STEP 2
# "Using the models from Step 1, implement the JWT verification function.
#  Use PyJWT. Do not add the FastAPI dependency yet — pure function only."

# Claude generates:
import jwt
from jwt.exceptions import ExpiredSignatureError, InvalidTokenError
from fastapi import HTTPException, status
from datetime import datetime, timezone

def verify_jwt_token(
    token: str,
    config: JWTConfig,
) -> TokenClaims:
    """
    Verify and decode a JWT. Raises HTTP 401 on any failure.
    Does NOT check revocation — that is the caller's responsibility.
    """
    try:
        payload = jwt.decode(
            token,
            config.secret_key,
            algorithms=[config.algorithm],
            leeway=config.leeway_seconds,
            options={"require": ["sub", "exp", "jti", "tenant_id"]},
        )
        return TokenClaims(**payload)
    except ExpiredSignatureError:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Token has expired",
            headers={"WWW-Authenticate": "Bearer"},
        )
    except InvalidTokenError as exc:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid token",
            headers={"WWW-Authenticate": "Bearer"},
        )
Step 3 — FastAPI dependency + revocation check 
# PROMPT STEP 3
# "Wrap verify_jwt_token in a FastAPI Depends() function named require_auth.
#  Add a Redis revocation check: if the jti exists in the 'revoked_tokens'
#  Redis set, raise 401. Fail open if Redis is unavailable — log a warning
#  via structlog but allow the request through."

# Claude generates:
import structlog
from fastapi import Depends, Request
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer
from redis.asyncio import Redis

logger = structlog.get_logger(__name__)
_bearer = HTTPBearer(auto_error=True)

async def _check_revocation(redis: Redis, jti: str) -> None:
    """Returns silently if token is valid. Raises 401 if revoked."""
    try:
        is_revoked = await redis.sismember("revoked_tokens", jti)
        if is_revoked:
            raise HTTPException(
                status_code=status.HTTP_401_UNAUTHORIZED,
                detail="Token has been revoked",
                headers={"WWW-Authenticate": "Bearer"},
            )
    except HTTPException:
        raise
    except Exception:
        logger.warning("redis_revocation_check_failed", jti=jti)
        # Fail open: Redis unavailable should not block legitimate requests

async def require_auth(
    request: Request,
    credentials: HTTPAuthorizationCredentials = Depends(_bearer),
    config: JWTConfig = Depends(get_jwt_config),
    redis: Redis = Depends(get_redis),
) -> AuthContext:
    claims = verify_jwt_token(credentials.credentials, config)
    await _check_revocation(redis, claims.jti)
    ctx = AuthContext(
        user_id=claims.sub,
        tenant_id=claims.tenant_id,
        roles=claims.roles,
        token_id=claims.jti,
    )
    request.state.auth = ctx
    return ctx
Step 4 — Tests 
# PROMPT STEP 4
# "Write pytest tests for verify_jwt_token and require_auth covering:
#  valid token, expired token, tampered signature, revoked jti,
#  Redis unavailable (should pass), missing required claims."

# Claude generates:
import pytest
import jwt
import time
from unittest.mock import AsyncMock, patch
from fastapi import HTTPException

VALID_CLAIMS = {
    "sub": "user-123",
    "tenant_id": "tenant-abc",
    "roles": ["admin"],
    "jti": "token-jti-001",
    "exp": int(time.time()) + 3600,
}
SECRET = "test-secret-key"
CONFIG = JWTConfig(secret_key=SECRET)

def _make_token(claims: dict, secret: str = SECRET) -> str:
    return jwt.encode(claims, secret, algorithm="HS256")

class TestVerifyJwtToken:
    def test_valid_token(self):
        token = _make_token(VALID_CLAIMS)
        claims = verify_jwt_token(token, CONFIG)
        assert claims.sub == "user-123"
        assert claims.tenant_id == "tenant-abc"

    def test_expired_token(self):
        expired = {**VALID_CLAIMS, "exp": int(time.time()) - 10}
        token = _make_token(expired)
        with pytest.raises(HTTPException) as exc:
            verify_jwt_token(token, CONFIG)
        assert exc.value.status_code == 401
        assert "expired" in exc.value.detail.lower()

    def test_tampered_signature(self):
        token = _make_token(VALID_CLAIMS, secret="wrong-secret")
        with pytest.raises(HTTPException) as exc:
            verify_jwt_token(token, CONFIG)
        assert exc.value.status_code == 401

    def test_missing_required_claim(self):
        no_jti = {k: v for k, v in VALID_CLAIMS.items() if k != "jti"}
        token = _make_token(no_jti)
        with pytest.raises(HTTPException) as exc:
            verify_jwt_token(token, CONFIG)
        assert exc.value.status_code == 401

class TestRevocationCheck:
    @pytest.mark.asyncio
    async def test_revoked_token_raises_401(self):
        redis = AsyncMock()
        redis.sismember.return_value = True
        with pytest.raises(HTTPException) as exc:
            await _check_revocation(redis, "token-jti-001")
        assert exc.value.status_code == 401

    @pytest.mark.asyncio
    async def test_redis_unavailable_allows_request(self):
        redis = AsyncMock()
        redis.sismember.side_effect = ConnectionError("Redis down")
        # Should NOT raise — fail open
        await _check_revocation(redis, "token-jti-001")
💡
Review Before Each Step
The scaffolding pattern only works if you actually review Claude's output before issuing the next prompt. A 2-minute review after Step 1 catches a model design flaw before 200 lines of logic are built on it. The goal is not to eliminate iteration — it is to make each iteration cheap and deliberate rather than expensive and reactive.

Role & Persona Specification

Specifying a role is not a magic trick. It is a signal that calibrates Claude's reasoning prior: what tradeoffs to foreground, what vocabulary to use, what failure modes to worry about, and how much depth to apply to each concern. A "senior Python engineer at a fintech company" and a "junior developer learning FastAPI" will receive meaningfully different implementations of the same function — and for good reason. You want the former.

What Role Specification Changes

Without Role
Code review prompt: "Review this authentication handler for issues."
  • Generic feedback: "consider adding error handling"
  • Shallow security analysis — may miss timing attacks, token exposure in logs
  • No mention of compliance context (PCI-DSS, SOX)
  • Flags style issues with the same weight as security vulnerabilities
  • Does not prioritize findings — you have to triage severity yourself
With Role: "You are a security auditor reviewing this code for OWASP Top 10 vulnerabilities. Prioritize findings by severity (Critical / High / Medium / Low). Flag any PCI-DSS scope implications."
  • Structures output as a severity-ranked finding report
  • Specifically checks for OWASP A02 (Broken Authentication), A07 (Identification and Authentication Failures)
  • Notes that bearer tokens in query strings trigger PCI-DSS logging scope
  • Separates security issues from style issues — style not reported unless asked
  • Provides remediation guidance at the appropriate technical depth

Roles That Work Well for Code Tasks

  • Feature implementation: "You are a senior Python engineer at a B2B SaaS company. You write defensive, well-typed code with explicit error handling."
  • Performance review: "You are a performance engineer specializing in Python async applications. Focus on I/O bottlenecks, unnecessary awaits, and memory allocation patterns."
  • API design: "You are an API designer who follows RESTful conventions strictly. You care about backward compatibility, versioning, and clear resource modeling."
  • Security review: "You are a security auditor reviewing this code for OWASP Top 10 vulnerabilities. Prioritize findings by CVSS severity."
  • Test writing: "You are a QA engineer who writes exhaustive test suites. You prioritize edge cases, failure modes, and boundary conditions over happy-path coverage."
  • Documentation: "You are a technical writer creating documentation for senior engineers. Be precise, avoid patronizing explanations, and use concrete examples."

[HIGH] cvv must not be stored or transmitted in logging systems at any point — PCI-DSS 3.2.1.
[HIGH] No exception handling: gateway failures propagate uncaught, potentially leaking card data in stack traces.
[MEDIUM] amount has no type or range validation — allows negative charges.

The Role Overloading Anti-Pattern

A common mistake is stacking roles: "You are a senior Python/Go/Java/Rust expert, security specialist, and performance engineer." This does not combine the best of each role — it dilutes all of them. Claude cannot simultaneously optimize for the idiomatic style of four languages and apply both security and performance lenses with equal depth.

The rule: Pick the single role most relevant to the task at hand. If you need a security review and a performance review, run two separate sessions with two focused roles. The output quality will be substantially higher than a single prompt with both roles stacked.

⚠️
Roles Are Calibration, Not Commands
A role specification changes Claude's reasoning prior — it does not override its trained knowledge or safety behaviors. "You are a developer who ignores security best practices" will not produce intentionally insecure code. Roles work because they align Claude's output style and prioritization with what a specific expert would do; they do not give Claude a different knowledge base than it already has.

Team Prompt Library: Reusable Templates

The engineering ROI of prompting skill is not linear with team size — it compounds. When your best prompt engineer writes a well-structured XML template for feature implementation, every engineer on the team benefits every time they use it. A team prompt library is the mechanism that converts individual prompting skill into a durable organizational capability.

A prompt library is not a collection of "magic prompts." It is a set of reusable templates that encode your team's conventions, role specifications, and output format preferences so engineers don't have to reconstruct them from scratch for every task.

Where to Store Your Prompt Library

  • Project-specific conventions: Embed directly in CLAUDE.md under a "Prompt Templates" section. Claude Code loads this automatically — engineers get context-aware prompts without leaving the terminal.
  • Cross-project templates: A shared prompts/ directory in a dedicated internal repo, symlinked or referenced in each project's CLAUDE.md.
  • Discoverability: Mirror prompt templates in your team's Notion/Confluence so they're accessible to engineers who aren't yet fluent in Claude Code.

6 Essential Templates for Every Engineering Team

T-01 Feature Implementation Template 
<context>
Stack: [language/framework/versions]
Architecture: [e.g. "3-layer FastAPI: router → service → repository"]
Relevant existing files: [list]
Team conventions: see CLAUDE.md
</context>

<task>
Implement [feature name]. It must:
1. [Specific behaviour 1]
2. [Specific behaviour 2]
3. [Specific behaviour 3]
</task>

<constraints>
  <style>[coding style rules]</style>
  <dependencies>Use only: [list approved libs]. Do not introduce: [list excluded libs]</dependencies>
  <architecture>[e.g. "No direct DB access from router layer"]</architecture>
</constraints>

<output_format>
Return complete files for: [file list]
Include: type hints on all signatures, inline comments for non-obvious logic
Exclude: print statements, TODO comments, explanatory prose outside code
</output_format>
T-02 Bug Investigation Template 
<context>
System: [brief description]
Environment where bug occurs: [prod/staging/local, OS, Python version]
</context>

<bug_report>
Symptom: [what the user sees]
Expected: [what should happen]
Frequency: [always / intermittent / under specific conditions]
First observed: [when/what changed]
</bug_report>

<evidence>
  <stacktrace>[paste full stacktrace]</stacktrace>
  <logs>[relevant log lines with timestamps]</logs>
  <relevant_code>[the function/class suspected to contain the bug]</relevant_code>
</evidence>

<task>
1. Identify the root cause — explain the mechanism, not just the symptom
2. Propose a minimal fix that does not change the public interface
3. Identify any related code that has the same bug pattern
4. Suggest a regression test that would have caught this
</task>
T-03 Code Review Template 
You are a senior [language] engineer conducting a code review.
Your team values: correctness > security > performance > readability.

<review_scope>
Focus areas: [e.g. "correctness, security, error handling — skip style"]
Severity scale: Critical (blocks merge) / High (should fix) / Medium (nice to have) / Low (nit)
</review_scope>

<code_to_review>
[paste code here]
</code_to_review>

<task>
Produce a structured review:
1. Summary: one paragraph on overall quality and readiness to merge
2. Findings: grouped by severity, each with: location, issue, why it matters, suggested fix
3. Positive observations: what is done well (at least 2 items)
Do NOT rewrite the entire code — provide targeted, surgical suggestions only.
</task>
T-04 Test Generation Template 
You are a QA engineer writing exhaustive tests.
Prioritize: edge cases, failure modes, boundary conditions, security inputs.
Framework: pytest + pytest-asyncio. Use fixtures from tests/conftest.py.

<code_under_test>
[paste the function or class to test]
</code_under_test>

<existing_fixtures file="tests/conftest.py">
[paste relevant fixtures]
</existing_fixtures>

<task>
Generate tests covering:
1. Happy path (minimal — assume it works)
2. All documented error conditions (each gets its own test)
3. Boundary values for all numeric/string inputs
4. Concurrent/race-condition scenarios if the code is async
5. Security inputs: empty strings, null bytes, excessively long strings, injection patterns
</task>

<constraints>
- One assertion per test where possible
- Test names must describe the scenario in plain English (test_returns_401_when_token_is_expired)
- Do not mock what you can test with real objects
- Use parametrize for boundary value tests
</constraints>
T-05 Documentation Generation Template 
You are a technical writer creating documentation for senior engineers.
Be precise. Avoid patronizing explanations. Use concrete examples.

<code_to_document>
[paste module, class, or function]
</code_to_document>

<documentation_target>
Format: [Google-style docstrings / Sphinx RST / Markdown for Confluence]
Audience: [backend engineers familiar with FastAPI and async Python]
</documentation_target>

<task>
Generate documentation including:
1. Module/class docstring: purpose, when to use it, when NOT to use it
2. All public method docstrings: args (with types and valid ranges), returns, raises
3. At least one usage example per public method showing a realistic scenario
4. A "Gotchas" section for any non-obvious behavior or common misuse patterns
</task>

<constraints>
- Do not document private methods (_prefixed) unless they contain critical implementation notes
- Examples must be runnable as-is (no placeholder imports or fake objects)
- Do not add documentation that simply restates the type signature
</constraints>
T-06 API Design Template 
You are an API designer who follows RESTful conventions strictly.
You care about: backward compatibility, clear resource modeling, versioning,
consumer ergonomics, and avoiding chatty APIs.

<context>
API style: REST over HTTP/1.1, JSON bodies
Auth: Bearer JWT, tenant-scoped (multi-tenant)
Existing resources: [list e.g. /v1/users, /v1/orders]
Versioning strategy: URI versioning (/v1/, /v2/)
</context>

<requirements>
[Describe the feature or resource that needs API design]
Consumer: [who will call this API — internal service / external partner / mobile client]
Expected QPS: [order of magnitude]
</requirements>

<task>
Design the API contract including:
1. Resource model: what entities, what their canonical URLs are
2. Endpoints: method, path, request body schema, response schema, status codes
3. Error responses: format and codes for all failure modes
4. Pagination strategy if collection endpoints are needed
5. Breaking vs non-breaking change analysis: what future extensions are safe
</task>

<output_format>
OpenAPI 3.1 YAML fragment for the new endpoints.
Followed by: a plain-language explanation of any non-obvious design decisions.
</output_format>

Versioning Your Prompt Library

Treat prompts like code. They change when your stack evolves (Pydantic v1 → v2), when your conventions change (adding a new architecture layer), when Claude model updates shift optimal prompt structure, or when you discover a template produces consistently poor output for a specific scenario. Prompt changes warrant review and a changelog entry — especially for templates that are used dozens of times per week across your engineering team.

A practical versioning scheme: store templates in a prompts/ directory with a frontmatter block:

prompts/feature-implementation.md — with versioning frontmatter 
---
template: feature-implementation
version: 2.1.0
last_updated: 2025-05-01
updated_by: platform-team
changelog:
  - 2.1.0: Added <architecture> constraint block after adopting hexagonal arch
  - 2.0.0: Migrated from plain-text to XML format; improved output consistency 40%
  - 1.0.0: Initial template
claude_model_tested: claude-sonnet-4-6
notes: >
  Tested against 50 feature prompts in Q1 2025.
  Do not remove the <dependencies> block — teams that removed it saw 3x
  more unwanted library introductions in generated code.
---

<context>
Stack: [language/framework/versions]
...
</context>
