Claude Code – Practitioner Level

Outcome: Move beyond "Claude as chatbot" and operate Claude Code as a true agentic coding collaborator — understanding its architecture, wiring it into team workflows via CLAUDE.md, and deploying it safely inside an enterprise engineering environment.

CLI · Agentic Architecture CLAUDE.md · Project Memory VS Code · JetBrains Enterprise Guardrails
01

What Is Claude Code

Claude Code is Anthropic's CLI-based agentic coding assistant. The word "agentic" is doing real work here — this is not autocomplete, not a chatbot embedded in an IDE, and not a code search tool. Claude Code is a reasoning process that can plan multi-step tasks, traverse your entire codebase, and execute actions inside your operating system with your explicit permission at each step.

The architectural distinction matters enormously for how you use it and how you govern it. A typical autocomplete tool sees the current file, maybe a few open buffers, and predicts the next token. Claude Code sees your project root. It can read 50 files in one pass, understand how a FastAPI router connects to a database model, trace a bug across three layers of abstraction, and then write, modify, and verify the fix — all within a single conversation turn.

┌─────────────────────────────────────────────────────────────────────┐ │ CLAUDE CODE ARCHITECTURE │ └─────────────────────────────────────────────────────────────────────┘ Developer Terminal │ user input (natural language + slash commands) ▼ Claude Code Process (Node.js, runs locally in your shell) │ ├── Context assembly ← reads CLAUDE.md, git history, open files │ ├── Tool calls │ ├── read_file → inspects any file in the project tree │ ├── write_file → creates or edits files (with diff preview) │ ├── run_command → executes shell commands (asks permission) │ └── search → grep/find across the entire codebase │ ▼ Anthropic API (claude-sonnet-4-6 or opus by default) │ returns planned actions + natural language reasoning ▼ Codebase Files & Shell │ changes applied, tests run, output fed back to Claude ▼ Developer Terminal (sees diff + explanation; approves or rejects)
🏗️
The Core Distinction: Agency vs. Autocomplete
Autocomplete tools predict your next token. Claude Code plans. It holds a goal, decomposes it into steps, executes each step with real tool calls, observes the output, and adjusts its plan. This makes it fundamentally different — and fundamentally more powerful, and fundamentally more important to govern correctly.

What "Full Project Context" Actually Means

When you open Claude Code in a project root, it does not wait for you to attach files. It reads the directory structure, ingests your CLAUDE.md (if present), and can read any file on demand. The practical implication: you do not need to copy-paste code into a chat window. You ask in natural language and Claude Code finds the relevant files itself.

  • Codebase traversal: Claude Code can walk the full directory tree and read any file it needs — it determines relevance autonomously, not by your file selection
  • Git awareness: It reads git history, diffs, and branch state — so it can answer "what changed in this module in the last 3 commits?" before it writes a line
  • Iterative execution: After running a test suite, Claude Code reads the failure output and uses that as new context — it closes the read-execute-observe loop without your intervention
  • Model tier: By default, Claude Code uses Sonnet — the best cost-performance tier for coding tasks. Switch to Opus via /model for complex architectural refactors
⚠️
Agency Requires Governance
Because Claude Code can write files and run shell commands, it has a blast radius. In enterprise contexts, every team using Claude Code should have a CLAUDE.md that explicitly bans destructive commands, specifies test requirements before merging, and limits which directories Claude Code is allowed to modify. This is not optional ceremony — it is your control plane. Section 4 covers CLAUDE.md in depth.
02

Installation & Setup

Claude Code installs as a global npm package. The setup takes under five minutes; the misconfiguration surface is small but consequential — get the API key storage right from the beginning, not after you've accidentally committed it to a repo.

Prerequisites

  • Node.js 18+: Claude Code is a Node.js process. Check node --version — if you're below 18, upgrade via nodejs.org or nvm
  • API key: Obtain from console.anthropic.com — requires an Anthropic account. For enterprise, your AI Platform team will provision keys; do not share or reuse personal keys
  • Git (optional but expected): Claude Code uses git context to understand change history. Projects without git still work, but you lose branch-awareness

Step 1 — Install

bash 
# Install globally (requires Node.js 18+)
npm install -g @anthropic-ai/claude-code

# Verify installation
claude --version

Step 2 — Authenticate

Two options: environment variable (preferred for scripts and CI) or interactive auth (preferred for developer workstations).

bash — Option A: Environment variable 
# Add to your shell profile (~/.zshrc or ~/.bashrc)
export ANTHROPIC_API_KEY="sk-ant-api03-..."

# Never hardcode in scripts. Never commit to git.
# For enterprise: source from a secrets manager instead:
export ANTHROPIC_API_KEY=$(aws secretsmanager get-secret-value \
  --secret-id prod/anthropic/api-key \
  --query SecretString \
  --output text)
bash — Option B: Interactive auth 
# Interactive login — opens browser for OAuth flow
claude auth

# Stores credentials in ~/.claude/credentials.json
# Useful for dev workstations where env var management is cumbersome

Step 3 — First Run

bash 
# Navigate to your project root first — context is set from here
cd my-fastapi-project

# Launch Claude Code (interactive REPL)
claude

# Or start with an immediate prompt (useful for scripts)
claude "Explain the architecture of this project in one paragraph"

# Non-interactive (returns JSON output, then exits)
claude --print "List all API endpoints defined in this codebase"

Permission Modes

Permission mode controls how Claude Code handles tool calls — specifically file writes and shell command execution. Choosing the wrong mode for your context creates either frustrating interruptions or unsupervised risk.

Default (Ask)

Claude Code pauses and asks before every file write and shell command.

Use for: Any unfamiliar codebase, onboarding, sensitive repos, production-adjacent work.

claude — no flags needed

Auto-Approve

Claude Code executes tool calls without confirmation. Moves fast; higher blast radius.

Use for: Throwaway branches, greenfield prototypes, after you trust the CLAUDE.md guardrails.

claude --dangerously-skip-permissions

Restricted

Disables file writes and shell execution entirely. Read-only exploration and Q&A only.

Use for: Code review, audits, regulated environments, shared accounts.

claude --no-file-write --no-shell

✅
Default Mode for Enterprise Onboarding
When rolling out Claude Code to a new team, mandate Default (Ask) mode for the first two weeks. Developers build intuition for what Claude Code wants to do before they start auto-approving. After that, auto-approve is reasonable on feature branches with proper CLAUDE.md guardrails. Restricted mode belongs in any environment where you wouldn't give a junior developer write access.
03

Core Commands & Slash Commands

Claude Code's interface is conversational by default — you type in natural language. Slash commands are shortcuts that invoke specific behaviors without constructing a full prompt. Think of them as power-user macros: one keystroke triggers a well-defined action. Knowing these cold is the difference between a developer who uses Claude Code occasionally and one who uses it as a primary workflow tool.

Slash Command Reference

Command What It Does When to Use It
/help Lists all available slash commands and current session status First run on any new machine; when you forget a command
/add <file> Explicitly adds a file or glob pattern to the active context window When Claude Code needs a file it hasn't loaded automatically — e.g. a config in a non-standard location
/clear Resets the conversation history; starts a clean context After completing a task before starting an unrelated one; when context is polluted with dead-end approaches
/review Requests a structured code review of staged changes or a named file Before opening a PR; after a large refactor; whenever you want a second pair of eyes without a teammate
/model Switches the underlying model tier for the current session Switch to Opus for complex multi-file architectural work; back to Sonnet for routine tasks
/diff Shows the diff of all changes Claude Code has made in this session Before committing; to review the full scope of a session's changes at a glance
/undo Reverts the last file write Claude Code performed When a generated change needs to be backed out without a full git reset
/status Shows current model, context size, token usage, and permission mode When debugging unexpected behavior; monitoring context window headroom

Keyboard Shortcuts

Shortcut Action
Ctrl+C Interrupt the current task safely — Claude Code stops mid-execution without corrupting files that are already saved. Partial writes are rolled back.
Ctrl+D Exit Claude Code session gracefully
↑ / ↓ Navigate command history in the REPL

The --print Flag for Non-Interactive Use

The --print flag is essential for CI pipelines, pre-commit hooks, and any scripted workflow. Claude Code runs the prompt, prints the result to stdout in JSON format, and exits. No REPL, no waiting for input.

bash — Non-interactive usage 
# Emit structured output for downstream tooling
claude --print "Does this codebase have any hardcoded API keys? List files and line numbers."

# Use in a shell script
review_output=$(claude --print "Review the staged changes in this repo for security issues")
echo "$review_output" >> security-audit.log

# Combine with jq for structured parsing
claude --print --output-format json "List all TODO comments in the codebase" \
  | jq '.content[0].text'

How a Command Flows Through the System

┌─────────────────────────────────────────────────────────────────────┐ │ CLAUDE CODE COMMAND FLOW │ └─────────────────────────────────────────────────────────────────────┘ You type: "Add pagination to the /users endpoint. Use cursor-based pagination." │ ▼ 1. Context Assembly ├── Reads CLAUDE.md (coding standards, banned patterns) ├── Reads relevant files (router, models, schemas, tests) └── Assembles full prompt (context + instruction → Anthropic API) │ ▼ 2. Planning Phase (visible in terminal as Claude reasons out loud) ├── Identifies files to modify: routes/users.py, schemas/pagination.py ├── Identifies tests to update: tests/test_users.py └── Plans execution order │ ▼ 3. Execution Phase (tool calls, each shown with permission prompt in Default mode) ├── write_file: routes/users.py → [Approve? y/n] ├── write_file: schemas/pagination.py → [Approve? y/n] ├── run_command: pytest tests/test_users.py → [Approve? y/n] └── read_file: reads test output, adjusts if failures │ ▼ 4. Summary (diff + explanation + next steps shown in terminal)
💡
Natural Language Over Commands
Slash commands are useful, but the real productivity unlock is describing intent, not instructions. "Refactor the authentication middleware to use JWT with a 1-hour expiry, replace all session-based auth calls, and update the tests" is a better prompt than trying to micromanage each file edit. Claude Code is more capable than you might initially give it credit for on scope.
04

CLAUDE.md: Project Memory

CLAUDE.md is the single most important configuration artifact in any Claude Code deployment. It is a Markdown file that lives at your project root and is automatically read by Claude Code on every session start — before the first token of your actual prompt is processed. Think of it as the briefing document you'd hand a senior contractor on their first day: tech stack, team conventions, what not to touch, and where things live.

The ROI on writing a thorough CLAUDE.md is exceptional. You write it once. Every developer on your team benefits from it every session. Every time a new hire runs Claude Code on the repo, they inherit the accumulated architectural decisions without anyone explaining them in Slack. It compounds.

CLAUDE.md Hierarchy

Claude Code resolves CLAUDE.md files in a layered hierarchy — global first, then project, then subdirectory. Each level adds or overrides the context established by the layer above.

Layer 1 — Global (User-Level)
~/.claude/CLAUDE.md
Personal defaults that apply to every project you open. Good for: your preferred code style, your name and role, tools you always have available (e.g., "I have ripgrep, bat, and fd installed"), conventions you enforce across all your work.
Layer 2 — Project (Team-Level)
/repo-root/CLAUDE.md
The primary team convention file. Committed to git, reviewed like code, owned by the team. Contains architecture decisions, coding standards, test commands, banned patterns, and directory structure. This is the file this section focuses on.
Layer 3 — Subdirectory (Module-Level)
/repo-root/services/payments/CLAUDE.md
Scoped overrides for a specific module or service. Useful in monorepos where the payments service has different conventions than the frontend. Only loaded when Claude Code is working in or asked about that directory.
✅
Treat CLAUDE.md as Code
Require pull request review for changes to CLAUDE.md just like you would for changes to a core library. If a developer decides to remove the rule banning raw SQL in CLAUDE.md, that decision should go through your normal review process — the same way changing a database migration policy would. CLAUDE.md is your team's contract with the AI.

What to Include in CLAUDE.md

  • Stack and versions: Python 3.12, FastAPI 0.111, SQLAlchemy 2.x, PostgreSQL 16 — so Claude Code doesn't suggest deprecated APIs
  • Test commands: The exact commands to run the test suite — Claude Code uses these after making changes so it can verify before handing back to you
  • Architecture decisions: "All database access goes through the repository layer; never call the ORM directly from routes" — these are the rules that don't appear in the code but shape every design decision
  • Banned patterns: Explicit list of things that must never appear in this codebase — raw SQL, print() debugging, mutable default arguments, etc.
  • Directory map: Where things live — especially in non-obvious layouts (monorepos, legacy structures)
  • Coding style: Line length, import ordering, docstring format, naming conventions — saves constant back-and-forth on trivial style issues

Complete CLAUDE.md Example (FastAPI + PostgreSQL)

CLAUDE.md — project root 
# CLAUDE.md — Acme Corp API Platform
# This file is read by Claude Code on every session start.
# Review changes via PR before merging.

## Stack
- Python 3.12 (use match/case, walrus operator where appropriate)
- FastAPI 0.111 with async endpoints throughout
- SQLAlchemy 2.x (async session; use `select()` not legacy `query()`)
- PostgreSQL 16 — hosted on AWS RDS
- Alembic for migrations (never edit migration files manually)
- Pydantic v2 for all request/response schemas
- pytest + httpx for testing; pytest-asyncio for async tests

## Test Commands
```bash
# Run all tests
pytest tests/ -v --tb=short

# Run a single module
pytest tests/test_users.py -v

# Run with coverage
pytest --cov=app --cov-report=term-missing

# After any database schema change, generate migration:
alembic revision --autogenerate -m "describe_change"
alembic upgrade head
```

## Project Structure
```
app/
  api/           # FastAPI routers — one file per resource
  core/          # Config, security, dependencies, lifespan
  models/        # SQLAlchemy ORM models (not Pydantic)
  repositories/  # All database access — never call ORM from routes
  schemas/       # Pydantic v2 request/response models
  services/      # Business logic layer between routes and repos
tests/           # Mirror app/ structure; fixtures in conftest.py
alembic/         # Migration files — do not edit manually
```

## Architecture Rules
- Routes call Services. Services call Repositories. Repositories call the ORM.
  Never skip a layer. Never call ORM from a route file.
- All endpoints must be async (`async def`). No sync database calls.
- Every new endpoint requires: a schema, a service method, a repository method,
  and at minimum one happy-path + one error-path test.
- Use dependency injection via FastAPI Depends for auth, db sessions, pagination.
- Pagination: cursor-based for all list endpoints. No offset pagination.

## Security Rules
- Never commit secrets. All config via environment variables or AWS Secrets Manager.
- All routes except /health and /docs require authentication.
- Validate and sanitize all user-supplied input at the schema layer (Pydantic).
- Do not expose internal error details in API responses to clients.

## Banned Patterns — Never generate these
- Raw SQL strings (use SQLAlchemy ORM or Core expression language only)
- `print()` statements (use `logging.getLogger(__name__)`)
- `SELECT *` queries
- Mutable default arguments (`def fn(data=[])`)
- `from module import *`
- Hardcoded IP addresses, ports, or credentials
- `time.sleep()` in async code (use `asyncio.sleep()`)

## Code Style
- Black formatting, line length 99
- isort for imports (profile=black)
- Docstrings: Google style for public functions and classes
- Type annotations required on all function signatures
- All Pydantic models use `model_config = ConfigDict(from_attributes=True)`

## PR Checklist (remind me before I commit)
- [ ] Tests pass (`pytest tests/ -v`)
- [ ] No new linting errors (`ruff check app/`)
- [ ] Migration generated if models changed
- [ ] New endpoint documented in OpenAPI description string
- [ ] No print statements, no hardcoded values

Notice what this CLAUDE.md does: it eliminates entire classes of back-and-forth. Claude Code will not suggest offset pagination, will not write raw SQL, will not forget to add tests, will not use the legacy SQLAlchemy query() API. Every one of those rules was probably learned the hard way by someone on your team. Now they are encoded once and enforced automatically for every developer, every session.

IDE Integration

Claude Code's primary interface is the terminal. This is a deliberate design choice — the terminal has access to your full shell environment, your git state, your installed CLIs, and your project's actual file system. IDE integrations layer UX conveniences on top of this foundation without replacing it.

The right mental model: the IDE extension is a context bridge and UX shortcut. It makes it easier to select code and ask questions about it, or to trigger Claude Code actions from a right-click menu. The full agentic capability — multi-file edits, test execution, iterative fixing — lives in the terminal. Use both.

Visual Studio Code VS Code
  • Install "Claude Code" from the VS Code Marketplace
  • Sidebar panel shows conversation history and current context
  • Select any code block → right-click → "Ask Claude" sends selection with context
  • Cmd+Shift+C (Mac) / Ctrl+Shift+C (Win/Linux) opens the Claude panel from anywhere
  • Inline diff review: proposed changes appear as red/green in the editor gutter before you accept
  • The integrated terminal inside VS Code gives full Claude Code CLI access alongside the extension
JetBrains IDEs JetBrains
  • Install "Claude Code" from JetBrains Marketplace (works in IntelliJ, PyCharm, WebStorm, GoLand, Rider)
  • Tool window panel docked to the right sidebar — persists across IDE restarts
  • Select code → right-click → "Ask Claude" / "Refactor with Claude" context actions
  • Diff viewer uses the native JetBrains merge tool — compare proposed changes side-by-side before accepting
  • Works best when you also have Claude Code running in the IDE's built-in terminal for complex multi-file tasks

Key IDE Workflows

  • Select + Ask: Highlight a function, right-click → "Ask Claude" → type "What does this do?" or "Explain the edge cases this misses" — gets you a focused answer without switching windows
  • Inline diff review: Claude Code proposes a change; the IDE shows the diff in the gutter. You accept or reject individual hunks, exactly like a git merge. Never accept blindly — review every hunk
  • Test generation: Select a function, ask "Write pytest tests for this function covering the happy path, type errors, and boundary conditions" — paste the generated tests into the test file
  • Docstring generation: Select a class or function, ask "Write a Google-style docstring" — faster than writing from scratch, consistently formatted
  • Terminal for agentic tasks: When the task spans more than one file, switch to the terminal and give Claude Code the full goal. The IDE extension is for single-file, targeted questions; the terminal is for agentic multi-step execution
💡
Terminal-First, IDE-Enhanced
Developers who use only the IDE extension often plateau at "better autocomplete." Developers who combine the terminal REPL with the IDE extension — using the terminal for complex planning tasks and the IDE for targeted in-editor questions — get the 10× productivity gain. The terminal is where the agency lives.

Using the Integrated Terminal

bash — Inside VS Code or JetBrains terminal 
# The integrated terminal has full project context because
# it launches from the project root automatically.

# Start a full agentic session
claude

# Example: multi-file refactor from the terminal while IDE is open
# Claude Code writes files → IDE refreshes automatically → you review in IDE diff
> Migrate all synchronous database calls in the services/ directory to async.
> Update the tests to use pytest-asyncio. Run the tests after each file change.

# IDE shows diffs in real time as files are written
06

Enterprise Setup & Guardrails

Handing developers Claude Code with a shared API key and no guidance is the fastest way to produce a governance incident. Enterprise deployments need deliberate setup across four dimensions: secret management, team conventions, rate/cost control, and audit logging. None of these are difficult; all of them are skipped by organizations that adopt Claude Code reactively instead of proactively.

API Key Management

The rule is simple: developers never see raw API keys. Keys are fetched at runtime from a secrets manager. This eliminates the entire class of "accidentally committed to git" incidents and gives your security team a single rotation point.

bash — Fetch from AWS Secrets Manager at shell startup 
# Add to ~/.zshrc or ~/.bashrc — executed at every shell start
# Developer never sees the key; it is fetched and injected into env

anthropic_key() {
  aws secretsmanager get-secret-value \
    --secret-id "prod/anthropic/claude-code-api-key" \
    --query SecretString \
    --output text \
    --region us-east-1 2>/dev/null
}

export ANTHROPIC_API_KEY=$(anthropic_key)

# For HashiCorp Vault:
export ANTHROPIC_API_KEY=$(vault kv get \
  -field=api_key \
  secret/engineering/anthropic)
bash — Block accidental commits with pre-commit hook 
#!/bin/sh
# .git/hooks/pre-commit — blocks commits containing API key patterns

if git diff --cached --name-only | xargs grep -l 'sk-ant-api' 2>/dev/null; then
  echo "ERROR: Anthropic API key pattern detected in staged files."
  echo "Remove it and use environment variables or secrets manager."
  exit 1
fi

exit 0

Team CLAUDE.md Governance

The project CLAUDE.md file should be committed to git and treated as a first-class configuration artifact — not a developer's personal notes file. Establish the following policies:

  • Require PR review: CLAUDE.md changes require at least one reviewer who is not the author — the same bar as changes to shared infrastructure
  • Own it explicitly: Assign a CODEOWNER for CLAUDE.md in your repo's CODEOWNERS file — typically the tech lead or platform team
  • Changelog: Keep a comment block at the top of CLAUDE.md logging significant rule additions with dates and rationale — "2025-03-12: Banned offset pagination after scaling incident in KGI-4421"
  • CI validation: Add a lint step that verifies CLAUDE.md contains the required sections (Stack, Test Commands, Banned Patterns) — prevents accidental deletion

Rate Limiting & Cost Controls

Per-Developer Keys
  • Each developer gets their own API key
  • Usage visible per-developer in Anthropic console
  • Easy to revoke when a developer offboards
  • No single key represents the entire team's blast radius
  • Best for: Teams of up to ~30 developers; direct API access
AI Gateway (Org-Level)
  • All Claude Code traffic routes through an internal gateway
  • Per-team token budgets enforced at the gateway
  • Cost attribution by team/project/environment
  • Unified audit log regardless of which key was used
  • Best for: 30+ developers; regulated environments; chargeback requirements

Audit Logging

Claude Code logs all sessions — including the prompts you send and the tool calls it makes — to ~/.claude/logs/ on each developer's machine. In an enterprise context, this is not sufficient; you need centralized, tamper-resistant logs for compliance and incident investigation.

bash — Forward Claude Code logs to a SIEM 
# Example: ship logs to Splunk via HEC (HTTP Event Collector)
# Run as a background daemon on developer machines or golden AMI

while inotifywait -e modify ~/.claude/logs/ 2>/dev/null; do
  for log in ~/.claude/logs/*.jsonl; do
    tail -n 1 "$log" | curl -s \
      -H "Authorization: Splunk $SPLUNK_HEC_TOKEN" \
      -H "Content-Type: application/json" \
      --data-binary @- \
      "https://splunk.internal:8088/services/collector/event"
  done
done

# For AWS CloudWatch Logs (simpler, agent-based approach):
# Add to your CloudWatch agent config:
# {
#   "file_path": "/home/*/. claude/logs/*.jsonl",
#   "log_group_name": "/enterprise/claude-code/sessions",
#   "log_stream_name": "{hostname}/{username}"
# }

Restricted Mode for Regulated Environments

In environments where a developer should not have write access — production codebases under change-freeze, shared review environments, or environments where a second-person rule applies — use restricted mode to disable file writes and shell execution entirely. Claude Code becomes a read-only analysis and Q&A tool.

bash — Restricted mode examples 
# Read-only: no file writes, no shell commands
claude --no-file-write --no-shell

# Alias for regulated environments — add to ~/.zshrc
alias claude-review='claude --no-file-write --no-shell'

# Useful for: compliance audits, code review sessions, production investigation
# Developers can still ask Claude Code to analyze, explain, and recommend —
# they just have to apply changes manually.

# Enforce via CLAUDE.md for the production branch:
# ## Restricted Mode Policy
# All Claude Code sessions on `main` branch must use --no-file-write.
# Changes must be made on a feature branch and reviewed before merge.
⚠️
Auto-Approve is a Privilege, Not a Default
The --dangerously-skip-permissions flag should be treated like sudo: available to developers who understand the consequences, not enabled by default, and never used in environments where changes require review. Enforce this via your onboarding documentation and CLAUDE.md policy — not by trusting developers to self-govern on production repos.

Day 2 · Module 1 — Key Takeaways

  • Agency, not autocomplete: Claude Code plans, executes, observes, and iterates — it is a reasoning process with tool access, not a smarter tab-complete
  • Install and auth: npm install -g @anthropic-ai/claude-code + API key from Secrets Manager; never commit keys; use interactive claude auth on dev workstations
  • Permission modes: Default (ask) for new environments, auto-approve on trusted feature branches, restricted for production-adjacent or regulated contexts
  • Slash commands: /clear, /add, /review, /model, /diff are your day-to-day power tools; --print for scripted/CI use
  • CLAUDE.md is your control plane: Encode team conventions, banned patterns, and test commands once — every developer and every session benefits automatically; review changes like code
  • IDE integration: VS Code and JetBrains extensions add UX shortcuts; the terminal is where agentic power lives — use both together
  • Enterprise guardrails: Per-developer keys or AI Gateway, CLAUDE.md in CODEOWNERS, logs forwarded to SIEM, restricted mode for regulated environments
