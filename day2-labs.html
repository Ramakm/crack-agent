Lab 01
Build a REST API with Claude Code
Use Claude Code to scaffold a complete Task Manager REST API in FastAPI — from bare directory to a tested, secured, production-shaped codebase — without leaving your terminal. The goal is to internalise the scaffold-verify-iterate loop that makes Claude Code effective at scale.
⏱ 35 minutes Intermediate Module 1

Scenario

You are the sole backend engineer on a two-week sprint. The product team needs a Task Manager API — create, list, update, delete tasks with per-user authentication — by end of day. You will use Claude Code to do in 35 minutes what would otherwise take three to four hours.

ℹ
All commands below are Claude Code CLI interactions — text you type into the claude terminal session or slash commands issued inside it. You are not calling the API directly; Claude Code is.

Step 1 — Scaffold the project

Create a fresh directory, enter it, and open Claude Code:

mkdir task-manager-api && cd task-manager-api
claude

Inside the Claude Code session, type this exact prompt:

You are a senior FastAPI engineer. Scaffold a production-ready Task Manager REST API with the following specification:

STACK: FastAPI 0.111, Python 3.11, SQLAlchemy 2.0 (async), SQLite for dev
MODELS: User (id, email, hashed_password, created_at), Task (id, title, description, status, priority, due_date, owner_id, created_at, updated_at)
TASK STATUS: todo | in_progress | done | cancelled
TASK PRIORITY: low | medium | high | critical

ENDPOINTS:
  POST   /auth/register
  POST   /auth/login          → returns JWT access token
  GET    /tasks               → list caller's tasks, supports ?status=&priority= filters
  POST   /tasks               → create task
  GET    /tasks/{id}
  PATCH  /tasks/{id}
  DELETE /tasks/{id}

RULES:
- All /tasks routes require a valid JWT Bearer token
- Users can only read/modify their own tasks (404, not 403, for tasks they do not own)
- Use Pydantic v2 models for request/response validation
- Separate concerns: routers/, models/, schemas/, services/, core/ layout
- Include a requirements.txt and a Makefile with: install, dev, test targets

Generate all files. Do not explain — just produce the code.

Claude Code will create the full project tree. When it stops, run:

Verify the server is running at http://localhost:8000/docs.

Step 2 — Add JWT authentication hardening

The scaffold uses a minimal JWT implementation. Ask Claude Code to harden it:

Review core/security.py. Harden the JWT implementation:
1. Use RS256 (asymmetric) instead of HS256 — generate a dev key pair and load from env
2. Add token expiry validation with a 15-minute access token and 7-day refresh token
3. Add a /auth/refresh endpoint that accepts a valid refresh token and returns a new access token
4. Blacklist revoked tokens in-memory (a set is fine for dev; note where Redis would replace it in prod)

Update all affected files. Show a diff summary when done.

Step 3 — Add input validation and error handling

Add the following validation and error handling across the codebase:
- Task title: 1–200 characters, strip leading/trailing whitespace
- Task description: max 2000 characters
- due_date: must be in the future at creation time; allow null
- Email: validated by Pydantic EmailStr; normalise to lowercase before storing
- Return RFC 7807 Problem Details JSON for all 4xx errors:
  {"type": "...", "title": "...", "status": 400, "detail": "...", "instance": "/tasks/99"}
- Add a global exception handler in main.py that catches unhandled exceptions,
  logs them with a correlation_id, and returns a 500 Problem Details response

Update schemas/, routers/, and main.py. Do not change the database models.

Step 4 — Generate a test suite

Write a pytest test suite in tests/ using httpx.AsyncClient and pytest-asyncio:
- tests/test_auth.py: register, login, refresh, duplicate email, bad credentials
- tests/test_tasks.py: full CRUD happy path, auth enforcement (401 on every endpoint without token),
  cross-user isolation (user A cannot read user B's tasks), filter combinations, due_date validation

Use factory functions to create test users and tasks; do not share state between tests.
Target ≥80% line coverage. Add a pytest.ini that sets asyncio_mode = auto.

Run the suite:

make test

/review

Lab 02
Systematic Debugging Exercise
A subtle async/sync mixing bug causes intermittent 500 errors in a FastAPI endpoint under load. Work through a four-step debugging protocol using Claude — reproduce, isolate, fix, verify — and compare the structured approach to an unstructured "here is my error" dump.
⏱ 25 minutes Foundation Module 2

The Bug

The following FastAPI endpoint works fine in development (single request, single worker) but throws intermittent 500 Internal Server Error under load (10+ concurrent requests). The error is not always in the logs — sometimes it silently corrupts the response.

# routers/tasks.py  — BROKEN version
import requests                           # ← sync HTTP client
from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.ext.asyncio import AsyncSession
from core.database import get_db
from services.task_service import get_task_by_id
from services.notification_service import NotificationService

router = APIRouter(prefix="/tasks", tags=["tasks"])

notification_svc = NotificationService()   # ← module-level singleton

@router.patch("/{task_id}/complete")
async def complete_task(
    task_id: int,
    db: AsyncSession = Depends(get_db),
    current_user=Depends(get_current_user),
):
    task = await get_task_by_id(db, task_id, current_user.id)
    if not task:
        raise HTTPException(status_code=404, detail="Task not found")

    task.status = "done"
    await db.commit()
    await db.refresh(task)

    # Send webhook notification to external system
    resp = requests.post(                  # ← BLOCKING call inside async handler
        "https://hooks.example.com/task-complete",
        json={"task_id": task_id, "user_id": current_user.id},
        timeout=5,
    )
    resp.raise_for_status()

    return {"id": task.id, "status": task.status}


# services/notification_service.py  — BROKEN version
import threading

class NotificationService:
    def __init__(self):
        self._lock = threading.Lock()      # ← threading.Lock in async context
        self._sent: list[int] = []

    def record_sent(self, task_id: int):
        with self._lock:                   # ← blocks the event loop during acquire
            self._sent.append(task_id)

    def get_sent(self) -> list[int]:
        with self._lock:
            return list(self._sent)

Root cause explained

Two compounding issues:

  1. Blocking I/O in an async handler. requests.post() is a synchronous blocking call. When it runs inside an async def handler, it blocks the entire asyncio event loop for the duration of the HTTP request (up to 5 seconds per call). Under concurrent load, this starves every other in-flight request, leading to timeout cascades and garbled responses from uvicorn.
  2. threading.Lock in an async context. threading.Lock.acquire() blocks the calling OS thread. In an asyncio event loop, all coroutines share one thread. When record_sent is awaited while the lock is held by another coroutine (impossible with a threading lock — it will actually deadlock or spin), the event loop cannot yield. The correct primitive is asyncio.Lock.

Debugging Protocol

  • Step 1 — Reproduce with a minimal reproduction
    Before involving Claude, nail down the failure condition. Create repro.py and run it against your local server: 
    import asyncio, httpx

TOKEN = "Bearer <your_test_token>"

async def hit(client, i):
    r = await client.patch(
        "http://localhost:8000/tasks/1/complete",
        headers={"Authorization": TOKEN},
    )
    print(f"{i:02d}  status={r.status_code}  body={r.text[:80]}")

async def main():
    async with httpx.AsyncClient() as client:
        await asyncio.gather(*[hit(client, i) for i in range(20)])

asyncio.run(main())
    With the broken code, several of the 20 concurrent requests will return 500 or hang.
  • Step 2 — Isolate: ask Claude to identify the issue
    Open Claude Code or paste directly into Claude. Use this structured prompt: 
    I have a FastAPI endpoint that causes intermittent 500 errors when 10+ requests hit it concurrently.
The errors do not always appear in logs — sometimes the response is silently wrong.

Here is the full endpoint and its dependency [paste routers/tasks.py and services/notification_service.py].

Symptom: running 20 concurrent PATCH /tasks/1/complete requests via httpx.AsyncClient
produces a mix of 200 and 500 responses. No consistent pattern.

Do NOT suggest logging more — I need the root cause.
Identify every async/sync mixing issue in this code, explain exactly why each one
causes failures under concurrent load, and rank them by severity.
    Claude will identify both the requests.post blocking call and the threading.Lock misuse, and explain the event loop starvation mechanism.
  • Step 3 — Fix: ask Claude to fix and explain the changes
    Fix both issues you identified. Requirements:
- Replace the blocking HTTP call with httpx.AsyncClient; reuse the client across requests
  (do not create a new client per request — that defeats connection pooling)
- Replace threading.Lock with asyncio.Lock in NotificationService
- The endpoint must still commit the DB change before sending the webhook;
  if the webhook fails, log the error but do NOT roll back the task status
  (eventual consistency — the webhook can be retried independently)
- Show the complete fixed files, not a diff
  • Step 4 — Verify: run the reproduction script against the fix
    Apply the fixed files, restart uvicorn, and re-run repro.py. All 20 requests must return 200. Then run make test to confirm no regressions in the existing test suite.

Why structured prompts win

Unstructured — what most developers send
My FastAPI endpoint keeps giving
500 errors. Here's the code [paste].
What's wrong?
Structured — the debugging protocol prompt
Symptom: 20 concurrent requests → mix of
200 and 500, no consistent pattern.
Code: [paste]
Do NOT suggest more logging.
Identify every async/sync mixing issue,
explain why each causes failures under
concurrent load, rank by severity.

The unstructured prompt gets generic advice ("add try/except", "check your logs"). The structured prompt forces Claude to commit to a specific mechanism — event loop starvation — and rank findings, making the output immediately actionable.

⚠️
The general rule: Claude performs best when you give it a falsifiable hypothesis to test, not an open-ended question. "What's wrong?" is an invitation to guess. "Identify every async/sync mixing issue and explain why each causes failures under concurrent load" is a precise engineering question.

The Scoring Framework

Score each prompt 0–2 on each of five elements. Maximum 10 points.

Element 0 — Absent 1 — Partial 2 — Complete
Role No role given Generic ("expert developer") Specific, relevant ("senior Python backend engineer familiar with SQLAlchemy async")
Context No context Some context, missing key details Stack, constraints, existing code, and the "why" are all present
Task Vague or open-ended Goal is clear but scope is fuzzy Precise action verb, clear scope, measurable success condition
Constraints None stated One constraint mentioned Explicit do/do-not list covering edge cases and non-obvious limitations
Output Format Not specified Format mentioned but vague Exact format: file names, function signatures, comment style, what to include/exclude

Lab 04
Git Diff Automation
Build a shell script that pipes git diff into Claude and receives a structured PR description back. Add conventional commit parsing, a test-exclusion flag, and install it as a git alias. Finish with a pre-push hook that saves a draft PR description automatically.
⏱ 20 minutes Foundation Module 4

Prerequisites

You need the Claude CLI installed and accessible as claude on your PATH, and a git repository with uncommitted or recently committed changes on a non-main branch. Confirm:

claude --version        # should print 1.x
git log --oneline -5    # should show recent commits

Step 1 — The basic script

Create ~/bin/pr-describe and make it executable:

#!/usr/bin/env bash
# pr-describe — generate a PR description from git diff
# Usage: pr-describe [--no-tests] [base-branch]
set -euo pipefail

BASE_BRANCH="${1:-main}"
EXCLUDE_TESTS=false

if [[ "${1:-}" == "--no-tests" ]]; then
  EXCLUDE_TESTS=true
  BASE_BRANCH="${2:-main}"
fi

# Collect the diff
if [[ "$EXCLUDE_TESTS" == "true" ]]; then
  DIFF=$(git diff "$BASE_BRANCH"...HEAD -- \
    ':!tests/' ':!**/*_test.py' ':!**/*.test.ts' ':!**/*.spec.ts')
else
  DIFF=$(git diff "$BASE_BRANCH"...HEAD)
fi

if [[ -z "$DIFF" ]]; then
  echo "No diff found against $BASE_BRANCH. Have you committed your changes?" >&2
  exit 1
fi

# Build the prompt
PROMPT="You are a senior software engineer writing a GitHub pull request description.

Analyse the following git diff and produce a PR description with these exact sections:

## Summary
3–5 bullet points. Each starts with a conventional verb (Add, Fix, Update, Remove, Refactor).
Focus on the *why* and *impact*, not a line-by-line restatement of the diff.

## Changes
A grouped list of files changed. Group by logical area (e.g. Auth, API Routes, Tests, Config).
For each group, one sentence describing the change.

## Test plan
A markdown checklist of the manual and automated tests a reviewer should verify.
Be specific — name the test files or test commands where applicable.

## Breaking changes
\"None\" if there are no breaking changes. Otherwise, list each breaking change and its
migration path for callers.

---
Output the PR description only. No preamble, no explanation.

GIT DIFF:
$DIFF"

echo "$PROMPT" | claude --print
chmod +x ~/bin/pr-describe
# Ensure ~/bin is on PATH — add to .zshrc/.bashrc if not:
export PATH="$HOME/bin:$PATH"

Test it:

pr-describe main

Step 2 — Conventional commit parsing

Add commit message parsing so Claude gets structured context about intent, not just diffs:

# Insert after the DIFF block, before PROMPT=
COMMITS=$(git log "$BASE_BRANCH"...HEAD --pretty=format:"%h %s" --no-merges)

COMMIT_CONTEXT="Branch: $BRANCH
Commits: $COMMIT_COUNT
Conventional types detected:
$TYPES

Recent commit messages:
$COMMITS"

Then prepend $COMMIT_CONTEXT to the prompt before the diff:

PROMPT="...
COMMIT CONTEXT:
$COMMIT_CONTEXT

GIT DIFF:
$DIFF"

Step 3 — The --no-tests flag

Already built into Step 1. The flag excludes common test file patterns from the diff so Claude focuses on production code changes. Verify it works:

You should see a PR description that does not mention test file changes in the Changes section, even if tests were modified.

The complete final script

#!/usr/bin/env bash
# pr-describe — generate a structured PR description from git diff + commit history
# Usage: pr-describe [--no-tests] [base-branch]
#
# Install:
#   chmod +x ~/bin/pr-describe
#   git config --global alias.pr-describe '!pr-describe'
set -euo pipefail

# ── Argument parsing ─────────────────────────────────────────────────────────
EXCLUDE_TESTS=false
BASE_BRANCH="main"

while [[ $# -gt 0 ]]; do
  case "$1" in
    --no-tests) EXCLUDE_TESTS=true; shift ;;
    *)          BASE_BRANCH="$1"; shift ;;
  esac
done

# ── Diff collection ───────────────────────────────────────────────────────────
if [[ "$EXCLUDE_TESTS" == "true" ]]; then
  DIFF=$(git diff "$BASE_BRANCH"...HEAD -- \
    ':!tests/' ':!**/test_*.py' ':!**/*_test.py' \
    ':!**/*.test.ts' ':!**/*.spec.ts' ':!**/*.test.js')
else
  DIFF=$(git diff "$BASE_BRANCH"...HEAD)
fi

if [[ -z "$DIFF" ]]; then
  echo "No diff found against '$BASE_BRANCH'. Commit your changes first." >&2
  exit 1
fi

# ── Commit context ────────────────────────────────────────────────────────────
BRANCH=$(git rev-parse --abbrev-ref HEAD)
COMMIT_COUNT=$(git rev-list --count "$BASE_BRANCH"...HEAD 2>/dev/null || echo "unknown")
COMMITS=$(git log "$BASE_BRANCH"...HEAD \
  --pretty=format:"%h %s" --no-merges 2>/dev/null | head -20)

TYPES=$(echo "$COMMITS" | grep -oE '^[a-f0-9]+ (feat|fix|docs|refactor|test|chore|perf|ci|build|revert)' \
  | awk '{print $2}' | sort | uniq -c | sort -rn \
  | awk '{printf "  - %s (%d commit%s)\n", $2, $1, ($1>1?"s":"")}' || true)

[[ -z "$TYPES" ]] && TYPES="  - (no conventional commit prefixes detected)"

COMMIT_CONTEXT="Branch: $BRANCH
Commits in this PR: $COMMIT_COUNT
Conventional commit types:
$TYPES

Recent commits:
$COMMITS"

# ── Claude prompt ─────────────────────────────────────────────────────────────
PROMPT="You are a senior software engineer writing a GitHub pull request description.

Analyse the commit context and git diff below and produce a PR description with
these exact sections in this exact order:

## Summary
3–5 bullet points. Start each with a conventional verb (Add, Fix, Update, Remove, Refactor).
Focus on impact and intent — not a restatement of the diff line by line.

## Changes
Files changed, grouped by logical area (e.g. Auth, API, Models, Tests, Config, CI).
One sentence per group describing what changed and why.

## Test plan
A markdown checklist. Be specific: name test files, endpoints, and edge cases to verify.
Include both automated (\`make test\`) and manual steps.

## Breaking changes
\"None.\" if no breaking changes. Otherwise: what breaks, who is affected, migration path.

---
Output the PR description only. No preamble, no meta-commentary.

COMMIT CONTEXT:
$COMMIT_CONTEXT

GIT DIFF:
$DIFF"

echo "$PROMPT" | claude --print

Extension challenge — Pre-push hook

Add a git pre-push hook that auto-generates a draft PR description and saves it to .git/PR_DRAFT.md whenever you push a branch other than main:

#!/usr/bin/env bash
# .git/hooks/pre-push
# Auto-generate a PR description draft before every non-main push.
set -euo pipefail

CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)

# Only run for non-main branches
if [[ "$CURRENT_BRANCH" == "main" || "$CURRENT_BRANCH" == "master" ]]; then
  exit 0
fi

# Only run if claude is available
if ! command -v claude &>/dev/null; then
  echo "[pre-push] claude not found — skipping PR draft generation"
  exit 0
fi

echo "[pre-push] Generating PR description draft..."

# Check for commits ahead of main
COMMIT_COUNT=$(git rev-list --count "main"...HEAD 2>/dev/null || echo "0")
if [[ "$COMMIT_COUNT" == "0" ]]; then
  echo "[pre-push] No commits ahead of main — skipping"
  exit 0
fi

# Generate and save
DRAFT_FILE="$(git rev-parse --git-dir)/PR_DRAFT.md"
if pr-describe main > "$DRAFT_FILE" 2>/dev/null; then
  echo "[pre-push] PR draft saved to .git/PR_DRAFT.md"
  echo "[pre-push] Preview:"
  head -10 "$DRAFT_FILE"
  echo "..."
else
  echo "[pre-push] PR draft generation failed — continuing push anyway"
fi

exit 0
chmod +x .git/hooks/pre-push

Now every git push on a feature branch saves a draft at .git/PR_DRAFT.md that you can copy directly into GitHub's PR description field.

5 Key Principles to Carry into Day 3

  • Scaffold, do not generate. Claude Code is most effective when it owns the full project graph — file structure, imports, and tests together. Asking for individual functions loses the structural leverage.
  • Give Claude a falsifiable hypothesis, not an open question. "Identify every async/sync mixing issue" produces a precise answer. "What's wrong?" produces guesses. The form of the question determines the quality of the answer.
  • Constraints prune the solution space. Every prompt for production code needs an explicit do-not list. Without constraints, Claude optimises for generality; with them, it optimises for your environment.
  • Verify before you trust. The /review pass and the test suite are not optional extras — they are the feedback loop that makes Claude Code safe to use at speed. Skip them and you are accumulating hidden debt.
  • Automate the toil layer. If you are doing something repetitive around code — writing PR descriptions, tagging issues, updating changelogs — a Claude CLI pipeline will do it faster and more consistently than any human. The git pre-push hook is one example of dozens.
