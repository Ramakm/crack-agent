Context Engineering & CLAUDE.md

Outcome: Build a complete, production-grade context layer for your team's Claude deployment — mastering CLAUDE.md, AGENTS.md, and SKILL.md design, understanding the three-layer context hierarchy, and applying optimization strategies that reduce token cost while improving output quality.

Context Hierarchy CLAUDE.md · AGENTS.md · SKILL.md Token Optimization Enterprise Governance
01

The Context Engineering Model

Context engineering is the discipline of deliberately shaping what Claude knows at the moment it reasons about your task. It is not prompt engineering — prompt engineering optimises a single request. Context engineering optimises the environment in which every request runs: the project knowledge, constraints, conventions, and tools that Claude has access to before the conversation even begins.

The distinction matters at enterprise scale. A single developer can re-explain their stack on every conversation. A team of 40 engineers cannot. Context engineering is the infrastructure answer to that problem — you encode institutional knowledge once, in the right places, and every Claude session inherits it automatically.

◈
The Context Debt Problem: Teams that skip structured context files force every developer to re-explain the same constraints, conventions, and architecture decisions on every session. This is context debt — and it compounds exactly like technical debt. The fix is the same: invest upfront, maintain it as code, pay it down before it owns you.

Three Layers of Context

Every Claude Code session assembles context from three distinct layers. Understanding which layer carries which information is the foundation of effective context architecture.

Layer 1 — Persistent Context Always Present
~/.claude/CLAUDE.md · {project}/CLAUDE.md · {subdir}/CLAUDE.md
CLAUDE.md files loaded automatically at session start. These encode durable facts: stack versions, architectural decisions, coding standards, security rules, banned patterns, and team conventions. This layer is managed like source code — versioned, reviewed, and owned.
Layer 2 — Session Context Conversation Lifetime
Conversation history · /add files · Claude's own observations
Everything accumulated during the current session: your messages, Claude's responses, files you've explicitly added with /add, and Claude's in-session observations about the codebase. This layer is ephemeral — it resets on /clear or session end.
Layer 3 — Injected Context Task-Specific
RAG retrieval · Tool results · MCP server data · Search output
Dynamic context injected programmatically at runtime — retrieved documentation, database query results, API responses, semantic search hits. This layer is generated fresh per task and carries the highest specificity. Requires MCP or custom tooling to populate.

The CLAUDE.md Inheritance Hierarchy

Within the persistent context layer, CLAUDE.md files form a scoped inheritance chain. Each layer narrows scope — global rules apply everywhere, project rules apply to one repo, subdirectory rules apply to one service or module. The closer the file to the current working directory, the higher its authority for scope-specific guidance.

┌─────────────────────────────────────────────────────────────────────┐ │ CLAUDE.md CONTEXT HIERARCHY │ └─────────────────────────────────────────────────────────────────────┘ ~/.claude/CLAUDE.md ← Global layer │ Applied to every Claude session for this user on this machine │ Org-wide standards, personal preferences, shell conventions │ ├── {repo-root}/CLAUDE.md ← Project layer │ Applied whenever Claude runs inside this repository │ Stack spec, architecture overview, test/lint/run commands │ Security rules, PR checklist, banned patterns │ │ ├── services/payments/CLAUDE.md ← Subdirectory layer │ │ Scope: payments microservice only │ │ PCI-DSS constraints, Stripe SDK conventions, │ │ idempotency key pattern, no logging of card data │ │ │ └── services/search/CLAUDE.md ← Subdirectory layer Scope: search microservice only Elasticsearch index naming, analyzer config rules, query DSL conventions, performance budgets ───────────────────────────────────────────────────────────────── All three layers feed into Claude's context window before any user message is processed Priority resolution: subdirectory > project > global Conflict: more specific file wins for overlapping rules
⚠
Key Principle: Well-engineered persistent context eliminates the need for verbose per-request prompts. If you find developers adding the same system context to every request, that content belongs in CLAUDE.md — not in the conversation.
02

Designing an Enterprise CLAUDE.md

The most important thing to understand about CLAUDE.md is what it is not. It is not documentation. It is not a README. It is not a style guide PDF that gets read once and forgotten. CLAUDE.md is executable context — Claude reads it before every session and acts on it. Every line you write is an instruction that will be followed. Write it like you're writing to a capable engineer joining your team on day one, not like you're writing release notes.

What Belongs in CLAUDE.md

The test is simple: will knowing this change how Claude writes or modifies code in this project? If yes, it belongs. If no, it wastes tokens.

✓
Include
Stack versions and dependencies. Run/test/lint/migrate commands. Directory structure and what lives where. Coding conventions. Security rules. Banned patterns and their replacements. PR checklist items.
✗
Exclude
Lengthy prose Claude doesn't act on. Secrets and credentials. Content that changes daily. Business context unrelated to code generation. Explanations that only make sense to humans.

Production CLAUDE.md: Python/FastAPI Microservice

The following is a complete, production-quality CLAUDE.md for a Python FastAPI microservice backed by PostgreSQL. Every section earns its place.

# CLAUDE.md — Payments Service

## Stack
- Python 3.12 (use match/case, structural pattern matching where appropriate)
- FastAPI 0.111 — use APIRouter, not app directly; mount routers in main.py
- SQLAlchemy 2.0 async ORM — AsyncSession, select() DSL, NOT legacy Query API
- PostgreSQL 16 via asyncpg driver
- Alembic for migrations — never ALTER TABLE manually
- Pydantic v2 for all request/response models
- httpx for outbound HTTP (NOT requests, NOT aiohttp)
- Redis 7 via redis-py async for caching and idempotency keys
- pytest + pytest-asyncio for tests; coverage threshold: 85%

## Commands
```bash
# Development
uvicorn app.main:app --reload --port 8001

# Tests
pytest tests/ -v --cov=app --cov-report=term-missing

# Lint + type check (must pass before commit)
ruff check app/ tests/
mypy app/ --strict

# Database migrations
alembic revision --autogenerate -m "description"
alembic upgrade head

# Rollback last migration
alembic downgrade -1
```

## Architecture
```
app/
  main.py          # FastAPI app factory, lifespan, router mounts
  config.py        # Settings via pydantic-settings; single source of truth
  models/          # SQLAlchemy ORM models — one file per domain entity
  schemas/         # Pydantic request/response models — never reuse ORM models
  routers/         # APIRouter instances — one file per resource
  services/        # Business logic — routers call services, services call repos
  repositories/    # DB access layer — all SQL lives here, nowhere else
  dependencies/    # FastAPI Depends() factories (db session, auth, rate limit)
  core/
    security.py    # JWT encode/decode, password hashing
    exceptions.py  # Custom HTTPException subclasses
  tests/
    conftest.py    # Fixtures: async_client, test_db, mock_redis
```

## Coding Conventions
- All functions that touch I/O must be async
- Type hints required on every function signature — no bare `Any` without comment
- No global mutable state — use FastAPI dependency injection exclusively
- Repository methods return domain models or None — never raise HTTP exceptions
- Service methods raise domain exceptions — routers convert them to HTTP responses
- Use `from __future__ import annotations` in every file
- Dataclasses over plain dicts for internal data transfer
- f-strings only — no % formatting, no .format()

## Security Rules
- NEVER construct SQL strings manually — use SQLAlchemy ORM or text() with bound params
- NEVER log request bodies that may contain card data, PII, or passwords
- NEVER hardcode secrets — all config via environment variables through config.py
- Validate ALL external inputs with Pydantic models at the router boundary
- Stripe webhook payloads MUST be verified with stripe.Webhook.construct_event()
- Idempotency keys required on all POST /payments endpoints — store in Redis with 24h TTL
- Authentication header must be validated before ANY business logic executes

## Banned Patterns
| Don't use | Use instead | Reason |
|-----------|-------------|--------|
| `requests.get()` | `httpx.AsyncClient` | Blocking I/O in async context |
| `session.query(Model)` | `select(Model)` | Legacy SQLAlchemy 1.x API |
| `os.environ["KEY"]` | `settings.key` from config.py | No central validation |
| `print()` for logging | `structlog.get_logger()` | No structured output |
| `time.sleep()` | `asyncio.sleep()` | Blocks event loop |
| Bare `except:` | `except SpecificException:` | Swallows bugs |
| `dict` for API response | Pydantic response model | No validation, no docs |

## PR Checklist
Before marking a PR ready for review, verify:
- [ ] `ruff check` passes with zero warnings
- [ ] `mypy --strict` passes
- [ ] New endpoints have request + response Pydantic models
- [ ] New DB queries are in the repository layer only
- [ ] New migrations tested with `alembic upgrade head` + `alembic downgrade -1`
- [ ] No secrets or PII in logs (grep for card, password, secret, token in log calls)
- [ ] New async functions have corresponding async tests
- [ ] Coverage remains above 85% (`pytest --cov` output attached to PR)
✓
The Banned Patterns Table is your highest-ROI section. A table of "don't use X, use Y instead" prevents entire categories of code review back-and-forth. Claude will see the table, understand the constraint, and apply the correct pattern without being asked. This is the difference between context as documentation and context as executable instruction.

AGENTS.md: Multi-Agent Context

AGENTS.md is the CLAUDE.md equivalent for agentic and multi-agent workflows. Where CLAUDE.md tells Claude about your codebase, AGENTS.md tells an orchestrator agent about its worker fleet — what agents exist, what they can do, what inputs they accept, what outputs they produce, and what to do when they fail.

Without AGENTS.md, an orchestrator must either guess at worker capabilities (producing unpredictable delegation decisions) or be re-briefed on every run (wasting context). AGENTS.md makes orchestration deterministic — the orchestrator reads the manifest, selects the right agent for the task, and delegates with the correct input format. No guessing. No drift.

Agent Entry Structure

Each agent entry in AGENTS.md should define the complete contract the orchestrator needs to use that agent reliably: name, purpose, input format, output format, known failure modes, and escalation path.

Researcher
Purpose Gather information from web sources, internal docs, and databases. Does not synthesise — returns raw findings.
Input JSON: {"query": str, "sources": list[str], "depth": "shallow|deep"}
Output Markdown list of findings, each with source URL and confidence score 0–1.
Failure Returns {"error": "NO_RESULTS"} if nothing found after 3 retries.
Escalate On NO_RESULTS, orchestrator widens query scope before retrying.
Analyst
Purpose Synthesise research findings into structured analysis. Identifies patterns, contradictions, and key insights.
Input Researcher output (markdown findings list) + JSON analysis brief with required dimensions.
Output Structured JSON: {"insights": list, "gaps": list, "confidence": float}
Failure Returns {"error": "INSUFFICIENT_DATA", "missing": list} if confidence < 0.4.
Escalate Orchestrator triggers another Researcher pass targeting the missing data points.
Writer
Purpose Convert structured analysis into polished prose for the specified output format and audience.
Input Analyst JSON output + format spec: {"format": "report|brief|slide", "audience": str, "length": str}
Output Markdown document. Always includes an executive summary section, regardless of format.
Failure Returns partial output with <!-- DRAFT --> marker if analyst input is ambiguous.
Escalate On DRAFT output, orchestrator sends back to Analyst with clarification request.
Reviewer
Purpose Validate factual accuracy, flag unsupported claims, check tone and format compliance.
Input Writer output (markdown) + original research findings for fact-checking reference.
Output JSON: {"approved": bool, "issues": list[{type, location, description}], "score": float}
Failure Returns {"approved": false, "issues": [...]} — never silently passes flawed content.
Escalate Score < 0.6: re-route to Writer with issue list. Score < 0.3: escalate to human review.

Complete AGENTS.md Example

# AGENTS.md — Research Intelligence System

## Overview
4-agent pipeline: Researcher → Analyst → Writer → Reviewer
Orchestrated by Claude with this file as the agent fleet manifest.
Never delegate to an agent with input that doesn't match its expected format.

## Communication Protocol
- All inter-agent messages: JSON encoded
- All errors: {"error": "ERROR_CODE", "detail": str, "retryable": bool}
- Max retries per agent: 3
- Timeout per agent call: 120 seconds
- After 3 failures: mark pipeline as FAILED, return partial results to caller

## Agents

### Researcher
Purpose: Gather raw information. Does not interpret or synthesise.
Input:
  query: str          # The information need
  sources: list[str]  # Allowed: ["web", "internal_docs", "database"]
  depth: str          # "shallow" (top 5 results) | "deep" (top 20 + follow links)
Output: Markdown list. Each item: "- [SOURCE] Finding text (confidence: 0.0–1.0)"
Failure codes:
  NO_RESULTS     — nothing found; retryable with broader query
  SOURCE_ERROR   — source unavailable; retryable with different source
  TIMEOUT        — exceeded 120s; retryable
Escalation: If NO_RESULTS after 3 retries, orchestrator broadens query or adds sources.

### Analyst
Purpose: Synthesise Researcher output into structured insights.
Input: researcher_output: str + analysis_brief: {dimensions: list[str], depth: str}
Output: {insights: list[str], contradictions: list[str], gaps: list[str], confidence: float}
Failure codes:
  INSUFFICIENT_DATA  — confidence < 0.4; not retryable without more research
  AMBIGUOUS_INPUT    — researcher output malformed; escalate immediately
Escalation: On INSUFFICIENT_DATA, return missing list to orchestrator for targeted re-research.

### Writer
Purpose: Convert structured analysis to polished prose.
Input: analyst_output: dict + format_spec: {format: str, audience: str, length: str}
Supported formats: "executive_report" | "technical_brief" | "slide_deck_outline"
Output: Markdown document. Always starts with "## Executive Summary" section.
Failure codes:
  INSUFFICIENT_ANALYSIS  — analyst output missing required dimensions
  FORMAT_UNKNOWN         — unrecognised format spec; escalate to orchestrator
Escalation: On INSUFFICIENT_ANALYSIS, return to Analyst with missing_dimensions list.

### Reviewer
Purpose: Gate output quality before delivery.
Input: writer_output: str + source_findings: str (original Researcher output for fact-check)
Output: {approved: bool, score: float, issues: list[{type, location, severity, description}]}
Issue types: "factual_error" | "unsupported_claim" | "tone_mismatch" | "format_violation"
Thresholds:
  score >= 0.8: approved: true, deliver to caller
  score 0.6–0.8: approved: false, return to Writer with issue list
  score < 0.6: approved: false, escalate to human review queue
Failure codes:
  REVIEW_TIMEOUT  — retryable once; if second timeout, approve with "REVIEW_SKIPPED" flag

## Orchestration Rules
1. Always run Researcher before Analyst — Analyst cannot operate without findings
2. Analyst must reach confidence >= 0.4 before Writer is invoked
3. Writer output always goes through Reviewer — no exceptions
4. If the pipeline fails after all retries, return {"status": "FAILED", "partial": {...}}
5. Human escalation queue: POST /api/human-review with payload and pipeline_id

SKILL.md: Reusable Behaviour Modules

SKILL.md defines reusable, composable behaviours that Claude can load on demand. Think of skills as a library of expert modes — Claude loads the relevant skill for the current task and operates with the specialised knowledge and constraints that skill defines. Without skills, developers either write verbose per-request instructions or rely on Claude's general behaviour, which is less precise.

The enterprise model is to maintain a shared skills/ directory in your team's configuration repository, treat skills like shared libraries, and reference them from project CLAUDE.md files. A skill is loaded via /load skills/sql-reviewer.md in Claude Code, or referenced in your CLAUDE.md with an include directive.

Skill Design Principles

Each skill has four mandatory components: a name (machine-readable), trigger conditions (when to activate this skill), behaviour instructions (what Claude does differently in this mode), and an output format (what the response looks like). Anything beyond these four is optional decoration.

sql-reviewer Security & Performance
Trigger Conditions
Activated when reviewing SQL queries, SQLAlchemy ORM code, database migrations, or any code that constructs or executes queries. Also triggered by: "review this query", "check this migration", "is this safe?".
Behaviour Instructions
Check for: (1) SQL injection vectors — string concatenation into queries, unparameterised user input; (2) missing index usage — queries on unindexed columns in large tables; (3) N+1 patterns — loops containing queries; (4) transaction scope — mutations outside explicit transactions; (5) lock contention — SELECT FOR UPDATE on high-traffic tables; (6) migration safety — column additions with NOT NULL but no DEFAULT on populated tables.
Output Format
Structured report: CRITICAL issues first (stop PR), then WARNINGS (should fix), then NOTES (consider). Each issue: location, description, severity, remediation code snippet. End with: "APPROVED" | "APPROVED WITH WARNINGS" | "BLOCKED: [reasons]".
api-designer Design & Standards
Trigger Conditions
Activated when designing new API endpoints, reviewing endpoint signatures, writing OpenAPI specs, or evaluating REST contract decisions.
Behaviour Instructions
Apply: (1) RESTful resource naming — plural nouns, no verbs in paths; (2) HTTP verb semantics — GET idempotent/safe, POST non-idempotent, PUT/PATCH for updates; (3) status code precision — 201 for created, 202 for accepted async, 204 for no-body success, 409 for conflict; (4) versioning strategy — URL path versioning /v1/ preferred; (5) pagination — cursor-based for large collections; (6) error envelope — always {"error": {"code": str, "message": str, "details": []}}; (7) idempotency key header on all mutation endpoints.
Output Format
OpenAPI 3.1 YAML snippet for the endpoint(s), followed by a "Design Decisions" section explaining any non-obvious choices and alternatives considered.
security-auditor Security Review
Trigger Conditions
Activated for security review requests, code handling authentication/authorisation, input validation, file upload handling, external HTTP calls, or any code touching credentials, sessions, or cryptography.
Behaviour Instructions
Evaluate against OWASP Top 10 + these project-specific checks: (1) SSRF — validate and restrict URLs in user-controlled outbound requests; (2) Path traversal — validate file paths against allowed prefixes; (3) JWT — verify algorithm is RS256/ES256, check expiry, validate audience claim; (4) Rate limiting — confirm auth endpoints have per-IP limits; (5) Secrets — grep for hardcoded keys, tokens, passwords in diff; (6) Dependency confusion — check import paths for typosquatting risk; (7) Logging PII — confirm no sensitive fields appear in log statements.
Output Format
CVSS-style severity rating (Critical/High/Medium/Low/Informational) per finding. Each finding: description, attack scenario, line reference, remediation. Final summary table with finding count per severity. Conclude with overall risk posture: ACCEPTABLE | NEEDS REMEDIATION | BLOCKED.

# SKILL.md — Engineering Platform Team

## How to Use Skills
Load a skill with: /load skills/{skill-name}.md
Reference in CLAUDE.md: `@include skills/sql-reviewer.md` (auto-loads for this project)
Skills stack — loading multiple skills applies all constraints simultaneously.

## Skill: sql-reviewer
Trigger: SQL queries, ORM code, Alembic migrations, "review this query"
Behaviour:
  - Check injection vectors: string concat, f-string interpolation into query text
  - Check N+1: queries inside loops or list comprehensions

  - Check transaction scope: mutations need explicit session.commit() or async with

  ### WARNING (should fix)

## Skill: security-auditor
Trigger: Auth code, file handling, outbound HTTP, crypto, secrets, "security review"

## Skill: documentation-writer
Trigger: "write docs", "document this", "add docstrings", "update README"
Behaviour:

  Inline docstrings inserted into the provided code, or Markdown document.

◈
Enterprise Skill Governance: Treat the skills/ directory like a shared library — it lives in a central config repo, it has owners, changes go through PR review, and breaking changes require a deprecation cycle. Individual teams reference org skills; they don't fork them. This keeps expert context consistent across all teams and repositories.

Context Window Optimization

Claude's 200K context window is large — but not free. Every token you consume costs money, adds latency, and potentially dilutes the signal-to-noise ratio of your session. The goal of context optimization is not to minimise tokens at all costs; it is to maximise the quality-per-token ratio. Stale, redundant, or poorly structured context is more expensive than a well-maintained 5K CLAUDE.md that covers the same ground.

Four Optimization Strategies

1
CLAUDE.md Replaces System Prompts
Every constraint in your CLAUDE.md eliminates one piece of per-request context. A well-maintained CLAUDE.md is a 0-token investment at request time — it is already in the persistent layer.
2
Surgical /add
Only add files directly relevant to the current task. Adding the entire services/ directory when you're fixing one endpoint is expensive and counterproductive — Claude will spread its attention across irrelevant files.
3
Clear on Drift
/clear when the session has accumulated noise from debugging detours, cancelled approaches, or topic switches. Starting fresh is cheaper and more effective than trying to correct accumulated session drift.
4
Structure Over Prose
Bullet lists, tables, and code blocks in CLAUDE.md convey the same information as prose in roughly 40–60% fewer tokens. Structured content is also parsed more reliably — Claude extracts constraints from lists more accurately than from paragraphs.

Token Budget Estimation

Before scaling a workflow to many developers or high frequency, estimate the session cost. The following snippet uses tiktoken to calculate token counts for your context files and a representative session.

"""
context_budget.py — Estimate Claude session token cost

Usage:
  python context_budget.py --claude-md CLAUDE.md --session-log session.txt
"""
import tiktoken
import argparse
import pathlib

# Claude uses cl100k_base tokenizer (same as GPT-4 family)
# For precise counts, use Anthropic's token counting API endpoint
ENC = tiktoken.get_encoding("cl100k_base")

# Claude 3.5 Sonnet pricing (as of 2025-Q2)
INPUT_COST_PER_MTK  = 3.00   # USD per million input tokens
OUTPUT_COST_PER_MTK = 15.00  # USD per million output tokens


def count_tokens(text: str) -> int:
    return len(ENC.encode(text))


def estimate_session(
    claude_md_path: pathlib.Path,
    session_log_path: pathlib.Path | None = None,
    avg_output_tokens: int = 800,
    sessions_per_day: int = 50,
) -> dict:
    claude_md_tokens = count_tokens(claude_md_path.read_text())

    session_tokens = 0
    if session_log_path and session_log_path.exists():
        session_tokens = count_tokens(session_log_path.read_text())

    # Total input = persistent context + session accumulation
    total_input = claude_md_tokens + session_tokens
    total_output = avg_output_tokens

    cost_per_session = (
        (total_input  / 1_000_000) * INPUT_COST_PER_MTK +
        (total_output / 1_000_000) * OUTPUT_COST_PER_MTK
    )
    daily_cost = cost_per_session * sessions_per_day
    monthly_cost = daily_cost * 22  # working days

    return {
        "claude_md_tokens":   claude_md_tokens,
        "session_tokens":     session_tokens,
        "total_input_tokens": total_input,
        "output_tokens":      total_output,
        "cost_per_session":   f"${cost_per_session:.4f}",
        "daily_cost":         f"${daily_cost:.2f}",
        "monthly_cost":       f"${monthly_cost:.2f}",
        "optimization_tip": (
            "CLAUDE.md exceeds 3K tokens — consider splitting "
            "subsections into subdirectory CLAUDE.md files."
        ) if claude_md_tokens > 3000 else "Token budget looks healthy.",
    }


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--claude-md", type=pathlib.Path, default="CLAUDE.md")
    parser.add_argument("--session-log", type=pathlib.Path, default=None)
    parser.add_argument("--sessions-per-day", type=int, default=50)
    args = parser.parse_args()

    result = estimate_session(
        args.claude_md,
        args.session_log,
        sessions_per_day=args.sessions_per_day,
    )
    for k, v in result.items():
        print(f"{k:25s}: {v}")

Claude will confidently apply outdated information — an old database schema, a deprecated API version, a convention your team abandoned three sprints ago — as if it were current. Reusable Context Design Patterns

Context files written ad-hoc for a single project are an improvement over no context. But the real productivity multiplier comes from treating context as a designed, reusable system — patterns and templates that new projects inherit, conventions that are enforced organisation-wide, and a governance model that keeps context accurate and authoritative.

1
The Stack Config Pattern
One CLAUDE.md per microservice (stack-specific, deployment-specific, service-specific security rules) backed by a shared root CLAUDE.md for org-wide invariants. The root file is the single source of truth for organisation standards; service files inherit it and extend without repeating. Teams pull the root file from the central config repo — they don't write org rules locally.
2
The Environment Switcher
Use conditional blocks in CLAUDE.md to differentiate behaviour by environment. For example: in dev, Claude can suggest DROP TABLE with a safety comment; in prod scripts, any destructive SQL is an automatic BLOCKED finding. Include environment detection instructions so Claude knows which block applies to the current context.
3
The Security Layer
Maintain security rules in a dedicated .claude/security.md file, included by reference in each project CLAUDE.md. This separation means the security team owns and reviews security context independently of application teams. When a security rule changes (new OWASP guidance, new compliance requirement), it changes in one place and propagates to all projects on next sync.
4
The Onboarding Template
A CLAUDE.md template that new projects clone from the organisation's scaffold repo. The template ships with the full structure pre-populated — Stack, Commands, Architecture, Conventions, Security, Banned Patterns, PR Checklist — with placeholder content teams replace on day one. This means every new service starts with correct context architecture, not a blank file.

Context Governance Model

CLAUDE.md files are source code. They require the same governance controls as application code — and the consequences of ignoring this are equally serious.

# Context Governance Policy — Acme Engineering

## Ownership
- org-root CLAUDE.md: owned by Platform Engineering (platform-eng@acme.com)
- skills/security/*.md: owned by Security Team (appsec@acme.com)
- skills/*.md: owned by Platform Engineering with contributions from any team
- templates/*.md: owned by Platform Engineering, reviewed quarterly
- services/{name}/CLAUDE.md: owned by service team, reviewed on major changes

## Change Process
1. All changes to shared context (root, skills, security) require:
   - PR with description of why the constraint/convention is changing
   - Review from file owner (CODEOWNERS enforced)
   - Testing: run validate-context.py on changed files
   - Communication: changelog entry + #eng-platform Slack notification

2. Service CLAUDE.md changes:
   - PR review by at least one team member
   - Validate: does it still include @include ../../security/owasp-rules.md ?
   - Validate: does it still include all required sections from the template?

## Validation (CI enforced)
# .github/workflows/validate-context.yml
# Runs on every PR that touches any *.md in claude-context/

python scripts/validate-context.py --file $CHANGED_FILE --schema schemas/claude-md.json

# Schema checks:
# - Required sections present: Stack, Commands, Security Rules, Banned Patterns
# - No hardcoded secrets (regex scan)
# - Security include present: @include ../../security/owasp-rules.md
# - Token count within budget: warn if > 4000 tokens

## Deprecation
When removing a constraint or convention:
1. Add deprecation comment: "# DEPRECATED: removing in 2025-Q3, replaced by X"
2. Keep for one quarter with the deprecation notice
3. Remove in the following quarter, with migration guide in PR description
