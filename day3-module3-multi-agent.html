01

Why Multi-Agent Systems

Single-agent architectures hit three hard walls as enterprise tasks grow in scope. Recognising exactly which wall you've hit — and only then reaching for multi-agent — is the difference between a system that scales and one that's just expensive and complex for no gain.

Wall 1 — Context Ceiling
200 K
Even a 200 K-token context window cannot hold a 50-file codebase, its test suite, its docs, and a long conversation simultaneously. Tasks that exceed the window require decomposition, not heroism.
Wall 2 — Sequential Bottleneck
1×
A single agent processes subtasks in series. If your report has three independent sections — market analysis, technical review, risk assessment — they queue behind each other at 1× speed. That's wasteful.
Wall 3 — No Specialisation
Generalist
A generalist system prompt is a compromise. An agent told to do everything well does nothing great. Specialist agents with narrow, deep expertise consistently outperform the generalist on domain tasks.

The Three Drivers — When to Go Multi-Agent

Use multi-agent architectures when one or more of these is true:

  • Scale: The task's total token footprint exceeds a single context. Example: analysing 200 contracts — each agent handles a batch.
  • Parallelism: Subtasks are independent and can run simultaneously. Example: researching three market segments in parallel rather than in series.
  • Specialisation: Different subtasks genuinely benefit from different model tiers, temperatures, or system prompts. Security auditing needs a different mindset than documentation writing.
⚠
When NOT to Use Multi-Agent
Multi-agent is not the default upgrade for all tasks. Avoid it when: (1) the task is simple and linear — coordination overhead will dwarf the benefit; (2) strict ordering is required — if step B always depends on step A's exact output, parallelism cannot help; (3) you need deterministic, reproducible ordering — network timing makes parallel agents non-deterministic. Start with a single agent. Reach for multi-agent only when you've hit a concrete wall.

✓
The Cost Equation
Three parallel Haiku agents can outperform one Sonnet agent at substantially lower cost for the right decomposition. Haiku runs at roughly 1/20th the per-token cost of Sonnet. If the subtask is focused and doesn't require deep reasoning, Haiku is the default choice for workers. Reserve Sonnet for orchestration and complex synthesis. Reserve Opus for tasks where quality is the only metric that matters.

Real-world trigger: "Our report generation pipeline was taking 4 minutes sequentially. Three parallel Haiku workers got the same quality output in 90 seconds, and the API bill dropped by 60%." That is the conversation that drives enterprise multi-agent adoption.

02

Orchestrator-Worker Pattern

The orchestrator-worker pattern is the canonical multi-agent topology. One orchestrator — running a capable model like Sonnet or Opus — owns the task decomposition, work assignment, result collection, and final synthesis. Workers are narrow, fast executors that do exactly one thing and return structured output.

Responsibility Split

Orchestrator
Sonnet
Decomposes the task into subtasks. Assigns each subtask to a worker. Validates worker output. Handles worker failures. Synthesises results into the final deliverable.
Workers
Haiku
Receive a single structured task. Execute it with full focus. Return a structured JSON result. Report any errors clearly. Never communicate with other workers directly.

Communication Protocol

All orchestrator-to-worker communication is structured JSON. This is non-negotiable. Unstructured natural language between agents creates ambiguity that compounds — a misread instruction in worker A affects the orchestrator's synthesis and therefore every other worker's context. Use strict schemas.

orchestrator_worker.py — complete implementation 
import anthropic
import asyncio
import json
import uuid
from dataclasses import dataclass, field
from typing import Any, Optional
from datetime import datetime, timezone

client = anthropic.Anthropic()

# ─── Data models ───────────────────────────────────────────────────────────────

@dataclass
class Task:
    task_id: str
    worker_capability: str          # must match a registered worker capability
    instruction: str                # the subtask prompt for the worker
    context: dict[str, Any] = field(default_factory=dict)
    max_retries: int = 3
    attempt: int = 0

@dataclass
class TaskResult:
    task_id: str
    success: bool
    output: Any                     # structured JSON from the worker
    error: Optional[str] = None
    latency_ms: int = 0
    model: str = ""
    input_tokens: int = 0
    output_tokens: int = 0

# ─── Worker ────────────────────────────────────────────────────────────────────

class Worker:
    """Narrow executor: receives one task, returns structured JSON."""

    SYSTEM_PROMPT = """You are a specialist worker in a multi-agent pipeline.
You will receive a single, focused task. Your job:
1. Execute the task precisely.
2. Return ONLY valid JSON — no commentary, no markdown fences.
3. If you cannot complete the task, return:
   {"success": false, "error": "clear description of why"}
4. On success return:
   {"success": true, "result": <your structured output>}

Never ask clarifying questions. Never explain your reasoning outside the JSON.
Your output will be parsed programmatically."""

    def __init__(self, model: str = "claude-haiku-4-5"):
        self.model = model

    async def execute(self, task: Task) -> TaskResult:
        start = datetime.now(timezone.utc)
        prompt = json.dumps({
            "task_id": task.task_id,
            "instruction": task.instruction,
            "context": task.context
        }, indent=2)

        try:
            response = await asyncio.to_thread(
                client.messages.create,
                model=self.model,
                max_tokens=2048,
                system=self.SYSTEM_PROMPT,
                messages=[{"role": "user", "content": prompt}]
            )
            latency = int((datetime.now(timezone.utc) - start).total_seconds() * 1000)
            raw = response.content[0].text.strip()
            output = json.loads(raw)
            return TaskResult(
                task_id=task.task_id,
                success=output.get("success", False),
                output=output.get("result"),
                error=output.get("error"),
                latency_ms=latency,
                model=self.model,
                input_tokens=response.usage.input_tokens,
                output_tokens=response.usage.output_tokens,
            )
        except json.JSONDecodeError as e:
            return TaskResult(task.task_id, success=False, output=None,
                               error=f"Worker returned invalid JSON: {e}")
        except Exception as e:
            return TaskResult(task.task_id, success=False, output=None,
                               error=str(e))

# ─── Orchestrator ──────────────────────────────────────────────────────────────

class Orchestrator:
    """Decomposes tasks, dispatches workers, synthesises results."""

    SYSTEM_PROMPT = """You are an orchestrator in a multi-agent system.
Your worker fleet has these capabilities (from AGENTS.md):
- market_analyst:    Analyses market data, sizing, trends.
- competitor_mapper: Maps competitive landscape and differentiators.
- risk_assessor:     Identifies regulatory and operational risks.

When asked to produce a research report you MUST:
1. Decompose the request into exactly those subtasks that map to worker capabilities.
2. Return a JSON task list — no other output:
   {"tasks": [{"task_id": "...", "worker_capability": "...", "instruction": "..."}]}
3. After receiving worker results, synthesise them into a coherent markdown report.
Return ONLY the markdown — no JSON wrapper."""

    def __init__(self, worker: Worker, model: str = "claude-sonnet-4-5"):
        self.worker = worker
        self.model = model

    async def decompose(self, user_request: str) -> list[Task]:
        response = await asyncio.to_thread(
            client.messages.create,
            model=self.model,
            max_tokens=1024,
            system=self.SYSTEM_PROMPT,
            messages=[{"role": "user", "content": user_request}]
        )
        data = json.loads(response.content[0].text)
        return [Task(**t) for t in data["tasks"]]

    async def run_with_retry(self, task: Task) -> TaskResult:
        for attempt in range(task.max_retries):
            task.attempt = attempt + 1
            result = await self.worker.execute(task)
            if result.success:
                return result
            if attempt < task.max_retries - 1:
                # Exponential backoff before retry
                await asyncio.sleep(2 ** attempt)
        # All retries exhausted — escalate to human
        result.error = f"[ESCALATE] Task {task.task_id} failed after {task.max_retries} attempts: {result.error}"
        return result

    async def synthesise(self, request: str, results: list[TaskResult]) -> str:
        payload = {r.task_id: r.output for r in results if r.success}
        response = await asyncio.to_thread(
            client.messages.create,
            model=self.model,
            max_tokens=4096,
            system=self.SYSTEM_PROMPT,
            messages=[
                {"role": "user", "content": request},
                {"role": "assistant", "content": '{"tasks": []}'},
                {"role": "user", "content": f"Worker results:\n{json.dumps(payload, indent=2)}\n\nNow write the final report."}
            ]
        )
        return response.content[0].text

    async def run(self, user_request: str) -> str:
        print("[orchestrator] decomposing task...")
        tasks = await self.decompose(user_request)
        print(f"[orchestrator] {len(tasks)} subtasks created, dispatching workers...")

        # Parallel dispatch with retry
        results = await asyncio.gather(
            *[self.run_with_retry(t) for t in tasks],
            return_exceptions=False
        )

        failed = [r for r in results if not r.success]
        if failed:
            print(f"[orchestrator] {len(failed)} worker(s) failed — escalating")
            for f in failed:
                print(f"  FAIL {f.task_id}: {f.error}")

        return await self.synthesise(user_request, results)

# ─── Entry point ───────────────────────────────────────────────────────────────

async def main():
    worker = Worker(model="claude-haiku-4-5")
    orch   = Orchestrator(worker=worker, model="claude-sonnet-4-5")
    report = await orch.run(
        "Produce a market research report on the global EV battery market in 2025."
    )
    print(report)

if __name__ == "__main__":
    asyncio.run(main())
ℹ
The Orchestrator Prompt Must Name the Worker Fleet
The orchestrator can only decompose tasks into subtasks that match real worker capabilities. Reference your AGENTS.md directly in the orchestrator system prompt so the model knows what it can delegate. If an orchestrator tries to assign a task to a capability that doesn't exist, the whole pipeline stalls. Keep AGENTS.md as the single source of truth for worker capability names.

Parallel Execution Patterns

True parallelism in Python multi-agent systems is achieved with asyncio.gather(). Each agent call is dispatched to asyncio.to_thread() (since the Anthropic SDK is synchronous), allowing multiple Claude API calls to be in-flight simultaneously.

Complete Async Parallel Research Agent

parallel_research.py — 3 agents in parallel 
import anthropic
import asyncio
import time
from dataclasses import dataclass
from typing import Optional

client = anthropic.Anthropic()

@dataclass
class ResearchTask:
    agent_id: str
    topic: str
    focus: str           # the specific angle this agent covers
    word_limit: int = 300

@dataclass
class ResearchResult:
    agent_id: str
    topic: str
    focus: str
    content: str
    latency_ms: int
    input_tokens: int
    output_tokens: int
    error: Optional[str] = None

RESEARCH_SYSTEM = """You are a specialist research agent.
You will receive a research topic and a specific focus area.
Write a concise, factual, well-structured analysis.
Stick to the word limit. Output plain text only — no headers, no markdown."""

async def run_research_agent(
    task: ResearchTask,
    progress_event: asyncio.Event,
    results_store: dict,
) -> ResearchResult:
    """Single agent coroutine — runs in a thread pool to avoid blocking the event loop."""
    start = time.monotonic()

    try:
        response = await asyncio.to_thread(
            client.messages.create,
            model="claude-haiku-4-5",
            max_tokens=1024,
            system=RESEARCH_SYSTEM,
            messages=[{
                "role": "user",
                "content": (
                    f"Topic: {task.topic}\n"
                    f"Your focus: {task.focus}\n"
                    f"Word limit: {task.word_limit} words"
                )
            }]
        )
        latency_ms = int((time.monotonic() - start) * 1000)
        result = ResearchResult(
            agent_id=task.agent_id,
            topic=task.topic,
            focus=task.focus,
            content=response.content[0].text,
            latency_ms=latency_ms,
            input_tokens=response.usage.input_tokens,
            output_tokens=response.usage.output_tokens,
        )
    except Exception as e:
        result = ResearchResult(
            agent_id=task.agent_id, topic=task.topic, focus=task.focus,
            content="", latency_ms=0, input_tokens=0, output_tokens=0,
            error=str(e)
        )

    # Store result and signal progress
    results_store[task.agent_id] = result
    progress_event.set()
    return result

async def progress_monitor(
    total: int,
    results_store: dict,
    progress_event: asyncio.Event
):
    """Emit progress updates as agents complete."""
    completed = 0
    while completed < total:
        await progress_event.wait()
        progress_event.clear()
        completed = len(results_store)
        print(f"[progress] {completed}/{total} agents done")

async def aggregate_results(results: list[ResearchResult], topic: str) -> str:
    """Orchestrator synthesises 3 parallel outputs into one coherent answer."""
    sections = "\n\n".join(
        f"=== {r.focus} ===\n{r.content}"
        for r in results if not r.error
    )
    response = await asyncio.to_thread(
        client.messages.create,
        model="claude-sonnet-4-5",
        max_tokens=4096,
        messages=[{
            "role": "user",
            "content": (
                f"You are synthesising three research sections on: {topic}\n\n"
                f"{sections}\n\n"
                "Write a unified, well-structured research report in markdown. "
                "Eliminate redundancy. Ensure logical flow between sections."
            )
        }]
    )
    return response.content[0].text

async def parallel_research_pipeline(topic: str) -> str:
    tasks = [
        ResearchTask("agent-market",     topic, "Market size, growth rate, and key players"),
        ResearchTask("agent-tech",       topic, "Technology landscape and innovation trends"),
        ResearchTask("agent-regulatory", topic, "Regulatory environment and compliance risks"),
    ]

    results_store: dict = {}
    progress_event = asyncio.Event()

    # Launch all agents + progress monitor concurrently
    agent_coros = [
        run_research_agent(t, progress_event, results_store) for t in tasks
    ]
    monitor_coro = progress_monitor(len(tasks), results_store, progress_event)

    results, _ = await asyncio.gather(
        asyncio.gather(*agent_coros),
        monitor_coro,
        return_exceptions=False
    )

    # Log token budget
    total_in  = sum(r.input_tokens  for r in results)
    total_out = sum(r.output_tokens for r in results)
    max_lat   = max(r.latency_ms   for r in results)
    print(f"[budget] input={total_in} output={total_out} wall_clock={max_lat}ms")

    return await aggregate_results(results, topic)

if __name__ == "__main__":
    report = asyncio.run(parallel_research_pipeline(
        "Global solid-state battery market 2025"
    ))
    print(report)

Rate Limit Awareness

Parallel calls hit the API concurrently. If you launch 10 agents simultaneously and each sends a large input, you can breach your requests-per-minute or tokens-per-minute limit in a single burst. Build in exponential backoff with jitter:

backoff_wrapper.py — rate-limit-safe API call 
import asyncio
import random
import anthropic
from anthropic import RateLimitError

async def call_with_backoff(fn, *args, max_retries=5, base_delay=1.0, **kwargs):
    """Wrap any synchronous Anthropic call with async exponential backoff + jitter."""
    for attempt in range(max_retries):
        try:
            return await asyncio.to_thread(fn, *args, **kwargs)
        except RateLimitError:
            if attempt == max_retries - 1:
                raise
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.5)
            print(f"[rate_limit] attempt {attempt+1} — sleeping {delay:.1f}s")
            await asyncio.sleep(delay)

# Usage:
# response = await call_with_backoff(
#     client.messages.create,
#     model="claude-haiku-4-5", max_tokens=512,
#     messages=[{"role": "user", "content": task.instruction}]
# )
⚠
Model the Token Budget Before Deploying
Parallel execution multiplies token usage. If 5 workers each use 10 K input tokens, one pipeline run consumes 50 K tokens — plus the orchestrator's synthesis pass. For a system that runs 100 times a day, that's 5 M tokens per day from workers alone. Run the arithmetic against your tier limits and cost targets before going to production. The practical sweet spot for parallel workers is 5–10. Beyond that, coordination overhead (orchestrator context, synthesis latency) begins to dominate.

Specialist Agent Design

The most common mistake in multi-agent design is building generalist workers. A worker told to "help with backend tasks" will produce mediocre output on everything. A worker told to "find SQL injection vulnerabilities in Django ORM queries and explain each one with its CVE class and remediation" will be exceptionally good at that one thing.

Design principle: one specialist = one capability. The specialist's system prompt is its entire definition. Keep it narrow, keep it deep.

Four Production Specialist System Prompts

🔒 Security Auditor Sonnet
Requires Sonnet — attack reasoning is complex. See prompt below.
🗄 SQL Optimizer Haiku
Haiku is sufficient — SQL optimisation is pattern-matching, not deep reasoning.
🔌 API Designer Haiku
REST/OpenAPI schema generation is formulaic once the domain model is provided.
📝 Documentation Writer Haiku
Fast, cheap, excellent at structured prose — ideal fit for Haiku.

Human-in-the-Loop Approval Workflows

The fastest way to destroy trust in an AI system is to let it execute an irreversible action without a human checkpoint. Deleting production data, sending external emails, deploying to prod, making financial transactions — these require a human gate, full stop. The approval gate is not a sign of distrust in the AI; it is a sign of mature system design.

Escalation Rules — Define Programmatically

approval_policy.py — action categorisation 
from enum import Enum

class ApprovalTier(str, Enum):
    AUTO     = "auto"       # executes immediately
    ASYNC    = "async"     # notify + wait for approval (async, non-blocking)
    SYNC     = "sync"      # block pipeline until approved
    DENY     = "deny"      # never permitted — hard-coded refusal

# Map action category → approval tier
APPROVAL_POLICY: dict[str, ApprovalTier] = {
    # Read-only — always auto
    "read_database":           ApprovalTier.AUTO,
    "read_file":               ApprovalTier.AUTO,
    "query_api_readonly":      ApprovalTier.AUTO,

    # Low-risk writes — auto in dev, async in prod
    "write_internal_record":   ApprovalTier.AUTO,
    "send_internal_slack":     ApprovalTier.AUTO,

    # Medium-risk — async human approval
    "send_external_email":     ApprovalTier.ASYNC,
    "update_customer_record":  ApprovalTier.ASYNC,
    "create_pr":               ApprovalTier.ASYNC,

    # High-risk — synchronous human approval required
    "deploy_to_production":    ApprovalTier.SYNC,
    "delete_records":          ApprovalTier.SYNC,
    "financial_transaction":   ApprovalTier.SYNC,
    "send_mass_email":         ApprovalTier.SYNC,

    # Hard deny — never execute regardless of instruction
    "drop_table":              ApprovalTier.DENY,
    "delete_all_records":      ApprovalTier.DENY,
    "modify_audit_log":        ApprovalTier.DENY,
}

def get_approval_tier(action_category: str) -> ApprovalTier:
    return APPROVAL_POLICY.get(action_category, ApprovalTier.SYNC)  # default: require approval

The Approval Gate — Complete Implementation

01
Agent generates action plan
The agent proposes an action with full detail: what it will do, why, expected outcome, and rollback procedure.
02
Action logged to pending_actions store
Persisted with a UUID, timestamp, expiry, and idempotency key. Status: PENDING.
03
Notification sent with approve / reject links
Slack message or email containing signed URLs. The approver clicks — no separate login required.
04
Approval endpoint fires
POST /approve/{action_id} or /reject/{action_id}. Updates status, triggers execution or cancellation.

Timeout escalation
If no decision within N hours, a background job auto-cancels and pages on-call. Never auto-approves on timeout.
approval_gateway.py — FastAPI approval endpoint pattern 
import asyncio
import hmac, hashlib, os, uuid
from datetime import datetime, timedelta, timezone
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import Callable, Optional

app = FastAPI()
SECRET = os.environ["APPROVAL_HMAC_SECRET"]
TIMEOUT_HOURS = 4

# In production: replace with Postgres / Redis
pending_actions: dict[str, dict] = {}
# Asyncio events: keyed by action_id, set when approval decision arrives
approval_events: dict[str, asyncio.Event] = {}

class ActionProposal(BaseModel):
    action_category: str
    description: str
    payload: dict
    proposed_by: str         # agent_id
    rollback_plan: str

def sign_action_id(action_id: str) -> str:
    """HMAC-sign the action_id so approve URLs cannot be forged."""
    return hmac.new(SECRET.encode(), action_id.encode(), hashlib.sha256).hexdigest()

async def request_approval(
    proposal: ActionProposal,
    execute_fn: Callable,
    notify_fn: Callable,
) -> bool:
    """
    Store the pending action, notify the approver, and wait for their decision.
    Returns True if approved and executed, False if rejected or timed out.
    """
    action_id = str(uuid.uuid4())
    sig = sign_action_id(action_id)
    expires_at = datetime.now(timezone.utc) + timedelta(hours=TIMEOUT_HOURS)

    pending_actions[action_id] = {
        "proposal": proposal.dict(),
        "status": "PENDING",
        "expires_at": expires_at.isoformat(),
        "idempotency_key": f"{proposal.action_category}:{proposal.proposed_by}:{action_id[:8]}",
    }
    approval_events[action_id] = asyncio.Event()

    base_url = os.environ.get("APPROVAL_BASE_URL", "https://agents.internal")
    approve_url = f"{base_url}/approve/{action_id}?sig={sig}"
    reject_url  = f"{base_url}/reject/{action_id}?sig={sig}"

    await notify_fn(proposal, approve_url, reject_url)

    try:
        await asyncio.wait_for(
            approval_events[action_id].wait(),
            timeout=TIMEOUT_HOURS * 3600
        )
    except asyncio.TimeoutError:
        pending_actions[action_id]["status"] = "TIMEOUT"
        await notify_fn(proposal, "", "", escalate=True)
        return False

    if pending_actions[action_id]["status"] == "APPROVED":
        await execute_fn(proposal.payload)
        return True
    return False

@app.post("/approve/{action_id}")
async def approve_action(action_id: str, sig: str):
    if not hmac.compare_digest(sig, sign_action_id(action_id)):
        raise HTTPException(status_code=403, detail="Invalid signature")
    if action_id not in pending_actions:
        raise HTTPException(status_code=404)
    pending_actions[action_id]["status"] = "APPROVED"
    approval_events[action_id].set()
    return {"status": "approved"}

@app.post("/reject/{action_id}")
async def reject_action(action_id: str, sig: str, reason: Optional[str] = ""):
    if not hmac.compare_digest(sig, sign_action_id(action_id)):
        raise HTTPException(status_code=403, detail="Invalid signature")
    if action_id not in pending_actions:
        raise HTTPException(status_code=404)
    pending_actions[action_id]["status"] = "REJECTED"
    pending_actions[action_id]["rejection_reason"] = reason
    approval_events[action_id].set()
    return {"status": "rejected"}
ℹ
Never Auto-Approve on Timeout
A common mistake is to auto-approve if the approver doesn't respond — "because the work needs to keep moving." This inverts the safety guarantee: you've now built a system where inaction equals permission. Always auto-cancel or auto-escalate on timeout. The cost of one cancelled action is far lower than the cost of one unauthorised destructive action executed in production.
06

Failure Modes & Resilience

Multi-agent systems fail in ways that single-agent systems cannot. The failure surface is larger: more API calls, more network hops, more intermediate state, more opportunities for one bad result to corrupt a downstream synthesis. You must design for failure from the start — it is not an afterthought.

Multi-Agent Failure Taxonomy

Failure Mode Severity Description Mitigation
Worker timeout MEDIUM Worker exceeds max execution time; orchestrator stalls waiting for response. asyncio.wait_for() with hard timeout; retry logic; fallback to partial result.
Invalid output format HIGH Worker returns malformed JSON or wrong schema; orchestrator fails to parse. Schema validation on all worker outputs before orchestrator uses them; reject and retry.
Hallucinated data HIGH Worker fabricates facts (market size, regulatory clauses, code logic) that the orchestrator trusts and includes in final output. Require workers to cite sources or confidence scores; cross-validate critical facts across two workers; red-team workers with known-false inputs.
Cascading failure HIGH One bad worker result contaminates the orchestrator's context and causes incorrect synthesis that infects the final output. Treat each worker result as untrusted until validated; use structured schemas; circuit breaker on repeated failures.
Confused orchestrator MEDIUM Orchestrator receives inconsistent results from parallel workers (e.g., two market size figures that disagree by 10×) and makes an incorrect synthesis decision. Require workers to include confidence/source; orchestrator explicitly resolves conflicts rather than silently averaging.
Rate limit cascade LOW Burst of parallel workers triggers rate limits; all fail simultaneously with 429. Exponential backoff with jitter; stagger parallel launches with small initial delays; monitor TPM usage.

Complete Resilience Pattern

resilience.py — retry decorator + circuit breaker + schema validation 
import asyncio
import functools
import json
import logging
import random
import time
from collections import deque
from dataclasses import dataclass, field
from typing import Any, Callable, Optional, Type

logger = logging.getLogger(__name__)

# ─── Retry decorator ─────────────────────────────────────────────────────────

def async_retry(
    max_attempts: int = 3,
    base_delay: float = 1.0,
    retryable_exceptions: tuple = (Exception,),
):
    """Async retry with exponential backoff and full jitter."""
    def decorator(fn):
        @functools.wraps(fn)
        async def wrapper(*args, **kwargs):
            for attempt in range(max_attempts):
                try:
                    return await fn(*args, **kwargs)
                except retryable_exceptions as e:
                    if attempt == max_attempts - 1:
                        logger.error(f"[retry] {fn.__name__} exhausted {max_attempts} attempts")
                        raise
                    delay = random.uniform(0, base_delay * (2 ** attempt))
                    logger.warning(f"[retry] {fn.__name__} attempt {attempt+1} failed: {e}. Sleeping {delay:.2f}s")
                    await asyncio.sleep(delay)
        return wrapper
    return decorator

# ─── Circuit breaker ─────────────────────────────────────────────────────────

@dataclass
class CircuitBreaker:
    """
    Tracks failure rate for a named worker/service.
    Opens (stops sending work) after failure_threshold failures
    in the rolling window_size call history.
    """
    name: str
    failure_threshold: int = 5       # open after this many failures
    window_size: int = 10            # rolling window of last N calls
    reset_timeout: float = 60.0      # seconds before attempting half-open

    _history: deque = field(default_factory=lambda: deque(maxlen=10))
    _opened_at: Optional[float] = field(default=None)

    @property
    def is_open(self) -> bool:
        if self._opened_at is None:
            return False
        if time.monotonic() - self._opened_at > self.reset_timeout:
            logger.info(f"[circuit_breaker:{self.name}] half-open — allowing one probe")
            return False  # allow one probe through
        return True

    def record_success(self):
        self._history.append(True)
        self._opened_at = None

    def record_failure(self):
        self._history.append(False)
        failures = sum(1 for x in self._history if not x)
        if failures >= self.failure_threshold:
            if self._opened_at is None:
                logger.error(f"[circuit_breaker:{self.name}] OPEN — {failures} failures in last {len(self._history)} calls")
            self._opened_at = time.monotonic()

    async def call(self, fn: Callable, *args, **kwargs) -> Any:
        if self.is_open:
            raise RuntimeError(f"Circuit breaker OPEN for {self.name} — not sending work")
        try:
            result = await fn(*args, **kwargs)
            self.record_success()
            return result
        except Exception:
            self.record_failure()
            raise

# ─── Output schema validation ────────────────────────────────────────────────

from pydantic import BaseModel, ValidationError

class WorkerOutput(BaseModel):
    success: bool
    result: Optional[Any] = None
    error: Optional[str] = None
    confidence: float = 1.0          # 0.0–1.0; workers should report uncertainty
    sources: list[str] = []          # citations or data sources used

def validate_worker_output(raw_text: str) -> WorkerOutput:
    """Parse and validate a worker's raw JSON response. Raises on schema violation."""
    try:
        data = json.loads(raw_text)
        return WorkerOutput(**data)
    except json.JSONDecodeError as e:
        raise ValueError(f"Worker returned non-JSON: {e}. Raw: {raw_text[:200]}")
    except ValidationError as e:
        raise ValueError(f"Worker output failed schema validation: {e}")

# ─── Observability logger ────────────────────────────────────────────────────

import hashlib

def log_agent_call(
    agent_id: str,
    task_id: str,
    input_text: str,
    output_text: str,
    latency_ms: int,
    model: str,
    input_tokens: int,
    output_tokens: int,
    cost_usd: float,
):
    """Emit a structured log entry for every agent call. Ship to your SIEM."""
    entry = {
        "ts": time.time(),
        "agent_id": agent_id,
        "task_id": task_id,
        "input_hash": hashlib.sha256(input_text.encode()).hexdigest()[:16],
        "output_hash": hashlib.sha256(output_text.encode()).hexdigest()[:16],
        "latency_ms": latency_ms,
        "model": model,
        "input_tokens": input_tokens,
        "output_tokens": output_tokens,
        "cost_usd": cost_usd,
    }
    # In production: ship to Datadog / Splunk / CloudWatch
    logger.info(json.dumps(entry))
⚠
Design All Agent Actions as Idempotent
When a worker fails and is retried, the retry must be safe to execute even if the original attempt partially succeeded. Use idempotency keys — a deterministic hash of the action category, agent ID, and input hash — and check the key against a persistent store before executing. If the key exists and the previous result was successful, return the cached result without re-executing. This pattern eliminates duplicate sends, duplicate writes, and duplicate API calls caused by network retries.

Observability Checklist — What to Log per Agent Call

  • agent_id — which specialist or orchestrator made the call
  • task_id — links to the parent orchestrator run
  • input_hash — SHA-256 prefix of input (for deduplication, not PII)
  • output_hash — SHA-256 prefix of output (for change detection)
  • latency_ms — wall-clock time for this agent call
  • model — which model was used (Haiku / Sonnet / Opus)
  • input_tokens / output_tokens — for cost attribution
  • cost_usd — derived from token counts × model pricing
  • circuit_breaker_state — CLOSED / HALF_OPEN / OPEN at call time
  • retry_attempt — 0 for first attempt; signals systemic instability if > 0 is frequent

Aggregate these logs into your SIEM and create alerts for: cost per pipeline run exceeding budget threshold, circuit breaker opening on any worker, retry rate exceeding 5%, and orchestrator synthesis latency exceeding SLO.
