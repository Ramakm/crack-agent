✦ Day 4 · Capstone

Capstone Project

Enterprise AI Coding Assistant Platform: Design and present a complete, production-ready AI coding assistant deployment for a 200-person engineering organisation — covering Claude model selection, MCP integrations, multi-agent workflows, CI/CD integration, security controls, compliance posture, and cost model.

3 Hours Design 10-Minute Presentation Architecture + Governance + Cost
01

Project Brief

ℹ
The Challenge
You are the AI Platform Lead at a 200-person engineering company. The CTO has approved a budget to deploy an Enterprise AI Coding Assistant Platform. You have 3 hours to design the platform and 10 minutes to present it to the Engineering Leadership Team (CTO, VP Engineering, CISO, and Head of Platform).

Company Context

👥
Team Size
200 engineers: 120 backend (Python/Go), 50 frontend (React/TypeScript), 20 data/ML, 10 platform/SRE
🏗️
Stack
Microservices on AWS EKS, PostgreSQL RDS, Redis, GitHub for SCM, Jira for PM, Slack for comms
📋
Compliance
SOC2 Type II certified, GDPR applicable (EU customers), handling PII, considering HIPAA in 2026
💰
Budget
$15,000/month maximum for AI API costs. One-time infrastructure setup budget: $50,000.

What Your Design Must Address

  1. Tool access model — which Claude deployment for which engineer roles; MCP server inventory and permissions
  2. Governance model — usage policies, approval workflows for high-risk operations, audit logging
  3. Multi-agent workflows — at least 2 specific agentic workflows that deliver measurable engineering productivity
  4. CI/CD integration — how Claude participates in the SDLC (PR review, test generation, deployment gates)
  5. Cost controls — monthly cost projection, per-tier allocation, budget alert configuration
  6. Security & compliance — how you satisfy SOC2 and GDPR; your prompt injection mitigations
📐
Deliverable 1
Architecture diagram (ASCII or drawn) showing Claude deployments, MCP servers, data flows, and human touchpoints
📄
Deliverable 2
Governance policy document — one page covering permitted uses, prohibited uses, approval workflows, and incident response
💵
Deliverable 3
Monthly cost estimate with per-feature breakdown, model tier rationale, and budget alert threshold
🛠️
Deliverable 4
Demo or detailed spec of one MCP server or multi-agent workflow — enough for a senior engineer to implement it in one sprint
02

Architecture Document Template

Complete each section of this template. The guiding questions are prompts to ensure your design is specific enough to be actionable — not a collection of vague aspirations.

1
Executive Summary

2-3 sentences: what the platform does, how many engineers it serves, and the primary productivity outcome you are optimising for.

Guiding questions: What is the single most valuable workflow this platform enables? How will you measure success in the first 90 days?

2
System Architecture

Architecture diagram + written description covering: Claude deployment model, MCP server topology, network zones, and human touchpoints. Reference the architecture patterns from Day 3 Module 4 and Day 4 Modules 1-3.

Guiding questions: Where does Claude run — on developer laptops (Claude Code) or as a shared service? How do your MCP servers communicate with internal systems without public internet exposure?

3
Claude Model Selection Rationale

Table mapping each use case to the appropriate Claude model tier. Justify every Sonnet or Opus choice — if you could use Haiku for the task, use Haiku.

Guiding questions: Which tasks are high-volume, simple classification or extraction (Haiku)? Which require sustained reasoning (Sonnet)? Is there anything requiring deep synthesis that truly needs Opus?

4
MCP Server Inventory

Table of every MCP server: name, type (official / custom), tools exposed, auth method, transport, and which engineer roles have access. Include the permission scoping for each server.

Guiding questions: Which servers need read-write access vs read-only? Which servers should require human approval for every write operation? Are any of these servers shared across the org vs per-team?

5
Multi-Agent Workflow Design

Describe at least 2 multi-agent workflows in detail: orchestrator, worker agents, model tiers, failure modes, and the human touchpoints where approval is required.

Guiding questions: What is the trigger for each workflow? What are the inputs and outputs? At which step can an agent produce an incorrect result that would cause the largest downstream impact?

6
CI/CD Integration Points

Map Claude participation into the SDLC: which GitHub Actions workflows use Claude, what triggers them, what they produce, and how results gate the pipeline.

Guiding questions: Is AI-generated PR review blocking or advisory? Who can override an AI-flagged concern? How do you prevent the CI pipeline from being a prompt injection attack vector?

7
Security Controls

Address each of the 15 security checklist items from Module 4. Mark which apply to your deployment and how you address them.

Guiding questions: What is your indirect prompt injection mitigation strategy? How do you handle secrets rotation for 200 engineers? What is the incident response playbook if an injection attack succeeds?

8
Compliance Posture

Map your deployment controls to GDPR, SOC2, and (if applicable) HIPAA requirements. List any gaps and your mitigation plan.

Guiding questions: Is Anthropic signed as a Data Processor under your GDPR DPA? How do you handle a GDPR erasure request for data that was in a Claude context window? What SOC2 evidence will you provide auditors?

9
Cost Model

Monthly cost estimate with per-feature breakdown. Show input vs output token split. Show how the total fits within the $15,000/month budget. Define the budget alert threshold.

Guiding questions: What is the most expensive feature per token? Which features benefit from prompt caching? What happens to cost if adoption doubles in month 3?

10
Rollout Plan

Phased deployment: pilot (weeks 1-4), limited rollout (weeks 5-12), full deployment (months 4-6). Define success metrics for each phase and the go/no-go criteria for advancing.

Guiding questions: Which team pilots first and why? What is your rollback plan if the pilot reveals security issues? How do you handle the "it changed my code without asking" complaint from engineers?

Example Enterprise Architectures

Three worked examples showing how different constraints produce different architecture decisions. These are not templates to copy — they are reasoning demonstrations.

Example A: Financial Services Firm (SOC2 + GDPR Heavy)

CONSTRAINTS: SOC2 Type II, GDPR, Bloomberg terminal data (non-public market info), human approval required on ALL write operations ARCHITECTURE CHOICES: Model tier: Claude Code for devs (Sonnet default), Haiku for log analysis, Opus reserved for risk assessment reports only MCP Inventory: internal-kb (custom, read-only, VPN-only, per-user OAuth tokens) github (official, PAT scoped to internal repos, read+write with HITL) postgres (official, analytics replica, read-only, claude_readonly user) NO Slack MCP — compliance prohibits AI access to internal comms NO browser MCP — Bloomberg data rules prohibit screen scraping Security posture: All MCP servers behind corporate VPN — not reachable from internet Every tool call requires human approval prompt in Claude Code UI Audit logs → Splunk SIEM → 7-year retention (financial regulation) GDPR DPA signed with Anthropic; data residency: EU region only Cost model: ~$8,200/month total, well within $15k budget Conservative: human-in-the-loop dramatically reduces accidental tool calls

Example B: HIPAA Healthcare Startup

CONSTRAINTS: HIPAA BAA required, PHI de-identification mandatory before LLM, small engineering team (40 engineers), speed over process ARCHITECTURE CHOICES: PHI pipeline: Patient data NEVER enters Claude context directly De-identification service runs before any KB or DB tool result reaches Claude ICD codes, age groups, procedure classes — OK in context Name, DOB, MRN, address, phone — ALWAYS redacted to pseudonym hash MCP Inventory: internal-kb (clinical documentation, custom, de-id pipeline in front) github (official, for code only — no PHI in codebases) postgres (dev/staging only — production DB never connected) Compliance: BAA signed with Anthropic before first deployment All API calls go to Anthropic US region (HIPAA covered) Audit logs → CloudWatch → exported to HIPAA-compliant S3 (encrypted, 6-year) Quarterly security review mandated in BAA exhibit Cost model: ~$4,100/month (small team, conservative use) De-identification service adds ~$300/month infrastructure cost

Example C: SaaS Developer Productivity Focus

CONSTRAINTS: Developer productivity is the primary metric, SOC2 only, no regulated data, 200 engineers, aggressive adoption targets ARCHITECTURE CHOICES: Multi-agent workflows (2 core): Workflow 1: Automated Code Review Pipeline Trigger: PR opened (GitHub Actions) Agent 1 (Haiku): classify PR size and domain (5 categories) Agent 2 (Sonnet): security review focused on OWASP top 10 Agent 3 (Sonnet): logic review focused on CLAUDE.md conventions Orchestrator (Sonnet): merge reviews, post structured comment Human gate: engineer reads comment, acts on REQUEST_CHANGES Workflow 2: Sprint Intelligence Report Trigger: Friday 17:00 cron job Agent 1 (Haiku): read Jira sprint board → extract ticket status Agent 2 (Haiku): read GitHub → PR merge rate, review cycle time Agent 3 (Sonnet): synthesise → identify blockers and at-risk items Output: Slack DM to team leads with summary + recommended actions MCP Inventory: github, jira, slack (post-only, #ai-summaries channel) postgres (analytics replica, read-only) internal-kb (custom Python server, Cloud Run, HTTP+SSE) Cost model: ~$12,400/month — high volume, rich tooling PR review bot: 300 PRs/day × Sonnet = dominant cost driver Budget alert at $14,000 (120% of $11,670 estimate with buffer)

Quick Reference Card

A four-day curriculum in one card. Commit these to memory before your capstone presentation.

Day 1 — Foundations
  • Haiku = classification/extraction. Sonnet = reasoning. Opus = deep synthesis.
  • Enterprise Claude is a data processor — your data is not used for training by default.
  • System prompt = the first and most important security boundary.
  • Rate limits and token quotas are your first cost control levers.
  • Claude for Work seats do not replace API access for agentic workflows.
Day 2 — Claude Code & Dev
  • CLAUDE.md is machine-readable context, not human documentation.
  • Banned patterns section prevents AI from repeating team's most common mistakes.
  • PR diff-based review: always exclude lockfiles and generated code.
  • Output tokens cost 3-5x more than input tokens — minimise response length where possible.
  • Claude Code permission controls: allowedTools, deniedTools, requireApproval.
Day 3 — Context & Multi-Agent
  • Three context layers: persistent (CLAUDE.md), session, injected (RAG/MCP).
  • Multi-agent pattern: orchestrator dispatches workers in parallel, runs sequential synthesis.
  • Per-worker timeout (30s) + partial result return prevents cascade failures.
  • Never let one failing worker stall the pipeline — return what you have.
  • Token cost model: always separate input vs output token counts per feature.
Day 4 — MCP & Security
  • MCP solves N×M integration: one protocol, unlimited server/client interop.
  • stdio for local dev, HTTP+SSE for enterprise shared servers.
  • Tool descriptions are Claude's only signal for which tool to call — write them well.
  • Indirect prompt injection (via tool results) is the primary enterprise AI threat.
  • Never bulk-dump data into context. Send minimum fields needed for the task.
  • Audit every tool call: who, what tool, what args, when, success/failure.
  • GDPR DPA + HIPAA BAA must be signed with Anthropic before handling regulated data.
