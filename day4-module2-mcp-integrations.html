GitHub MCP Integration

The GitHub MCP server is the most widely deployed enterprise MCP integration. It gives Claude direct access to your repositories, pull requests, issues, and code — enabling workflows that previously required custom scripts and API integrations.

Available Tools

🔍
Discovery
search_repositories, list_commits, list_pull_requests, list_issues, search_code
📄
Read
get_file_contents, get_pull_request, get_issue, get_commit
✏️
Write
create_issue, create_pull_request, create_branch, push_files, add_issue_comment

Authentication: PAT vs GitHub App

Auth MethodBest ForScopes NeededEnterprise Consideration
Personal Access Token (PAT) Individual developers, local Claude Code repo read:org Token tied to individual. If developer leaves, token becomes invalid. Rotate every 90 days.
GitHub App Token Enterprise shared MCP servers, CI/CD Installation-scoped per repo Org-owned identity. Fine-grained permissions per installation. Automatic expiry (1h). Preferred for production.
json
// .claude/settings.json — GitHub MCP configuration
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        // For individuals: fine-grained PAT scoped to specific repos
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"

        // For enterprise GitHub App (generate via App installation):
        // "GITHUB_APP_ID": "${GH_APP_ID}",
        // "GITHUB_APP_PRIVATE_KEY": "${GH_APP_KEY}",
        // "GITHUB_APP_INSTALLATION_ID": "${GH_INSTALL_ID}"
      }
    }
  }
}

Real Workflow: PR Review Pipeline

This example shows Claude using multiple GitHub tools in sequence to complete a real engineering task — reviewing all open PRs and summarizing blockers. Claude orchestrates the tool calls autonomously.

USER: "Review all open PRs in acme-org/api-service and tell me what's blocking each one" Step 1 → Claude calls list_pull_requests → Returns 7 open PRs with numbers, titles, authors, labels Step 2 → Claude calls get_pull_request for each PR (in parallel) → Returns diff stats, review status, merge conflicts, CI checks Step 3 → Claude calls list_issues for any referenced issue numbers → Returns linked issue descriptions for context CLAUDE OUTPUT: PR #142 "Add rate limiting to /orders endpoint" Blocked: 2 review changes requested (alice, bob) CI failing: unit tests — 3 failures in OrderService.test.ts Merge conflict: src/middleware/rateLimiter.ts PR #138 "Migrate auth to RS256" Ready: 2 approvals, CI passing, no conflicts Note: merge window closes Friday (release freeze)
⚠
Enterprise GitHub: Scope Your PAT
Create fine-grained PATs scoped to specific repositories only, not all repositories. Grant only the permissions your workflow requires: contents:read for read-only workflows, add pull_requests:write only if you need Claude to create or comment on PRs. Rotate tokens every 90 days and store them in your organization's secrets manager.

Slack MCP Integration

The Slack MCP server connects Claude to your team's communication layer. This enables powerful async workflows — summarising discussions, generating standup digests, searching for decisions buried in thread history, and drafting announcements.

Available Tools

The core tools are: list_channels, post_message, reply_to_thread, list_messages, search_messages, get_channel_info, get_user_info.

Real Workflow: Daily Engineering Standup Digest

PROMPT: "Read the last 24 hours of #engineering, #platform, and #backend. Summarise: what was discussed, any blockers raised, any decisions made." Claude calls list_messages for each channel (oldest=yesterday_timestamp) Claude reads 3 channels in parallel, each returning message objects Claude synthesises across all channels OUTPUT: #engineering (47 messages) Decisions: Merge window for v2.4 closes Friday 18:00 UTC Blockers: @alice flagged flaky integration tests in CI — no owner yet FYI: @bob demoed new observability dashboard — positive reception #platform (12 messages) Decisions: ECS task definition update for Node 20 approved for Thursday Blockers: None #backend (23 messages) Blockers: Postgres replica lag spiking at 09:00 UTC — being investigated FYI: Rate limiter middleware PR ready for review (PR #142)
✗
Never Grant chat:write to Fully Autonomous Agents
A Slack Bot Token with chat:write scope can post to any channel it's been invited to. If you allow Claude to post autonomously without human review, a prompt injection attack or model error could post incorrect or sensitive information to public channels. For autonomous Slack-writing agents, require human approval on all outbound messages, or restrict the bot to a dedicated #ai-assistant channel.

Database MCP Integration

Database MCP servers give Claude the ability to explore and query your data directly. This is transformative for data analysis, schema understanding, and debugging production issues — but requires careful permission scoping to prevent data exfiltration or accidental writes.

Read-Only vs Read-Write Modes

✓
Read-Only Mode (Default)
Claude can call list_tables, describe_table, and query (SELECT only). The database user has no INSERT/UPDATE/DELETE/DROP grants. This is the correct default for all production environments.
⚠
Read-Write Mode (Restricted)
Adds execute tool for INSERT/UPDATE/DELETE. Requires human-in-the-loop approval for every write operation. Should never be granted on production primary database — use a staging environment or a dedicated analytics write-back schema.
bash
# Create a dedicated read-only database user for Claude MCP
# Run on PostgreSQL production replica — NEVER on the primary write instance

CREATE USER claude_readonly WITH PASSWORD '${GENERATED_STRONG_PASSWORD}';

# Grant SELECT on specific schemas only — not all tables
GRANT CONNECT ON DATABASE app_db TO claude_readonly;
GRANT USAGE ON SCHEMA public, analytics TO claude_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO claude_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics TO claude_readonly;

# Explicitly deny access to sensitive tables
REVOKE SELECT ON TABLE users, payment_methods, api_keys FROM claude_readonly;

# Set statement timeout to prevent runaway queries
ALTER USER claude_readonly SET statement_timeout = '30s';
ALTER USER claude_readonly SET lock_timeout = '5s';
json
// Database MCP — always point to analytics replica, never primary
{
  "mcpServers": {
    "database": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://claude_readonly:${DB_READ_PASS}@analytics-replica.internal:5432/app_db"
      ]
    }
  }
}

Real Workflow: Schema Understanding and Query Optimization

A common high-value use case: an engineer pastes a slow query and asks Claude to explain the schema and suggest indexes. Claude uses describe_table on all referenced tables, examines the query, and reasons about missing indexes.

✗
PII Tables Require Audit Logging
If your MCP server has query access to tables containing PII (emails, names, addresses, phone numbers), you must enable audit logging that records every query with the session user, timestamp, and query text. Enable PostgreSQL pgaudit or equivalent. Without audit logging, you cannot demonstrate GDPR data access compliance, and you cannot investigate a data breach.

Jira & Project Management Integrations

The Jira MCP integration brings Claude into your project management workflow — searching backlogs, creating tickets from bug reports, transitioning issue states, and generating sprint reports. The same patterns apply to Linear, Azure DevOps, and GitHub Issues.

Available Tools

Core tools: search_issues (JQL), get_issue, create_issue, update_issue, add_comment, transition_issue, list_sprints, get_board.

json
// Jira MCP configuration — Atlassian API token auth
{
  "mcpServers": {
    "jira": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-jira"],
      "env": {
        "JIRA_URL": "https://acme.atlassian.net",
        "JIRA_EMAIL": "claude-bot@acme.com",       // service account email
        "JIRA_API_TOKEN": "${JIRA_API_TOKEN}",       // from id.atlassian.com/manage-profile/security/api-tokens
        "JIRA_PROJECT_KEY": "PLAT"                  // limit to specific project
      }
    }
  }
}

JQL Query Patterns for Sprint Planning

jql
# Current sprint — all open issues
project = PLAT AND sprint in openSprints() AND status != Done ORDER BY priority DESC

# Blocked issues — high priority with no assignee
project = PLAT AND priority in (High, Critical) AND assignee is EMPTY AND status != Done

# Ready to test — awaiting QA
project = PLAT AND status = "In Review" AND sprint in openSprints()

# Overdue from last sprint — carried over work
project = PLAT AND sprint in closedSprints() AND status != Done AND sprint not in openSprints()

# Issues linked to a specific PR (when GitHub integration is enabled)
project = PLAT AND "GitHub PR" is not EMPTY AND status in ("In Progress", "In Review")

Browser Automation & Google Workspace

Browser automation via Playwright MCP and Google Workspace integrations unlock a class of workflows that go beyond API-first tools. Playwright can interact with any web application that doesn't have an API. Google Workspace gives Claude access to the documents, calendars, and emails that drive enterprise operations.

Playwright MCP: Browser Automation

Tools: navigate, click, fill, screenshot, extract_text, evaluate_js, wait_for_element.

⚠
Run Browser MCP in Docker, Not on Developer Laptops
Playwright MCP running on a developer laptop has access to the developer's browser cookies, saved passwords, and local sessions. In enterprise deployments, run the Playwright MCP server inside a Docker container with no access to host credentials. Mount only the specific site credentials the automation needs, using environment variables, not the host keychain.
dockerfile
# Dockerfile for isolated Playwright MCP server
FROM mcr.microsoft.com/playwright/python:v1.44.0-jammy

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt

# Install only the browser you need (reduces image size)
RUN playwright install chromium --with-deps

COPY playwright_mcp_server.py .

# No volumes from host — no access to host credentials
# Secrets injected at runtime via --env flags
CMD ["python3", "playwright_mcp_server.py"]

Module 2 Complete: Integration Patterns
You now have production patterns for all major enterprise MCP integrations. The key principles: always use service accounts or app tokens in production, never personal credentials. Point database MCP at read-only replicas. Require human approval for any write operation with external side effects. Never run browser MCP on a host with live user sessions.
