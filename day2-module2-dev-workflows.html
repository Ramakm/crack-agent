AI-Assisted Development Workflows

Claude Code's real power isn't autocomplete — it's a repeatable loop that makes every phase of engineering work faster and safer. This module codifies the workflows that senior engineers use in practice: from feature exploration to code review, and everything in between.

Explore → Plan → Code → Verify Debugging Protocols Safe Refactoring TDD & Test Generation AI-Assisted Code Review
01

Feature Development Workflow

Most developers reach for Claude and immediately ask it to write code. This is backwards. The Explore → Plan → Code → Verify loop is the correct mental model — and the reason it works is that Claude's highest-leverage contribution is in the first two phases, not in generating boilerplate.

Explore
Understand the existing system before touching anything. Ask Claude to read and explain the relevant code.
Plan
Draft a concrete implementation plan. Surface risks, dependencies, and tradeoffs before writing a single line.
Code
Implement incrementally — one logical unit at a time — so you can review and correct at each step.
Verify
Run tests, review the diff, and ask Claude to explain its decisions. Never merge without understanding what changed.

Step 1: Explore — Orient Before You Act

Before adding OAuth2 to your application, you need to understand how authentication currently works. Don't ask Claude to "add OAuth2". Ask Claude to explain the current auth system. This is the single most effective habit change you can make when adopting AI-assisted development.

You
/add app/auth/ app/middleware/auth.py app/models/user.py Explain the current authentication system in this codebase. I want to understand: 1. How a request is authenticated end-to-end 2. Where sessions are created and stored 3. What user identity information is persisted 4. Any known limitations or hardcoded assumptions
Claude
Here's what I can see from the three files: Authentication flow: 1. Request hits `AuthMiddleware.before_request()` in middleware/auth.py 2. Middleware reads `session_id` from a signed cookie (using `itsdangerous.TimestampSigner`) 3. Session is looked up in Redis (key: `session:{session_id}`) with a 24h TTL 4. If found, `g.current_user` is populated from the session JSON Session creation: POST /auth/login → validates credentials → creates session dict → writes to Redis The session dict contains: {user_id, email, role, created_at} No refresh token logic — sessions simply expire after 24h Limitations I noticed: - `role` is stored in the session, not re-read from the DB on each request. If a user's role changes, they won't see it until their session expires. - The Redis key prefix is hardcoded as "session:" — no namespace isolation between environments if they share a Redis instance. - No concurrent session limiting — a user can have unlimited active sessions. This matters for OAuth2: Any OAuth2 implementation needs to either reuse this Redis session layer or introduce a parallel identity layer. Touching both simultaneously would be risky.
💡
The Explore step has compounding value
Claude's analysis in the Explore step surfaces constraints that would otherwise become bugs: the role staleness issue, Redis namespace collision, and session layer coupling. You now have a design checklist before writing any code. Most developers discover these things mid-implementation or in a post-deployment incident.

Step 2: Plan — Draft the Architecture First

With the Explore context established, ask Claude to produce a concrete implementation plan. The critical constraint is that you are not asking Claude to write code yet. You want a plan you can review, challenge, and modify.

You
Based on everything you've read, draft an implementation plan for adding Google OAuth2 support. Constraints: - Do NOT modify the existing session layer (Redis + itsdangerous) - The existing login flow must continue to work unchanged - We need to handle the case where a Google account email matches an existing local account Give me the plan as numbered steps, identify which files change, and flag any risks.
Claude
Implementation Plan: Google OAuth2 (additive, no session layer changes) New files: app/auth/oauth.py — Google OAuth2 client (uses authlib) app/auth/routes_oauth.py — /auth/google and /auth/google/callback endpoints Modified files: app/models/user.py — add google_id column (nullable), add classmethod find_or_create_from_google() app/config.py — GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GOOGLE_REDIRECT_URI alembic/versions/ — new migration for google_id column Steps: 1. Add `google_id` nullable column to users table (migration first) 2. Add `User.find_or_create_from_google(profile)` — matches on email if google_id is null, links account. Creates new user if no match. 3. Implement /auth/google → redirects to Google with state param (CSRF protection) 4. Implement /auth/google/callback → validates state, exchanges code for token, fetches profile, calls find_or_create_from_google(), then creates session using the EXISTING session creation logic from routes.py (no duplication) 5. Add GOOGLE_* env vars to config.py and .env.example Risks: - Email collision: a user registers locally with the same email as a Google account. find_or_create_from_google() will silently link accounts. You need to decide if this is the intended behaviour or if it requires user confirmation. - State parameter: must be stored server-side (Redis) or use a signed value. A simple random string in the session cookie is not sufficient for production. - Scope creep: if you later need to access Google Calendar/Drive, the OAuth scopes need to be set correctly now. Adding scopes later forces users to re-authorize.

Step 3: Code — One Logical Unit at a Time

Now that you have an agreed plan, ask Claude to implement one step at a time. Never ask for all five steps at once. The incremental approach means you review each piece, catch errors early, and stay in control of what enters the codebase.

⚠️
The "do it all in one shot" trap
Asking Claude to "implement the entire OAuth2 feature" produces a large, tangled diff that's difficult to review. You lose the ability to catch mistakes early. The plan you just created is your implementation checklist — work through it one step at a time, review and run tests between steps, and only proceed when each piece is correct.
Incremental code prompt example 
# Step 1 only — database migration
"Implement step 1 from the plan: create the Alembic migration to add
the google_id nullable column to the users table. Show me the migration
file only — do not change any application code yet."

# After reviewing and running the migration:
"Good. Now implement step 2: add the google_id column to the User model
and write the find_or_create_from_google() classmethod. Match the style
of the existing classmethods in app/models/user.py."

Step 4: Verify — Never Merge Without Understanding

After each implementation step, run your tests and then ask Claude to explain what it changed and why. This builds team trust, creates an audit trail, and surfaces assumptions Claude made that differ from your intent.

You
Tests pass. Before I commit this, explain what decisions you made in find_or_create_from_google() that weren't explicitly specified in the plan, and why.
Claude
Three decisions I made that weren't explicit in the plan: 1. Case-insensitive email matching I used `User.query.filter(func.lower(User.email) == func.lower(profile['email']))`. Google normalizes email to lowercase but some existing users may have registered with mixed case. This prevents duplicate accounts for the same person. 2. Updating google_id on link When an existing user is matched by email, I update their google_id column. This means subsequent Google logins will use google_id lookup (faster, unambiguous) rather than email lookup. The alternative — always matching by email — is slower and fails if a user later changes their Google account email. 3. No auto-verification of email I did NOT set `email_verified = True` on newly linked accounts. Even though Google guarantees verified emails, your existing users table doesn't have this column, so I left it out rather than assume your verification semantics.

These three explanations are exactly the kind of information a code reviewer needs. By asking for them explicitly, you've turned a black-box code generation into a documented engineering decision log.

Systematic Debugging with Claude

Debugging with Claude is most effective when you apply the same discipline a good senior engineer would: reproduce the issue reliably, isolate the cause, then fix it. Skipping directly to "here's my error, fix it" produces low-quality patches because Claude lacks the context it needs to understand the root cause.

1
Reproduce — Give Claude a reliable trigger
Provide the exact request, input data, or sequence of actions that reliably triggers the bug. Include the full stack trace, not just the last line of the error. A bug that "sometimes happens" needs a reproduction recipe before Claude can help effectively.
2
Isolate — Point Claude at the right files
Use /add to bring in only the files relevant to the error path. Don't dump the entire codebase. The stack trace tells you which files to add. Context precision matters: too little and Claude misses the cause; too much and the signal-to-noise ratio drops.
3
Fix — Hypothesis first, then code
Always ask Claude to explain its hypothesis before generating a fix. If the hypothesis is wrong, the fix will be wrong — even if it makes the error disappear. A patch that hides symptoms without addressing the root cause is worse than the original bug.

The Anatomy of a Useful Bug Report

The quality of Claude's debugging output is directly proportional to the quality of your bug report. Use this template every time:

Effective Bug Report Template
  • Error message: Full text, including HTTP status code or exception class
  • Stack trace: Complete trace from application code (not just the framework wrapper)
  • Reproduction: The exact request/input that triggers it, including headers if relevant
  • Relevant files: Added via /add — routes, models, middleware on the error path
  • What you tried: Any fixes you already attempted, and what changed (or didn't)
  • Environment context: Local vs staging vs prod, relevant env var values (not secrets)

The Hypothesis-First Pattern

The most important prompt pattern in debugging is: "Before suggesting a fix, explain what you think is causing this and why." This one constraint forces Claude to reason about causality rather than pattern-matching to a syntactically similar fix.

Example: FastAPI 422 Unprocessable Entity

A 422 from FastAPI is one of the most common errors new FastAPI users hit. Here is a real debugging session pattern:

You
/add app/routers/orders.py app/schemas/order.py I'm getting a 422 Unprocessable Entity when calling POST /orders. Here's the exact request: curl -X POST http://localhost:8000/orders \ -H "Content-Type: application/json" \ -d '{"product_id": 42, "quantity": 3}' FastAPI response: { "detail": [ { "type": "missing", "loc": ["body", "customer_id"], "msg": "Field required", "input": {"product_id": 42, "quantity": 3}, "url": "https://errors.pydantic.dev/2.0/v/missing" } ] } The frontend is definitely sending the payload I showed above. Before suggesting a fix, explain what you think is causing this and why.
Claude
Hypothesis: The 422 is coming from FastAPI's request body validation — Pydantic is rejecting the payload because `customer_id` is defined as required in `OrderCreate` but the client isn't sending it. Looking at app/schemas/order.py: class OrderCreate(BaseModel): product_id: int quantity: int customer_id: int # ← no default value, so Pydantic treats this as required The request body `{"product_id": 42, "quantity": 3}` is missing `customer_id`, so validation fails before your route handler even executes. Two possible root causes — you need to choose: Option A: customer_id should be extracted from the authenticated user's token, not sent by the client. In this case, remove customer_id from OrderCreate and read it from request.state.user.id in the route handler. Option B: The frontend genuinely needs to send customer_id but has a bug where it's omitting it. Check the frontend's order submission code. I'd recommend Option A — clients should never self-report their own identity. If the API trusts a client-supplied customer_id, any authenticated user can create orders on behalf of any other user. Which approach should I implement?
✅
When to ask Claude vs. when to read the docs
Ask Claude when the error is in your application code — Claude can trace the execution path through your codebase. Read the docs yourself when the issue is in a third-party library's behaviour or an undocumented API edge case. Claude's training data on library internals can be stale, and hallucinated method signatures are a real risk. Use Claude to narrow down where to look; use official documentation to understand how the library actually behaves.

Strategic Use of /add for Debugging

A common mistake is either adding too few files (Claude can't see the bug) or adding entire feature directories (signal drowns in noise). The right approach is to trace the stack:

Stack-driven /add strategy 
# Read the stack trace, identify the files it touches, add exactly those:
# Example stack: routes/orders.py → services/order_service.py → models/order.py

/add app/routers/orders.py app/services/order_service.py app/models/order.py

# DO NOT add the entire app/ directory "just in case"
# DO NOT add unrelated models or utility files
# If Claude asks for more context, add one file at a time

Refactoring with Confidence

Refactoring is where Claude Code genuinely excels — and where most AI coding tools fall short. The reason is context scale: a robust refactor requires understanding every call site, every type signature, every side effect. Claude can hold 200K tokens of code in context and trace cross-file dependency chains that would take a human engineer an hour to map manually.

⚠️
Safe refactoring rule: tests before refactor, always
Before asking Claude to refactor anything, you must have tests that cover the behaviour being refactored. If the tests don't exist, write them first (see Section 5). A refactor without tests is a rewrite — Claude may preserve the structure while silently changing the behaviour, and without tests you have no way to detect it.

Incremental vs. Big-Bang Refactoring

The incremental approach — refactor one function, verify, move to the next — is the only safe strategy with Claude. Big-bang refactors produce diffs that are impossible to review, break bisectability of your git history, and compound errors. An incremental refactor keeps every commit green and reviewable.

Approach Diff Size Review Complexity Error Detection Git History
Incremental (one function at a time) Small Low Immediate Bisectable
Big-bang (entire module at once) Large High Late / missed Black box

Common Refactoring Prompts That Work

Proven refactoring prompt patterns
  • Extract function "Extract the validation logic in lines 45-72 of order_service.py into a private function _validate_order_constraints(). Keep the same behaviour, add a docstring explaining what it validates."
  • Rename across codebase "Rename user_token to access_token everywhere in the codebase. Show me every file that changes before making the edit."
  • Sync → async "Convert fetch_user_profile() from synchronous requests to async httpx. Match the pattern used in services/notification_service.py. Don't change the public interface."
  • Library migration "Migrate all HTTP calls in services/ from the requests library to httpx. Make them async. One file at a time, starting with order_service.py."
  • Type annotations "Add full type annotations to models/user.py. Use the Python 3.11 syntax. Don't change any logic."

Real Example: Migrating requests to httpx (sync → async)

A common migration task: your service layer uses the synchronous requests library, but you need async support for your FastAPI application. Here is the exact workflow:

Building Team Trust with the "Explain What Changed" Pattern

When multiple engineers share a codebase, AI-generated refactors can erode trust if teammates can't understand what changed and why. The fix is simple: always close a refactoring task by asking Claude to produce a one-paragraph plain-English summary of what changed. This becomes your commit message, your PR description, and your code review context — written in under 30 seconds.

Post-refactor explanation prompt 
"Write a two-sentence commit message explaining what you changed in this
refactor and why the old approach needed to change. Write it in the
imperative mood (e.g. 'Convert X to Y because...')."

Codebase Understanding & Onboarding

Joining a new codebase is one of the highest-friction events in an engineer's career. A senior teammate can compress weeks of learning into a few well-placed conversations. Claude Code can play this role 24/7, with perfect recall of every file you show it, and no impatience with beginner questions.

Questions That Produce High-Quality Answers

Not all codebase questions are equal. Claude produces much better answers when questions are specific and scoped rather than open-ended. Here is a progression from weak to strong:

Weak Question Strong Question
"How does this app work?" "What does the services/ module do, and how does it relate to the routers/ module?"
"Where is X?" "Where is the function that validates an incoming webhook signature? Which file is it in?"
"What calls this function?" "What calls process_payment()? Trace every caller and tell me what data each one passes."
"Why was this written this way?" "This retry loop in worker.py uses exponential backoff with jitter. What failure mode is this designed to handle, and what would break if the jitter were removed?"

Generating Architecture Diagrams from Code

One of the most powerful onboarding uses of Claude is generating architecture diagrams directly from source code. You don't need to maintain a Confluence page — you ask Claude to produce an ASCII or Mermaid diagram from the actual code, which means it's always up to date.

The Strategic /add Approach for Understanding

The single most common mistake when using Claude for codebase understanding is adding everything. Counter-intuitively, adding the entire codebase degrades the quality of Claude's answers because relevant signals are diluted. Use a layered strategy:

1
Start with the entry point
Add the router/controller file for the feature you're investigating. Ask Claude what it does and what it depends on.
2
Follow the dependencies Claude identifies
Claude will mention service classes, models, or utility functions. Add those files next, in response to Claude's questions or your own.
3
Add configuration and middleware last
Once you understand the happy path, add middleware and config files to understand the security and observability layers.

Building a Mental Model: Data Flow, Error Handling, Auth Chain

For a new engineer joining a Django application, a high-value first session with Claude might look like this:

You
/add myapp/views/api.py myapp/middleware.py myapp/models.py myapp/serializers.py I'm a new engineer on this team. Walk me through these four files and answer: 1. Where does authentication happen — middleware or the view? 2. What ORM models does the main API endpoint touch? 3. How are validation errors returned to the client — what shape is the error response? 4. Is there anything in this code that a new engineer typically trips over?
Claude
1. Authentication location: Authentication happens in both places, in sequence. middleware.py contains JWTAuthenticationMiddleware which runs first and attaches request.user. The views then use the @login_required decorator which checks that request.user.is_authenticated is True. Unauthenticated requests are rejected at the middleware layer with 401, not by the view. 2. ORM models touched by the main endpoint (GET /api/v1/items): - Item (SELECT with filter on owner_id) - ItemTag (prefetch_related — avoids N+1) - Category (select_related — single JOIN) The query is in views/api.py line 88. It uses select_related + prefetch_related, which is good practice. Note: it filters by request.user.id, so each user only sees their own items. 3. Error response shape: Validation errors come from DRF serializers and are shaped as: {"field_name": ["error message"]} for field errors {"non_field_errors": ["message"]} for object-level errors This is standard DRF format, documented at django-rest-framework.org. 4. Things that trip up new engineers: - The `owner` field on Item is NOT the same as `created_by`. owner can be transferred (see Item.transfer_ownership()). created_by is immutable. New engineers often use them interchangeably and create subtle permission bugs. - ItemSerializer has a custom to_representation() that excludes soft-deleted tags. If you add a tag and it disappears, check the tag's is_deleted flag.

TDD & Test Generation

Claude is a capable test author — but only when given the right context. Without context, Claude generates generic tests that pass on the first try against a correct implementation but miss edge cases, don't match your team's testing conventions, and don't integrate with your fixture setup. With the right context, Claude generates tests that a senior engineer would be proud to commit.

💡
The right order: failing test first, then implementation
The TDD loop with Claude is: (1) ask Claude to write a failing test for the behaviour you want, (2) verify it fails for the right reason, (3) ask Claude to make it pass with the simplest correct implementation, (4) review what it changed. This keeps you in control of the specification and prevents Claude from writing tests that vacuously pass against a hollow implementation.

What Context Claude Needs to Write Good Tests

Required context for high-quality test generation
  • The function under test: Add the source file with the function via /add
  • An example test file: Add tests/test_similar_thing.py so Claude matches your team's style — pytest fixtures, factory patterns, assertion style
  • The test framework and key fixtures: If you have shared fixtures in conftest.py, add it
  • Explicit coverage requirements: State the cases you need covered — don't leave it to Claude's judgment

The Test Generation Prompt Pattern

High-signal test generation prompt 
/add app/services/user_service.py tests/test_auth_service.py tests/conftest.py

Write pytest tests for register_user() in user_service.py.

Match the style and fixture patterns in tests/test_auth_service.py exactly:
- Use the `db_session` fixture from conftest.py
- Use UserFactory for creating test users
- Use `pytest.raises` for expected exceptions
- One assertion per test function, named test___

Cover these cases:
  1. Happy path: valid email + password → returns User object with hashed password
  2. Missing required field: no email → raises ValidationError
  3. Duplicate email: same email registered twice → raises DuplicateEmailError
  4. Weak password: fewer than 8 chars → raises PasswordTooWeakError
  5. Email normalisation: "User@EXAMPLE.COM" should be stored as "user@example.com"

This prompt produces useful tests because it specifies style (fixture patterns, assertion style, naming), coverage (five explicit cases), and constraints (don't invent new fixtures). Compare that to the low-signal alternative:

Low-Signal Prompt Result
"Add tests for register_user()" Generic tests, wrong fixture names, misses edge cases, may not run
"Write comprehensive tests for user_service.py" Broad coverage of easy paths, sparse on error conditions, invented mock patterns
The prompt above with explicit coverage list Tests that run immediately, match team style, cover specified cases

Common Failure Mode: "Add Tests" Without Context

The most common way teams get low value from Claude's test generation is asking it to "add tests" on a file it has never seen, with no example tests and no specification of what to cover. Claude will generate something that looks like a test suite but: uses the wrong test runner, invents fixtures that don't exist, passes trivially because the mocks are too permissive, and misses every business-critical edge case.

The fix is not a better model — it is a better prompt. Always add the source file, always add an example test file, always enumerate the cases you want covered.

Code Review & Quality Gates

Code review is one of the highest-leverage places to apply Claude Code — not because it replaces human reviewers, but because it functions as a pre-review filter that catches a large class of mechanical issues before the diff reaches a human. Your reviewers spend their time on architecture, intent, and business logic; Claude handles the checklist.

Claude as Pre-Review Reviewer

The workflow is straightforward: before opening a PR, pipe your diff to Claude with a specific review checklist. This is not a vague "review this code" — it is a structured audit against a defined set of risk categories.

Pre-PR review prompt 
# Generate the diff against main branch
git diff main > /tmp/feature-diff.patch

# In Claude Code:
/add /tmp/feature-diff.patch CLAUDE.md

Review this diff against the following checklist. For each item, either confirm
it's clean or quote the exact line and explain the problem:

SECURITY:
  [ ] SQL injection: any raw string interpolation into SQL queries?
  [ ] Missing authentication/authorisation checks on new endpoints?
  [ ] Secrets or credentials hardcoded or logged?
  [ ] Untrusted user input passed directly to shell commands or file paths?

CORRECTNESS:
  [ ] Null/None checks missing where the value could be absent?
  [ ] Off-by-one errors in loops or pagination?
  [ ] Race conditions in async code (e.g. check-then-act on shared state)?
  [ ] Error paths that silently swallow exceptions?

PERFORMANCE:
  [ ] N+1 queries introduced (ORM calls inside loops)?
  [ ] Unbounded queries (missing LIMIT on user-controlled filters)?
  [ ] Synchronous blocking calls in async context?

STYLE (per CLAUDE.md):
  [ ] Function names match our snake_case convention?
  [ ] All new public functions have docstrings?
  [ ] No commented-out code left in?

Adding CLAUDE.md to the Review Context

CLAUDE.md is your team's coding standards document — the file Claude Code reads to understand your conventions. When you include it in a code review prompt, Claude can flag style inconsistencies against your actual standards rather than generic style advice. If your team requires type annotations on all public functions, Claude will catch the missing ones. If you have a specific error handling pattern, Claude will flag deviations.

💡
CLAUDE.md is your living style guide
Every team should maintain a CLAUDE.md at the repo root that documents: code style decisions, architectural patterns, forbidden patterns (e.g. "never use raw SQL"), test conventions, and CI requirements. This file serves double duty: it guides Claude Code's behaviour in interactive sessions, and it gives the review prompt the context it needs to catch standard deviations.

What Claude Reliably Catches

CLAUDE CODE REVIEW — DETECTION RELIABILITY High reliability (catches consistently): ■ Missing null/undefined checks on method return values ■ String interpolation in SQL queries (classic SQLi pattern) ■ Secrets/API keys in code or log statements ■ ORM calls inside for/while loops (N+1 query pattern) ■ Missing error handling on external API calls (unhandled exceptions) ■ Style violations when CLAUDE.md is in context Medium reliability (catches most, misses some): ■ Race conditions in async code (detects obvious patterns) ■ Missing input validation on user-controlled parameters ■ Authorisation gaps on new endpoints ■ Off-by-one errors in pagination/slicing logic Low reliability (requires human review): ■ Performance regressions under production load (needs profiling data) ■ Correctness under concurrent high-traffic scenarios ■ Business logic errors (requires domain knowledge) ■ Subtle auth bypass via chained permission checks

CI Integration: Non-Blocking Advisory Step

The most scalable way to deploy Claude code review is as a non-blocking advisory step in CI. On every PR, a CI job runs git diff main, pipes the diff to Claude via the API, and posts the review as a PR comment. Engineers see Claude's findings immediately — before human reviewers engage — and can address mechanical issues proactively.

CI advisory review script (GitHub Actions) 
# .github/scripts/claude_review.py
import anthropic, subprocess, os, sys

client = anthropic.Anthropic()

# Capture diff from main branch
diff = subprocess.check_output(
    ["git", "diff", "origin/main...HEAD"],
    text=True
)

if not diff.strip():
    print("No changes detected.")
    sys.exit(0)

prompt = f"""Review this diff for a Python/FastAPI codebase:

{diff[:30000]}  # truncate to stay within token budget

Flag ONLY real issues. For each issue, output:
  FILE: path/to/file.py:LINE_NUMBER
  SEVERITY: critical | warning | info
  ISSUE: one-sentence description
  FIX: one-sentence recommendation

If the diff is clean, output: NO_ISSUES_FOUND"""

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=2048,
    messages=[{"role": "user", "content": prompt}]
)

print(response.content[0].text)  # captured by CI and posted as PR comment
⚠️
Keep AI review non-blocking — never gate merges on it
Claude's code review should inform human reviewers, never replace them or block merges automatically. Claude does not understand your production load patterns, your deployment constraints, or the organisational context behind a design decision. A false positive that blocks a critical hotfix is worse than a false negative that requires a follow-up ticket. Configure the CI step with continue-on-error: true and treat Claude's output as advisory commentary, not a quality gate.

What Claude Misses — and Why Human Review Remains Essential

Claude's code review operates on static analysis of the diff. It has no access to production metrics, no memory of previous incidents, and no knowledge of the organisational context behind a design decision. Specifically, Claude cannot catch:

  • Performance regressions at production scale: Claude can identify N+1 patterns but cannot predict whether a query will be acceptable at 10K RPS. That requires load test data and production query plans.
  • Correctness under concurrency: Race conditions that only manifest under specific timing and load patterns are extremely difficult to detect from static analysis alone.
  • Business logic correctness: Whether a discount calculation or an access control rule matches the product specification requires domain knowledge Claude doesn't have.
  • Intentional tech debt: Claude will flag code it considers suboptimal even when the team has consciously chosen the simpler approach for now. Context about the roadmap matters.
