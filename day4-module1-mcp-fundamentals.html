01

What is MCP and Why It Matters

The Model Context Protocol (MCP) is an open standard introduced by Anthropic that defines how AI models communicate with external tools and data sources. Before MCP, every team building an AI-powered product had to solve the same problem independently: how do you give Claude access to your GitHub repositories, your database, your internal APIs, your Slack workspace? Each integration required bespoke glue code — custom prompting, custom parsing of results, custom error handling, and custom security logic.

MCP solves the N×M integration problem. Without MCP, if you have N AI applications and M tools, you need N×M custom integrations. With MCP, each AI application implements the MCP client protocol once, and each tool implements the MCP server protocol once. The result is N+M implementations that interoperate freely. This is the same insight that made the Language Server Protocol transformative for editors — one protocol, unlimited extensibility.

ℹ
The Analogy: USB for AI Tools
Before USB, every peripheral needed its own port and driver. After USB, you plug anything into anything. MCP does the same for AI capabilities: Claude is the USB host, MCP servers are the peripherals, and the protocol defines the standard connector shape. You write the server once; it works with any MCP-compatible client.

Client-Server Architecture

MCP uses a strict client-server model. The MCP client is the AI application — in our case, Claude Code or any application built on the Anthropic API. The MCP server is the process that exposes your tools and data. Claude never calls your GitHub API directly; instead, it sends a structured tool call to your GitHub MCP server, which handles authentication, makes the API call, and returns structured results.

Claude only speaks MCP — never raw GitHub/DB/Slack APIs.

The Integration Problem: Before and After MCP

✗
Without MCP
Bespoke prompt engineering per tool. Custom result parsing for each API. Duplicated auth logic. Security review needed for every new integration. No standardised error format. Each team reinvents the wheel.
✓
With MCP
Standardised JSON-RPC 2.0 protocol. Claude discovers tools automatically via capability negotiation. One server implementation works across all MCP clients. Reusable community-built servers. Auditable tool calls with structured inputs and outputs.
⚠
MCP Is Not a Magic Security Boundary
MCP standardises the communication protocol, not the permission model. A poorly configured MCP server with write access to production databases is just as dangerous as any other over-permissioned service. Day 4 Module 4 covers the security governance layer you must build on top of MCP.
02

The Three Core Primitives (+ Sampling)

MCP defines four fundamental primitives — the building blocks from which all MCP capabilities are constructed. Understanding which primitive to use and who controls it is the foundation of good MCP server design.

Primitive Controller When It Runs Example Use Cases
Tools Model-controlled When Claude decides to call a function search_database, create_issue, send_slack_message, run_sql_query
Resources Application-controlled When the application explicitly reads a resource File contents, database rows, API responses, documentation pages
Prompts User-controlled When the user selects a prompt template Code review template, sprint planning prompt, security audit checklist
Sampling Server-controlled When an MCP server requests a Claude completion Agentic loops within servers, dynamic tool orchestration, server-initiated reasoning

Tools: The Most Used Primitive

Tools are functions that Claude can invoke during a conversation. When Claude encounters a task that requires external data or action — searching a database, creating a GitHub issue, posting a message — it selects the appropriate tool, constructs arguments that match the tool's JSON Schema, and sends a tools/call request to the MCP server. The server executes the action and returns a structured result.

The key insight: Claude decides when to call a tool based solely on the tool's name and description. The model never sees your implementation. This means the quality of your tool descriptions is the most important factor in whether Claude uses your tools correctly.

python
# Minimal MCP server with one Tool (using the mcp[cli] Python SDK)
from mcp.server import Server
from mcp.server.models import InitializationOptions
from mcp.types import Tool, TextContent
import mcp.server.stdio

# Create the MCP server instance
server = Server("knowledge-base")

# Register a tool using the decorator
@server.list_tools()
async def handle_list_tools() -> list[Tool]:
    return [
        Tool(
            name="search_kb",
            description="""Search the internal knowledge base for articles matching a query.
            Use this tool when the user asks about internal processes, policies, architecture
            decisions, runbooks, or any company-specific knowledge. Returns a list of matching
            articles with titles, summaries, and URLs. The query supports natural language.""",
            inputSchema={
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": "Natural language search query"
                    },
                    "max_results": {
                        "type": "integer",
                        "description": "Maximum number of results to return (default: 5)",
                        "default": 5
                    }
                },
                "required": ["query"]
            }
        )
    ]

# Handle tool calls
@server.call_tool()
async def handle_call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "search_kb":
        results = await search_knowledge_base(
            query=arguments["query"],
            max_results=arguments.get("max_results", 5)
        )
        return [TextContent(type="text", text=format_results(results))]
    raise ValueError(f"Unknown tool: {name}")

# Run the server over stdio
async def main():
    async with mcp.server.stdio.stdio_server() as (read_stream, write_stream):
        await server.run(
            read_stream, write_stream,
            InitializationOptions(
                server_name="knowledge-base",
                server_version="1.0.0"
            )
        )

Resources: Structured Data Access

Resources expose data that Claude can read — similar to how a file system exposes files. Each resource has a URI (like kb://articles/runbooks/deployment), a MIME type, and content. Unlike tools, resources are read-only and application-controlled, meaning the application decides which resources to expose rather than Claude deciding to call them.

Sampling: Enabling Agentic MCP Servers

Sampling is the most powerful and least understood primitive. It allows MCP servers to request Claude completions — enabling servers to implement their own agentic loops, multi-step reasoning, or dynamic orchestration. This is how complex MCP servers can go beyond simple function calls and implement genuinely intelligent behaviours.

✓
Design Principle: Start with Tools, Add Resources Later
For most enterprise MCP servers, Tools are the right starting point. They give Claude callable capabilities immediately. Add Resources when you need Claude to browse large collections of documents. Add Sampling only when your server needs to implement its own reasoning loops — it significantly increases implementation complexity.
03

Transport Mechanisms Deep Dive

MCP is transport-agnostic at the protocol level. The same JSON-RPC 2.0 messages can flow over different transport mechanisms depending on your deployment model. Understanding when to use each transport is critical for enterprise architecture decisions.

stdio (Local Process)
Mechanism: JSON-RPC over stdin/stdout. The MCP server runs as a child process of Claude Code.

Best for: Developer tools, personal productivity, local tooling, Claude Code integrations.

Pros: Zero network overhead. Trivial setup. Natural process isolation. Inherits user's shell env.

Cons: Can't be shared across team. Server dies when Claude Code session ends.
HTTP + SSE (Remote Server)
Mechanism: Client sends requests via HTTP POST; server sends events via Server-Sent Events.

Best for: Enterprise shared MCP servers, team-wide deployments, production services.

Pros: Shareable across org. Deployable to cloud. Standard auth headers (Bearer tokens).

Cons: Network latency. Requires auth management. SSE has browser connection limits.
WebSocket (Bidirectional)
Mechanism: Full-duplex WebSocket connection. Both sides can send at any time.

Best for: High-frequency tool calls, real-time data streams, interactive agent sessions.

Pros: Low latency. Bidirectional push. Ideal for sampling-heavy servers.

Cons: More complex to manage. Requires WebSocket-aware load balancers and proxies.
┌─────────────────────────────────────────────────────────────────┐ │ TRANSPORT MECHANISM COMPARISON │ └─────────────────────────────────────────────────────────────────┘ stdio (Local Process Transport) ────────────────────────────── Claude Code ──stdin──▶ MCP Server Process Claude Code ◀─stdout── MCP Server Process Lifecycle: server starts with Claude Code, dies with it Auth: inherits from parent process environment variables Latency: ~0ms (IPC) HTTP + SSE (Remote Transport) ──────────────────────────── Claude Code ──POST /message──▶ MCP HTTP Server ──▶ Your Backend Claude Code ◀──SSE /events─── MCP HTTP Server ◀── Your Backend Lifecycle: server runs independently, persists across sessions Auth: Authorization: Bearer <token> header on every POST Latency: network RTT (10–100ms typical) Decision Matrix: Local dev tools → stdio Shared org-wide server → HTTP+SSE Real-time agent loops → WebSocket

Configuring stdio Transport in Claude Code

For stdio MCP servers, Claude Code manages the server process lifecycle automatically. You declare the server in .claude/settings.json and Claude Code spawns, monitors, and communicates with the process transparently.

json
// .claude/settings.json — stdio MCP server configuration
{
  "mcpServers": {
    "knowledge-base": {
      "command": "python3",
      "args": ["/path/to/kb_server.py"],
      "env": {
        "KB_API_KEY": "${KB_API_KEY}",
        "KB_BASE_URL": "https://kb.internal.acme.com"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://readonly_user:${DB_PASS}@db.internal:5432/production"
      ]
    }
  }
}
⚠
Never Hardcode Secrets in MCP Config
The settings.json file may be committed to version control. Always reference environment variables using ${VAR_NAME} syntax. Claude Code expands these from the shell environment at startup time. Store actual secrets in your system keychain or a secrets manager, not in the config file.

Tool Discovery and the Capability Lifecycle

When Claude Code starts a session with MCP servers configured, it executes a capability negotiation handshake with each server. This handshake follows a strict sequence that determines what Claude can see and use for the rest of the session.

┌─────────────────────────────────────────────────────────────────┐ │ MCP CAPABILITY NEGOTIATION LIFECYCLE │ └─────────────────────────────────────────────────────────────────┘ Claude Code MCP Server │ │ │──── initialize { clientInfo } ────────▶│ │ │ │◀─── initializeResult { capabilities }──│ │ { tools: {}, resources: {}, │ │ prompts: {}, sampling: {} } │ │ │ │──── initialized (notification) ───────▶│ │ │ │──── tools/list ───────────────────────▶│ │◀─── [Tool, Tool, Tool, ...] ───────────│ │ (name, description, inputSchema) │ │ │ Claude now knows every available tool and its schema │ --- User asks a question --- │ │──── tools/call { name, arguments } ───▶│ │◀─── CallToolResult { content: [...] } ─│ │ Claude incorporates result into response

Tool Schema Anatomy

Every tool Claude discovers has three required fields. Getting these right is the most important part of MCP server design because Claude makes all tool selection decisions based on them.

python
# Good tool definition — Claude will use this correctly
Tool(
    name="get_customer_order_history",  # verb_noun, specific
    description="""Retrieve the complete order history for a customer account.
    Use this tool when the user asks about past orders, purchase history,
    what a customer has bought, or needs to look up a specific order.
    Returns orders sorted by date descending, with order ID, items,
    total amount, status (pending/shipped/delivered/refunded), and
    tracking number if available. Requires either customer_id or email.""",
    inputSchema={
        "type": "object",
        "properties": {
            "customer_id": {
                "type": "string",
                "description": "Internal customer UUID (preferred over email)"
            },
            "email": {
                "type": "string",
                "description": "Customer email address (used if customer_id is unknown)"
            },
            "limit": {
                "type": "integer",
                "description": "Max orders to return (default: 20, max: 100)",
                "default": 20,
                "maximum": 100
            }
        },
        "required": []  # Neither required alone — validation happens in handler
    }
)

# Bad tool definition — Claude will struggle to use this correctly
Tool(
    name="get_data",            # too generic — what data?
    description="Gets data.",   # useless — Claude can't infer when to call it
    inputSchema={
        "type": "object",
        "properties": {
            "id": {"type": "string"}  # id of what? no description
        }
    }
)
✗
The #1 MCP Server Bug: Vague Descriptions
If Claude calls the wrong tool, or fails to call the right tool when it should, the problem is almost always the tool description. Claude reads descriptions and reasons about which tool fits the user's request. Write descriptions as if you're explaining the tool to a new engineer on their first day. Be explicit about: when to use it, what it returns, and which parameters are optional.

Resource and Prompt Discovery

Beyond tools, Claude Code can also list and access resources and prompts from MCP servers. Resources are discovered via resources/list and accessed via resources/read with a URI. Prompts are discovered via prompts/list and expanded via prompts/get. Both follow the same discover-then-use pattern as tools.

05

Using MCP with Claude Code

Claude Code is designed as a first-class MCP client. It manages server lifecycles, surfaces available tools to the model in every conversation, and handles the tool call/result cycle transparently. The /mcp command lets you inspect, manage, and debug your MCP server connections at any time.

Official Anthropic MCP Servers

📁
filesystem
Read/write local filesystem with configurable allowed paths. The foundation for all file-manipulation workflows.
🐙
github
Full GitHub API access: repos, issues, PRs, commits, branches, files. Requires GitHub PAT.
🗃️
postgres
Read-only PostgreSQL access: list tables, describe schema, execute SELECT queries. Connection string in config.
💬
slack
Slack workspace access: list channels, post messages, read threads. Requires Slack OAuth Bot Token.

Complete Multi-Server Configuration

json
// .claude/settings.json — enterprise multi-server MCP configuration
{
  "mcpServers": {
    // Filesystem — scoped to project directory only
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/dev/projects/acme-platform"  // restrict to project root
      ]
    },

    // GitHub — org-scoped PAT with read:org + repo permissions
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_PAT}"
      }
    },

    // PostgreSQL — read-only analytics replica, not primary
    "database": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://claude_readonly:${DB_PASS}@analytics-replica.internal:5432/app_db"
      ]
    },

    // Custom internal knowledge base — Python stdio server
    "knowledge-base": {
      "command": "python3",
      "args": ["/opt/mcp-servers/kb_server.py"],
      "env": {
        "KB_API_TOKEN": "${KB_API_TOKEN}",
        "KB_ENDPOINT": "https://kb.acme.internal/api/v1"
      }
    }
  },

  // Permission controls — restrict what Claude can do with these tools
  "permissions": {
    "allow": [
      "mcp__filesystem__read_file",
      "mcp__filesystem__list_directory",
      "mcp__github__search_repositories",
      "mcp__github__get_pull_request",
      "mcp__database__query",
      "mcp__knowledge-base__search_kb",
      "mcp__knowledge-base__get_article"
    ],
    "deny": [
      "mcp__filesystem__write_file",  // require explicit approval
      "mcp__github__push_files"        // require explicit approval
    ]
  }
}
