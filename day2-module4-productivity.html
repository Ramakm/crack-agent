Shell Pipelines & Claude CLI

Claude Code ships with a --print flag that turns it into a non-interactive Unix citizen. No TUI, no streaming spinner — just stdin in, stdout out. That single flag is what makes Claude composable with every other tool in your shell. If you understand Unix pipes, you already understand the pattern.

💡
Non-interactive Mode Is Not a Downgrade
claude --print skips the interactive TUI entirely. This means no context window accumulation across invocations, no session state, and deterministic single-shot execution. That is exactly what you want in pipelines, hooks, and watch scripts. Treat each invocation as a pure function: input in, analysis out.

The Bash Wrapper Pattern

The most immediately useful thing you can do is alias claude --print as a smart shell command. Drop this in your .zshrc or .bashrc:

~/.zshrc — ask-claude wrapper 
# Smart shell assistant — use like: ask-claude "what does this mean?"
function ask-claude() {
  if [ -p /dev/stdin ]; then
    # Data piped in: prepend it to the prompt
    piped=$(cat)
    claude --print "$piped

$*"
  else
    # No pipe: prompt only
    claude --print "$*"
  fi
}

# Convenience aliases
alias ac='ask-claude'
alias explain='ask-claude "Explain this to a senior engineer, be concise:"'

Watch Mode — Auto-Explain File Changes

Combine fswatch (macOS) or inotifywait (Linux) with Claude to build a lightweight "explain this change" feedback loop while you edit:

Watch mode — explain every save to a Python file 
# macOS: brew install fswatch
# Watches a file and explains what changed after each save
function watch-explain() {
  file="$1"
  prev=$(cat "$file")
  echo "Watching $file — will explain each change..."

  fswatch -o "$file" | while read; do
    curr=$(cat "$file")
    diff=$(diff <(echo "$prev") <(echo "$curr"))
    if [ -n "$diff" ]; then
      echo "---"
      echo "$diff" \
        | claude --print "In one sentence: what did this code change do?"
      prev="$curr"
    fi
  done
}

# Usage: watch-explain src/payments/processor.py
⚠️
Never Pipe Secrets into Claude
Before building any pipeline that processes files automatically, audit what those files might contain. Log files, config files, and API responses regularly contain API keys, tokens, and credentials. Add a scrubbing step before any pipe that processes files you don't fully control — or use an allowlist of specific log fields rather than raw log lines.
02

Production Log Analysis

Production log files are a war zone: thousands of lines per minute, mixed severity, noisy repetition from health checks, and the one critical error buried three screens down. APM tools (Datadog, CloudWatch, New Relic) solve the metrics problem. They don't solve the meaning problem — understanding why a pattern emerged, what sequences of events preceded an incident, or whether an unusual request pattern indicates a security anomaly.

Claude augments your APM tooling. It does not replace it. The workflow is: APM tools surface that something is wrong, Claude helps you understand what and why.

The Triage Workflow

01
Pull logs for the incident window
Export from Datadog (datadog-ci), CloudWatch Logs Insights, or pull directly from the server. Scope to ±15 minutes around the alert trigger.
02
Filter by time window and severity
Use grep, awk, or jq to strip health-check noise and focus on ERROR/WARN/CRITICAL. Target 300–800 lines — enough signal, not so much that Claude loses the thread.
03
Pipe to Claude with a structured triage prompt
Use the prompt pattern below. Ask for specific outputs: severity grouping, timeline, security flags, and recommended immediate actions.
04
Route the analysis
Paste the Claude summary into your incident Slack channel. Use it as the starting point for postmortem. Never use Claude's analysis as a substitute for reading the logs yourself on P0 incidents.

The Triage Prompt Pattern

This prompt template is production-tested. It asks Claude for structured output, which you can parse or paste directly into your incident doc:

Log triage — full prompt pattern 
INCIDENT_WINDOW="2025-05-21 14:30 to 14:55 UTC"
SERVICE="payments-api"

# Pull filtered logs from CloudWatch JSON export
cat incident-logs.json \
  | jq -r '.events[] | "\(.timestamp) [\(.level)] \(.message)"' \
  | grep -v "health_check\|/ping\|/metrics" \
  | claude --print "You are a senior SRE triaging a production incident.
Service: $SERVICE
Window: $INCIDENT_WINDOW

Here are the filtered production API logs.

Identify and report:
1. ERROR SPIKES: Groups of errors that appear together. For each group:
   - Error type and message pattern
   - First occurrence timestamp
   - Approximate frequency (errors/minute)
   - Likely root cause

2. UNUSUAL PATTERNS: Request sequences that look abnormal —
   high latency, unexpected 4xx bursts, retry storms, cascade failures.

3. SECURITY FLAGS: Anything that looks like a scan, brute-force,
   unusual IP concentration, or unexpected auth failures.

4. TIMELINE: A bullet-point chronological sequence of significant events.

5. IMMEDIATE ACTIONS: Top 3 things the on-call engineer should
   check or do right now.

Format as markdown sections. Be specific — quote log lines when relevant."

💡
Claude Cannot Query Live Systems
Claude operates on the text you give it. It cannot run queries against your log platform, connect to your observability stack, or read logs in real-time. Always pre-pull and pre-filter log excerpts before passing them to Claude. For live dashboards and alerts, keep your APM tooling as the source of truth — Claude is the analyst, not the database.

Automated PR Summaries & Changelogs

Writing a PR description after finishing a feature is the worst possible moment to do it. You're context-switched out, you've already moved on mentally, and the description ends up as "fixes stuff" or a wall of bullet points copied from commit messages. Claude does this work at the moment the diff exists and it has maximum context.

The One-Liner That Changes Your Workflow

Generate a PR description from the current branch diff 
git diff main...HEAD \
  | claude --print "Write a pull request description for this diff.

Structure it exactly as:
## Summary
(2-3 sentences: what this PR does and why)

## Key Changes
(bullet points — implementation decisions, not file names)

## Testing Notes
(how to verify this works — specific steps)

## Breaking Changes
(any API surface changes, DB migrations, env var additions; 'None' if clean)

## Reviewers: please pay attention to
(the 1-2 places in the diff that are most risk-relevant)

Tone: direct, no marketing language. Senior engineer writing to senior reviewers."

The Pre-Push Git Hook

Wrap that one-liner in a pre-push hook so a draft PR description is generated and copied to your clipboard before every push. Make it opt-in via an env var so engineers who don't want it aren't affected:

.git/hooks/pre-push — opt-in PR description generator 
#!/usr/bin/env bash
# .git/hooks/pre-push
# Opt-in: set CLAUDE_PR_HELPER=1 in your shell profile to enable.
# chmod +x .git/hooks/pre-push

if [ "$CLAUDE_PR_HELPER" != "1" ]; then
  exit 0  # silently skip if not opted in
fi

# Detect the base branch (main or master)
BASE=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null \
        | sed 's@^refs/remotes/origin/@@')
BASE=${BASE:-main}

# Only run if there are commits ahead of base
AHEAD=$(git rev-list --count "origin/$BASE..HEAD" 2>/dev/null)
if [ "$AHEAD" -eq 0 ] 2>/dev/null; then
  exit 0
fi

echo "[claude-pr-helper] Generating PR description..."

PR_DESCRIPTION=$(git diff "$BASE...HEAD" \
  | claude --print "Write a pull request description for this diff.

## Summary
(2-3 sentences)

## Key Changes
(bullet points — decisions, not file names)

## Testing Notes
(specific verification steps)

## Breaking Changes
(API changes, migrations, env vars; or 'None')

## Reviewers: pay attention to
(1-2 risk-relevant sections)

Direct tone. Senior engineer to senior reviewer.")

if [ -n "$PR_DESCRIPTION" ]; then
  # Save to a temp file and open in editor (optional)
  TMPFILE=$(mktemp /tmp/pr-description-XXXXXX.md)
  echo "$PR_DESCRIPTION" > "$TMPFILE"

  # Copy to clipboard (macOS: pbcopy | Linux: xclip -sel clip)
  echo "$PR_DESCRIPTION" | pbcopy 2>/dev/null || \
  echo "$PR_DESCRIPTION" | xclip -sel clip 2>/dev/null || true

  echo "[claude-pr-helper] PR description copied to clipboard."
  echo "[claude-pr-helper] Saved to: $TMPFILE"
fi

exit 0  # always exit 0 — never block the push
💡
Team Adoption: Always Make It Opt-In First
Never force-install hooks team-wide without a trial period. Add CLAUDE_PR_HELPER=1 to your own shell profile, use it for two weeks, then demonstrate the output quality to the team. Engineers adopt tools they see producing real value — not tools that appear in their pre-push hook from a top-down mandate.

Linting & Code Quality Automation

Static analysis tools — ruff, mypy, eslint, semgrep — are fast, consistent, and excellent at detecting a well-defined class of issues. They are blind to semantics. A function that calls an external payment API from inside a utility module is not a ruff violation. Missing error handling on a database call that could return null is not a mypy error if the type annotation is wrong. Claude catches the issues that require understanding intent, not just syntax.

┌──────────────────────────────────────────────────────────────────┐ │ THREE-LAYER CODE QUALITY STACK │ ├──────────────────────────────────────────────────────────────────┤ │ │ │ Layer 1: Static Analysis ruff · mypy · eslint · semgrep │ │ Fast, deterministic. Catches: style, type errors, │ │ known antipatterns, import issues │ │ │ │ Layer 2: Claude Semantic Review git diff --staged | claude │ │ Catches: missing error handling, security issues, │ │ context-specific rule violations, architectural drift │ │ │ │ Layer 3: Human Code Review PR review by a second engineer │ │ Catches: design disagreements, business logic errors, │ │ team knowledge gaps, requirements misunderstandings │ │ │ │ Each layer catches different classes of issues. │ │ Claude does not replace Layer 1 or Layer 3 — it fills the gap. │ └──────────────────────────────────────────────────────────────────┘

The Staged-Diff Review Pattern

Run Claude against your staged diff before committing. This is the highest-signal moment — you've already decided these changes are ready. Claude acts as a fast second pair of eyes before the commit is sealed:

Review staged diff — security + error handling + standards 
# Run manually before committing, or wire into a pre-commit hook
git diff --staged \
  | claude --print "Review this staged diff. Check for:

1. SECURITY ISSUES: SQL injection, unsanitised inputs, hardcoded secrets,
   missing auth checks, insecure deserialization, path traversal.

2. MISSING ERROR HANDLING: unhandled exceptions, no null checks on external
   data, missing try/except around I/O or network calls, silent failures.

3. STYLE VIOLATIONS vs our CLAUDE.md standards:
   - All service layer functions must be async
   - External API calls must go through the api_client module only
   - Database queries must go through repository classes — never raw SQL in endpoints
   - New functions in the payments/ module require a docstring

4. LOGIC ISSUES: off-by-one errors, incorrect boolean logic, race conditions.

Output format:
- If all clear: one line 'LGTM — no issues found'
- If issues found: markdown list, one item per issue, format:
  [SEVERITY] [CATEGORY] Description. File:line if visible.

Severities: BLOCKER | WARNING | SUGGESTION"

The Complete Pre-Commit Hook

This hook runs ruff and mypy first (fast, deterministic), then calls Claude only if the static analysis passes. This keeps the hook fast for clean code and reserves Claude's API cost for changes that cleared the baseline:

.git/hooks/pre-commit — three-layer quality gate 
#!/usr/bin/env bash
# .git/hooks/pre-commit
# chmod +x .git/hooks/pre-commit
# Requires: ruff, mypy, claude CLI installed in the project venv

RED='\033[0;31m'; YELLOW='\033[1;33m'; GREEN='\033[0;32m'; NC='\033[0m'
FAIL=0

# ── Layer 1a: ruff ──────────────────────────────────────────────
echo "[pre-commit] Running ruff..."
if ! ruff check --quiet .; then
  echo -e "${RED}[BLOCKER] ruff found issues. Fix before committing.${NC}"
  FAIL=1
fi

# ── Layer 1b: mypy ──────────────────────────────────────────────
echo "[pre-commit] Running mypy..."
STAGED_PY=$(git diff --cached --name-only --diff-filter=ACM | grep '\.py$')
if [ -n "$STAGED_PY" ]; then
  if ! mypy --quiet $STAGED_PY; then
    echo -e "${RED}[BLOCKER] mypy type errors. Fix before committing.${NC}"
    FAIL=1
  fi
fi

# If static analysis failed, stop here — don't burn Claude API credits
if [ "$FAIL" -eq 1 ]; then
  exit 1
fi

# ── Layer 2: Claude semantic review ─────────────────────────────
# Skip if CLAUDE_LINT=0 (for quick WIP commits)
if [ "${CLAUDE_LINT:-1}" == "0" ]; then
  echo "[pre-commit] Claude semantic review skipped (CLAUDE_LINT=0)"
  exit 0
fi

DIFF=$(git diff --staged)
if [ -z "$DIFF" ]; then
  exit 0
fi

echo "[pre-commit] Running Claude semantic review..."
REVIEW=$(echo "$DIFF" \
  | claude --print "Review this diff for security issues, missing error handling,
and violations of our coding standards in CLAUDE.md.
If all clear: respond with exactly 'LGTM'
If issues: list them as [SEVERITY] [CATEGORY] description.
Severities: BLOCKER | WARNING | SUGGESTION")

if echo "$REVIEW" | grep -q "^LGTM"; then
  echo -e "${GREEN}[pre-commit] Claude: LGTM${NC}"
  exit 0
fi

# Show the review
echo -e "${YELLOW}[pre-commit] Claude review findings:${NC}"
echo "$REVIEW"

# Block on BLOCKER severity, warn on WARNING/SUGGESTION
if echo "$REVIEW" | grep -q "\[BLOCKER\]"; then
  echo -e "${RED}[pre-commit] BLOCKER issues found. Fix or run: CLAUDE_LINT=0 git commit${NC}"
  exit 1
fi

echo -e "${YELLOW}[pre-commit] Warnings found — proceeding. Review before pushing.${NC}"
exit 0

Custom Rule Enforcement

The most powerful use of Claude as a semantic linter is enforcing rules that are too context-specific for static analysis. Add these to your CLAUDE.md and reference them in your pre-commit prompt:

  • Architectural boundaries: "Functions in src/payments/ must not call external APIs directly — all external calls must go through api_client.py"
  • Security patterns: "Every new route that modifies data must verify current_user.has_permission() before the database operation"
  • Observability requirements: "New service layer functions that perform I/O must emit a structured log line with correlation_id"
  • Data handling rules: "Functions that receive user input must not pass that input directly to a subprocess call, file path, or template without sanitization"

Merge Conflict Resolution

Syntactic merge conflicts are the easy kind — git shows you the markers, you pick a side or combine them, and you move on. Semantic conflicts are the dangerous kind: both sides compile, both sides type-check, but the two changes are logically incompatible. One branch added retry logic with a 3-second backoff. The other added a 2-second request timeout. Neither git nor your linter can tell you these two changes will fight at runtime.

Claude's approach to merge conflicts is to understand the intent of each side before proposing a resolution. That is the same mental model a senior engineer uses — and it is exactly what makes Claude useful here.

The Conflict Resolution Prompt Pattern

Pipe a conflict file to Claude for intent-first resolution 
# After a failed merge, find all conflict files
git diff --name-only --diff-filter=U

# For each file with conflicts:
cat src/payments/processor.py \
  | claude --print "This file has git merge conflict markers (<<<<<<<, =======, >>>>>>>).

For each conflict block:
1. INTENT (OURS): What is the HEAD version trying to achieve?
2. INTENT (THEIRS): What is the incoming branch trying to achieve?
3. Are these intents compatible, or do they conflict logically?
4. PROPOSED RESOLUTION: Show the code that correctly preserves both intents.
   If they are logically incompatible, explain why and flag for human review.

Output the fully resolved file with all conflict markers removed.
Add a brief comment above each resolved block explaining the decision." \
  > src/payments/processor.resolved.py

# Review the resolved file before accepting it
diff src/payments/processor.py src/payments/processor.resolved.py

A Real Python Conflict — Both Sides Were Right

This is a real class of conflict that happens when two engineers independently improve the same function:

HEAD (our branch)
def charge_customer(
    user_id: str,
    amount_cents: int,
    currency: str = "usd",
) -> ChargeResult:
    # Added: input validation
    if amount_cents <= 0:
        raise ValueError(
            "amount_cents must be positive"
        )
    if amount_cents > 999_999_99:
        raise ValueError(
            "amount exceeds Stripe limit"
        )
    return stripe.charge(
        user_id, amount_cents, currency
    )
THEIRS (incoming branch)
def charge_customer(
    user_id: str,
    amount_cents: int,
    currency: str = "usd",
    idempotency_key: str = "",
) -> ChargeResult:
    # Added: idempotency support
    if not idempotency_key:
        idempotency_key = (
            f"charge-{user_id}-"
            f"{amount_cents}-"
            f"{uuid4().hex[:8]}"
        )
    return stripe.charge(
        user_id, amount_cents,
        currency, idempotency_key
    )

Claude correctly identifies that both changes are independent improvements with no logical conflict. Its resolution merges the input validation guards from HEAD with the idempotency parameter from THEIRS, producing a function that is both safer and idempotent. It would also note that the idempotency key generation belongs in a separate helper for testability.

⚠️
Mandatory Escalation List — Never Auto-Merge
Claude should flag the following conflict types for human review and never auto-commit a resolution: security code (auth, permissions, session management), database migrations (schema changes, data transforms), cryptographic operations (key handling, signing, hashing), payment processing logic (amounts, rounding, idempotency), and infrastructure-as-code (IAM policies, network rules). In these areas, a wrong resolution can cause data loss, security breaches, or financial errors. Claude can explain the conflict — the human makes the call.

The Safety Rule — Never Auto-Commit Claude's Resolution

Safe conflict resolution workflow 
# ALWAYS follow this sequence — never skip the review step

# 1. Get Claude's proposed resolution
cat conflicted-file.py \
  | claude --print "Resolve this merge conflict..." \
  > proposed-resolution.py

# 2. Review the diff between Claude's output and the current file
diff conflicted-file.py proposed-resolution.py

# 3. If satisfied, copy manually (don't pipe directly to the file)
cp proposed-resolution.py conflicted-file.py

# 4. Verify no conflict markers remain
grep -n "<<<<<<<\|=======\|>>>>>>>" conflicted-file.py

# 5. Run tests before staging
python -m pytest tests/unit/test_payments.py -x

# 6. Stage and commit only after tests pass
git add conflicted-file.py
git commit -m "resolve: merge conflict in charge_customer — combine validation + idempotency"

Stash Management

Explain what is in each stash entry 
# List all stashes with their diffs and ask Claude to describe each
function stash-explain() {
  count=$(git stash list | wc -l | tr -d ' ')
  if [ "$count" -eq 0 ]; then
    echo "No stash entries."; return
  fi

  echo "=== Stash Summary ==="
  for i in $(seq 0 $(($count - 1))); do
    ref="stash@{$i}"
    label=$(git stash list | sed -n "$((i+1))p")
    echo "--- $ref: $label"
    git stash show -p "$ref" 2>/dev/null \
      | claude --print "In one sentence: what does this stash contain?
What problem was the developer working on?"
    echo
  done
}

Bisect Assistance

Claude-guided git bisect session 
# Start a bisect session
# Claude helps you know what to test at each step
git bisect start

# At each bisect step, ask Claude what to verify
function bisect-hint() {
  current=$(git log -1 --oneline)
  diff=$(git diff HEAD~1..HEAD -- . ':(exclude)*.lock' | head -200)
  echo "$diff" \
    | claude --print "We are in a git bisect session hunting a regression.
Current commit: $current

Given this diff, what should I test to check if this commit introduced the bug?
Give me 2-3 specific, concrete test steps — terminal commands or user actions."
}

# Then mark good or bad based on your test result

The Complete .gitconfig Aliases Section

Drop this into your ~/.gitconfig to make Claude a first-class citizen of your git workflow. Each alias is a one-liner that composes git commands with Claude:

~/.gitconfig — Claude-powered aliases 
[alias]
  # cc: conventional commit — generate message from staged diff
  cc = "!git diff --staged | claude --print \"Write a conventional commit message (type(scope): description). Output the message only.\""

  # pr-desc: draft PR description from current branch vs main
  pr-desc = "!git diff main...HEAD | claude --print \"Write a PR description with ## Summary, ## Key Changes, ## Testing Notes, ## Breaking Changes sections.\""

  # changelog: CHANGELOG section from a tag range
  # Usage: git changelog v1.2.0 HEAD
  changelog = "!f() { git log --oneline \"$1..$2\" | claude --print \"Generate a CHANGELOG.md section from these commits. Group by: Added, Fixed, Changed. User-facing language.\"; }; f"

  # explain-diff: explain what the last commit actually did
  explain-diff = "!git diff HEAD~1..HEAD | claude --print \"Explain what this commit does in 3 bullet points. Senior engineer reading speed.\""

  # branch-name: suggest branch names for a given task
  # Usage: git branch-name "add pagination to the users list API"
  branch-name = "!f() { claude --print \"Suggest 3 git branch names (kebab-case, feat/fix/refactor prefix) for: $*. Output one per line.\"; }; f"

  # review: semantic review of staged changes
  review = "!git diff --staged | claude --print \"Review for security issues, missing error handling, and code quality. LGTM if clean, list issues otherwise.\""

  # release-notes: generate release notes between two tags
  # Usage: git release-notes v1.2.0 v1.3.0
  release-notes = "!f() { git log --format=\"%h %s\" \"$1..$2\" | claude --print \"Write GitHub release notes for $2. Audience: developers. Sections: What's New, Features, Bug Fixes, Migration Guide.\"; }; f"

The Golden Rule of Claude Workflow Automation
Claude automates the cognitive overhead — the formatting, categorisation, and transcription work that consumes your attention without requiring your judgment. You retain final say on everything. Never configure automation that commits, pushes, or deploys based on Claude's output without a human review step. The value is in the draft, not in the bypass. Your instincts about what a commit message should say, whether a PR description is accurate, and whether a conflict resolution is correct are the signal — Claude is the first draft.

What to Automate vs. What Not to Automate

Task Automate with Claude? Why
Commit message draft Yes — generate, review, then commit Claude sees the full diff; you validate the intent is correctly described
Auto-commit Claude's message No You own the commit history; a wrong message is your name on it
PR description draft Yes — generate, edit, then submit Saves 10–20 minutes; reviewers get more context
Auto-merging conflict resolutions Never Semantic conflicts require human judgment; wrong resolution silently corrupts logic
Changelog entries Yes — generate, edit, then commit 80% of changelog work is structural; Claude handles structure, you handle accuracy
Pre-commit linting (BLOCKER block) Yes — with human override escape hatch High-confidence signals (hardcoded secrets, SQL injection) are worth blocking on
Auto-deploying based on Claude analysis Never Claude can be wrong; deployments need deterministic, human-reviewed signals

Day 2 · Module 4 — Key Takeaways

  • Shell pipelines: claude --print makes Claude a Unix citizen — compose it with grep, jq, awk, and fswatch for automated analysis at any point in your workflow
  • Log analysis: Claude augments APM tools by explaining patterns and intent, not by replacing real-time monitoring — always pre-filter and pre-scope log excerpts before passing them in
  • PR descriptions: git diff main...HEAD | claude --print generates a first draft at the moment the diff is freshest — wire it into a pre-push hook as opt-in via CLAUDE_PR_HELPER=1
  • Linting: Claude fills the semantic gap between static analysis and human review — run Layer 1 (ruff/mypy) first, then Claude, to make pre-commit hooks fast on clean code
  • Merge conflicts: Ask Claude to explain the intent of each side before proposing a resolution — always review the resolved diff and run tests before committing; never auto-merge security, auth, or migration conflicts
  • Git workflow: Six .gitconfig aliases cover the most common manual steps — commit messages, PR descriptions, changelogs, branch naming, stash explanation, and release notes
  • The golden rule: Automate the cognitive overhead, not the decisions — Claude produces the draft; you maintain final authority over every commit, merge, and release
