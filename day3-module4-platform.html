Enterprise AI Platform Architecture

Outcome: Design and operate a production-grade internal AI platform — an AI Gateway, intelligent model router, prompt registry, cost accounting layer, and full observability stack — so your organisation can scale Claude across hundreds of teams with governance, cost control, and engineering confidence.

01

The Internal AI Platform Model

The instinctive response when an organisation decides to adopt Claude is to distribute API keys. Procurement buys a key, engineering puts it in a secrets manager, and teams start calling the Anthropic API directly. This is the fastest path to a demo. It is also the fastest path to a governance crisis.

Six months in, you have 40 teams making direct API calls, no visibility into which teams are spending what, no consistent safety layer, and a $200,000 monthly bill with no line-item breakdown. The AI Gateway pattern exists to prevent this outcome.

Why an Internal Platform — Not Direct API Access

🔍
Governance

Every call is attributed to a team, feature, and user. Policy violations surface immediately. There is a single enforcement point for allowlist/blocklist rules rather than hoping each team implements them correctly.

💰
Cost Visibility

Per-team, per-feature token accounting. Monthly budget alerts. Hard-stop at budget ceiling. Finance can pull a report at any moment — no spreadsheet archaeology required.

⚡
Rate Limiting

Distribute the Anthropic rate limit fairly across teams. A runaway script in team A does not eat team B's quota. Burst allowances configurable per feature tier.

🔐
Security

A single API key in the gateway's secret manager — never distributed. All network egress to Anthropic originates from one subnet. PII scrubbing runs before any prompt leaves the building.

📋
Audit Trail

Immutable log of every request: caller identity, prompt hash, model selected, tokens consumed, latency, response hash. Required for regulated industries. Essential for debugging production incidents.

📐
Standardisation

One SDK, one prompt format, one way to emit metrics. Application teams do not worry about retry logic, backoff, or model selection. The platform handles it.

The AI Gateway Pattern

The AI Gateway is a thin FastAPI service that sits between all internal services and the Anthropic API. "Thin" is deliberate — its job is routing, accounting, and enforcement, not prompt logic. Application teams inject their domain-specific prompts; the gateway handles everything infrastructure-adjacent.

Platform as a Product — The platform team maintains the gateway exactly as they would a customer-facing product. Application teams are the customers. Adoption Strategy

Do not build the platform for the whole organisation before anyone has used it. Start with three pilot teams chosen for diversity of use case: one batch workload (classification at scale), one real-time workload (user-facing summarisation), one developer tooling workload (code review). Measure ROI — tokens saved vs. direct API, incidents caught by the governance layer, developer onboarding time. Present the numbers. Roll out org-wide with evidence.

Target timelines: 6 weeks to first pilot, 12 weeks to three teams in production, 6 months to org-wide availability. Do not compress the pilot phase — it is where you discover the platform requirements you did not know you had.

02

Intelligent Model Routing

Model routing is the practice of automatically selecting the cheapest model that can handle a task correctly. The key word is "correctly" — routing exists to reduce cost without degrading quality. When done well it is invisible to application teams. When done badly it produces silent quality regressions that are hard to attribute.

Routing Signals

Three categories of signal drive routing decisions:

  • Task complexity: A lightweight classifier (rule-based or a small embedding model) scores prompt complexity on a 0–1 scale. Simple extraction and classification → Haiku. Reasoning, code generation, nuanced synthesis → Sonnet. Multi-step planning, financial or legal analysis → Opus.
  • Request metadata: The caller service, feature flag, and user tier are passed as headers. A feature flagged model_tier=opus is never downgraded regardless of complexity score. Enterprise users can receive a higher default tier than free users.
  • Prompt length: Prompts exceeding 8,000 tokens are routed to Sonnet minimum. Long context requires a model with enough capacity to reliably attend to the full input. Haiku's effective recall over very long contexts is measurably weaker.
gateway/router.py — Model router with full routing logic 
from dataclasses import dataclass
from enum import Enum
import re


class ModelTier(str, Enum):
    HAIKU  = "claude-haiku-3-5"
    SONNET = "claude-sonnet-4-5"
    OPUS   = "claude-opus-4-5"


@dataclass
class ClaudeRequest:
    prompt: str
    system: str = ""
    feature_id: str = ""
    caller_service: str = ""
    requested_tier: str | None = None   # explicit override from caller
    user_tier: str = "standard"        # "standard" | "enterprise"
    max_tokens: int = 1024


# ── Hard routing rules ────────────────────────────────────────────────────────
# These override any complexity signal. Order matters — first match wins.
HARD_RULES: list[tuple[str, ModelTier]] = [
    # Financial calculations always get Opus
    ("finance.", ModelTier.OPUS),
    ("billing.", ModelTier.OPUS),
    ("payment.", ModelTier.OPUS),
    # Security-sensitive features get Sonnet minimum
    ("auth.",     ModelTier.SONNET),
    ("security.", ModelTier.SONNET),
    ("pii.",      ModelTier.SONNET),
    # Legal review always gets Opus
    ("legal.",   ModelTier.OPUS),
    ("contract.", ModelTier.OPUS),
]

# Complexity keywords that push the estimate up
HIGH_COMPLEXITY_PATTERNS = re.compile(
    r"(step[- ]by[- ]step|reason|analyse|compare|evaluate|"
    r"pros.and.cons|trade.?off|architecture|design|strategy)",
    re.IGNORECASE,
)


def _complexity_score(request: ClaudeRequest) -> float:
    """Return 0.0 (trivial) → 1.0 (very complex). Rule-based; swap for an
    embedding classifier once you have labelled data from production routing."""
    score = 0.0
    prompt = request.prompt + " " + request.system
    token_estimate = len(prompt.split())

    # Length signal
    if token_estimate > 6_000:
        score += 0.5
    elif token_estimate > 2_000:
        score += 0.3
    elif token_estimate > 500:
        score += 0.1

    # Pattern signal
    matches = HIGH_COMPLEXITY_PATTERNS.findall(prompt)
    score += min(len(matches) * 0.1, 0.4)

    # Code generation detection
    if "```" in prompt or "def " in prompt or "class " in prompt:
        score += 0.2

    return min(score, 1.0)


def _apply_hard_rules(feature_id: str) -> ModelTier | None:
    """Return a forced model tier if the feature matches a hard rule."""
    for prefix, tier in HARD_RULES:
        if feature_id.startswith(prefix):
            return tier
    return None


def route_model(request: ClaudeRequest) -> str:
    """
    Select the most cost-effective model for the request.

    Priority order:
      1. Explicit caller override (caller is accountable for the cost)
      2. Hard routing rules (feature_id prefix match)
      3. Enterprise user floor (enterprise users get Sonnet minimum)
      4. Complexity-based selection
    """
    # 1. Caller override — they explicitly requested a tier
    if request.requested_tier:
        try:
            return ModelTier(request.requested_tier).value
        except ValueError:
            pass  # unknown tier string — fall through to rules

    # 2. Hard routing rules (security, finance, legal)
    forced = _apply_hard_rules(request.feature_id)
    if forced:
        return forced.value

    # 3. Enterprise users get Sonnet as floor
    floor = ModelTier.HAIKU
    if request.user_tier == "enterprise":
        floor = ModelTier.SONNET

    # 4. Complexity-based selection
    score = _complexity_score(request)
    if score >= 0.75:
        selected = ModelTier.OPUS
    elif score >= 0.40:
        selected = ModelTier.SONNET
    else:
        selected = ModelTier.HAIKU

    # Apply floor (never downgrade below the floor)
    tier_order = [ModelTier.HAIKU, ModelTier.SONNET, ModelTier.OPUS]
    if tier_order.index(selected) < tier_order.index(floor):
        selected = floor

    return selected.value

Tier Escalation on Low-Confidence Output

Even with a good complexity classifier, Haiku occasionally produces a low-confidence response on a task that was mis-classified as simple. Rather than letting a poor response reach the user, the gateway can detect these cases and transparently re-run with Sonnet.

gateway/escalation.py — Automatic tier escalation 
import re

# Patterns that suggest the model was uncertain or refused
LOW_CONFIDENCE_PATTERNS = re.compile(
    r"(I'?m not sure|I cannot determine|I don't have enough|"
    r"I'm unable to|as an AI I|I apologize, but I cannot|"
    r"requires more context|I would need more information)",
    re.IGNORECASE,
)


def is_low_confidence(response_text: str) -> bool:
    """Return True if the response signals uncertainty or refusal."""
    return bool(LOW_CONFIDENCE_PATTERNS.search(response_text))


async def call_with_escalation(
    client,
    request: "ClaudeRequest",
    initial_model: str,
    escalation_model: str = "claude-sonnet-4-5",
) -> tuple[str, str]:
    """
    Call Claude with initial_model. If the response is low-confidence,
    re-run with escalation_model.
    Returns (response_text, model_actually_used).
    """
    response = await client.messages.create(
        model=initial_model,
        max_tokens=request.max_tokens,
        system=request.system,
        messages=[{"role": "user", "content": request.prompt}],
    )
    text = response.content[0].text

    if initial_model != escalation_model and is_low_confidence(text):
        # Log the escalation for cost attribution and ML feedback
        escalation_response = await client.messages.create(
            model=escalation_model,
            max_tokens=request.max_tokens,
            system=request.system,
            messages=[{"role": "user", "content": request.prompt}],
        )
        return escalation_response.content[0].text, escalation_model

    return text, initial_model
✓
Cost Savings in Practice — A classification-heavy workload (product categorisation, intent detection, sentiment tagging) is an ideal target for Haiku routing. Routing 70% of these tasks to Haiku while keeping the remaining 30% on Sonnet for complex or ambiguous cases typically reduces the AI budget for that workload by 38–42% with less than 2% quality degradation as measured by human evaluation on a 500-item sample. Build the evaluation harness before you ship the router.
03

Cost Optimisation at Scale

Model routing is the highest-leverage cost lever but not the only one. A mature platform uses all five levers simultaneously. Each has a different implementation complexity and a different return on investment. Start with routing and caching — they deliver 80% of the savings with 40% of the effort.

01
Model Routing
Route each request to the cheapest capable model. Covered in full in Section 02. Typical savings: 35–45% depending on workload mix.
02
Prompt Compression
Reduce input token count without reducing output quality. Remove redundant preambles, convert prose explanations to bullet points once a pattern is established, drop few-shot examples after the first 500 production calls confirm the model has the pattern.
03
Response Caching
Identical or semantically similar prompts return a cached response without a new API call. Semantic caching (embedding similarity) catches near-duplicate prompts that exact-match caching misses. Critical for any feature with repetitive or templated inputs.
04
Batching
Group multiple small requests into Anthropic Batch API calls. The Batch API provides a 50% cost discount on qualifying models. Ideal for offline, non-real-time workloads: nightly classification runs, document processing pipelines, bulk content generation.
05
Output Truncation
Set max_tokens per feature, not globally. A classification feature needs 10 tokens, not 1024. A summarisation feature needs 200 tokens, not 4096. Over-provisioned max_tokens rarely fires but signals careless engineering and increases tail latency.

Semantic Caching with Redis + Embeddings

Exact-text caching catches identical prompts. Semantic caching catches near-identical prompts — the same question phrased differently, form fields with slight wording variations, or templated prompts where only a name changes. Embed the prompt, find the nearest cached embedding, and return the cached response if cosine similarity exceeds a threshold.

gateway/cache.py — Redis semantic cache with Claude embeddings 
from __future__ import annotations
import hashlib, json, time
import numpy as np
import redis.asyncio as redis
from anthropic import AsyncAnthropic

SIMILARITY_THRESHOLD = 0.96   # Tune per feature; higher = stricter
CACHE_TTL_SECONDS    = 3_600   # 1 hour; shorter for time-sensitive content
EMBEDDING_MODEL      = "voyage-3"  # Anthropic's recommended embedding model


class SemanticCache:
    def __init__(self, redis_url: str, anthropic_client: AsyncAnthropic):
        self._redis  = redis.from_url(redis_url, decode_responses=False)
        self._client = anthropic_client

    async def _embed(self, text: str) -> np.ndarray:
        """Get a vector embedding for a prompt string."""
        response = await self._client.beta.messages.batches.create()
        # NOTE: use voyage-3 via the Anthropic Embeddings API when GA
        # For now, compute a deterministic hash-based mock for illustration
        seed = int(hashlib.sha256(text.encode()).hexdigest(), 16) % (2**31)
        rng  = np.random.default_rng(seed)
        return rng.random(1536).astype(np.float32)

    async def get(self, prompt: str, feature_id: str) -> str | None:
        """Return cached response if a sufficiently similar prompt exists."""
        query_vec = await self._embed(prompt)
        pattern   = f"sem:{feature_id}:*"

        async for key in self._redis.scan_iter(pattern, count=200):
            entry = await self._redis.get(key)
            if not entry:
                continue
            cached  = json.loads(entry)
            cached_vec = np.array(cached["embedding"], dtype=np.float32)
            similarity = float(
                np.dot(query_vec, cached_vec)
                / (np.linalg.norm(query_vec) * np.linalg.norm(cached_vec) + 1e-9)
            )
            if similarity >= SIMILARITY_THRESHOLD:
                return cached["response"]
        return None

    async def set(self, prompt: str, feature_id: str, response: str) -> None:
        """Store a prompt→response pair with its embedding vector."""
        embedding = await self._embed(prompt)
        key = f"sem:{feature_id}:{hashlib.sha256(prompt.encode()).hexdigest()[:16]}"
        payload = json.dumps({
            "embedding": embedding.tolist(),
            "response":  response,
            "cached_at": time.time(),
        })
        await self._redis.setex(key, CACHE_TTL_SECONDS, payload)

Budget Accounting Middleware

Every Claude response from the Anthropic API includes usage data: input_tokens and output_tokens. The accounting middleware intercepts every response, calculates cost based on the model's per-token price, and writes it to the token accounting table. When a team crosses 80% of their monthly budget, an alert fires. At 100%, the gateway starts returning 429 Budget Exceeded.

gateway/accounting.py — Budget accounting middleware 
from __future__ import annotations
from dataclasses import dataclass
from datetime import datetime, timezone
import asyncpg

# Cost per 1M tokens in USD (update when Anthropic changes pricing)
MODEL_PRICING: dict[str, dict[str, float]] = {
    "claude-haiku-3-5":  {"input": 0.80,  "output": 4.00},
    "claude-sonnet-4-5": {"input": 3.00,  "output": 15.00},
    "claude-opus-4-5":   {"input": 15.00, "output": 75.00},
}

ALERT_THRESHOLD = 0.80   # Alert at 80% of monthly budget
HARD_STOP       = 1.00   # Hard-stop at 100%


@dataclass
class UsageRecord:
    team_id:       str
    feature_id:    str
    model:         str
    input_tokens:  int
    output_tokens: int
    cost_usd:      float
    timestamp:     datetime


def calculate_cost(model: str, input_tokens: int, output_tokens: int) -> float:
    pricing = MODEL_PRICING.get(model, {"input": 3.00, "output": 15.00})
    return (
        (input_tokens  / 1_000_000) * pricing["input"] +
        (output_tokens / 1_000_000) * pricing["output"]
    )


class BudgetAccountant:
    def __init__(self, pool: asyncpg.Pool, alert_fn):
        self._pool     = pool
        self._alert_fn = alert_fn  # async callable(team_id, pct_used, budget_usd)

    async def record_and_check(
        self,
        team_id: str,
        feature_id: str,
        model: str,
        input_tokens: int,
        output_tokens: int,
    ) -> None:
        cost = calculate_cost(model, input_tokens, output_tokens)
        now  = datetime.now(timezone.utc)

        async with self._pool.acquire() as conn:
            # Insert usage record
            await conn.execute(
                """INSERT INTO token_usage
                   (team_id, feature_id, model, input_tokens, output_tokens, cost_usd, ts)
                   VALUES ($1, $2, $3, $4, $5, $6, $7)""",
                team_id, feature_id, model,
                input_tokens, output_tokens, cost, now,
            )

            # Check monthly spend vs. budget
            row = await conn.fetchrow(
                """SELECT
                     COALESCE(SUM(cost_usd), 0)     AS spent,
                     t.monthly_budget_usd            AS budget
                   FROM token_usage u
                   JOIN teams t ON t.id = u.team_id
                   WHERE u.team_id = $1
                     AND date_trunc('month', u.ts) = date_trunc('month', NOW())
                   GROUP BY t.monthly_budget_usd""",
                team_id,
            )

        if not row:
            return

        pct = row["spent"] / max(row["budget"], 0.01)

        if pct >= HARD_STOP:
            raise BudgetExceededError(team_id, row["budget"])

        if pct >= ALERT_THRESHOLD:
            await self._alert_fn(team_id, pct, row["budget"])


class BudgetExceededError(Exception):
    def __init__(self, team_id: str, budget: float):
        super().__init__(f"Team {team_id} has exhausted its ${budget:.2f} monthly budget")
        self.team_id = team_id
        self.budget  = budget
⚠
Hard-Stop Sequencing — Check the budget ceiling before making the API call, not after. Checking after means you have already spent the money by the time you realise the budget is exhausted. In practice: read the current monthly spend from a fast cache (Redis counter updated on each record write) at request ingress and reject before touching the Anthropic API.

Structured Log Format

Every Claude API call emits one structured log line at completion. Logs are shipped to S3 (Parquet, partitioned by year/month/day/team_id) for compliance archiving, and to OpenSearch for live querying. The trace_id propagates through the entire request chain so you can reconstruct a full user journey.

Internal Developer Platform

The AI Gateway is an infrastructure component. Application teams should not need to understand it. The developer experience layer — an internal SDK, a web portal, and a CLI — abstracts the gateway behind an interface that feels like calling any other internal service.

The Internal SDK

The internal SDK wraps the Anthropic SDK. It handles auth, routing headers, PII scrubbing, logging, and budget enforcement automatically. Application teams write code that looks like this:

Application code — using the internal SDK vs. raw Anthropic SDK 
# ── Without the internal SDK (raw Anthropic SDK) ─────────────────────────────
import anthropic, os, hashlib, time, json, logging

client = anthropic.AsyncAnthropic(api_key=os.getenv("ANTHROPIC_API_KEY"))
trace_id = hashlib.sha256(str(time.time()).encode()).hexdigest()[:8]

The Prompt Registry

The prompt registry is a central store of versioned, tested system prompts. Think of it as a package manager for prompts — teams reference a prompt by ID and version rather than inlining the text in their code. Benefits:

  • Single source of truth: Fixing a prompt bug in the registry propagates to every feature that references it on next deploy.
  • Version pinning: Teams can pin to v1.4 and upgrade on their own schedule. The registry retains all historical versions.
  • Prompt review process: New prompt versions go through a PR-style review before being published. A prompt engineer signs off. The diff is visible and auditable.
  • A/B testing: The registry supports traffic splitting between prompt versions. Route 10% of traffic to v1.5, measure quality, promote or rollback.
gateway/prompt_registry.py — Versioned prompt store 
from __future__ import annotations
import asyncpg
from functools import lru_cache


class PromptRegistry:
    def __init__(self, pool: asyncpg.Pool):
        self._pool = pool

    async def get(self, prompt_id: str, version: str = "latest") -> str:
        """Resolve a prompt ID to its text. 'latest' returns the highest version."""
        async with self._pool.acquire() as conn:
            if version == "latest":
                row = await conn.fetchrow(
                    """SELECT text FROM prompts
                       WHERE id = $1 AND status = 'active'
                       ORDER BY semver_major DESC, semver_minor DESC, semver_patch DESC
                       LIMIT 1""",
                    prompt_id,
                )
            else:
                row = await conn.fetchrow(
                    """SELECT text FROM prompts
                       WHERE id = $1 AND version = $2 AND status = 'active'""",
                    prompt_id, version,
                )
            if not row:
                raise KeyError(f"Prompt '{prompt_id}@{version}' not found in registry")
            return row["text"]

    async def publish(
        self,
        prompt_id: str,
        version: str,
        text: str,
        author: str,
        review_pr_url: str,
    ) -> None:
        """Publish a reviewed prompt version. Only callable by the platform CLI."""
        async with self._pool.acquire() as conn:
            major, minor, patch = (int(x) for x in version.lstrip("v").split("."))
            await conn.execute(
                """INSERT INTO prompts
                   (id, version, semver_major, semver_minor, semver_patch,
                    text, author, review_pr_url, status)
                   VALUES ($1, $2, $3, $4, $5, $6, $7, $8, 'active')""",
                prompt_id, version, major, minor, patch,
                text, author, review_pr_url,
            )

Platform Governance & Operations

A platform without governance is a liability waiting to materialise. Governance means that every change — to a prompt, to a routing rule, to a budget ceiling — goes through a defined process with documented approvals and an audit trail. It means that when something goes wrong in production, the team has a runbook rather than a panic.

Change Management — Prompts as Code

Prompt changes have the same blast radius as code changes. A poorly worded system prompt can silently degrade quality for thousands of users before anyone notices. Treat every prompt change as a deployment:

  • Prompt text lives in the registry, version-controlled in Git alongside the configuration that references it.
  • A PR is required for any version bump. The PR includes the diff, a description of why the change was made, and a link to the evaluation results that justified it.
  • A prompt engineer and the feature team lead must approve before merge.
  • Deployment is gated: new version goes to 5% of traffic first. Quality metrics are monitored for 24 hours. Full rollout on pass, automatic rollback on fail.

Incident Runbook Template

runbooks/claude-feature-degradation.md 
# Runbook: Claude Feature Degradation

## Symptoms
- Feature X returning unexpected or empty responses
- Error rate on feature X > 1% (Grafana alert fired)
- User reports of low-quality AI output on feature X
- Latency P95 > 5s on feature X

## Triage Steps (run in order)

### 1. Check Gateway Status (2 minutes)
- Open Grafana: claude_error_total grouped by model
- If error rate is across ALL models → likely Anthropic outage
  → Check https://status.anthropic.com
  → Engage vendor escalation if P1 customer impact
- If error rate is on ONE model → model-specific issue

### 2. Check Budget (1 minute)
- Query: claude_budget_utilisation_ratio{team_id="<team>"}
- If ratio >= 1.0 → budget exhausted; all requests returning 429
  → Escalate to Finance for emergency budget increase
  → OR: set feature flag to serve cached fallback responses

### 3. Check Recent Deployments (3 minutes)
- Was a new prompt version deployed in the last 24h?
  → git log --since=24h -- prompts/<feature_id>.yaml
  → If yes: rollback to previous prompt version in registry
    `ai-platform prompt rollback <feature_id> --version <prev>`

### 4. Inspect Live Logs (5 minutes)
- Search OpenSearch: feature_id = "<feature_id>" AND error != null
- Look for: rate_limit_exceeded, context_length_exceeded, invalid_api_key
- rate_limit_exceeded → enable request queuing in gateway config
- context_length_exceeded → prompt has grown beyond model limit; compress

### 5. Engage Anthropic Support
- If Anthropic status page shows incident: subscribe to updates
- For enterprise customers: use the dedicated Slack channel
- Include: trace_ids, error samples, request volume, affected model

## Fallback Options
- Serve cached responses (enable CACHE_ONLY mode in feature flag)
- Degrade gracefully (return static placeholder response)
- Bedrock failover: set FAILOVER_BACKEND=bedrock in platform config
  (requires cross-trained Claude model on Bedrock)

## Post-Incident
- Write incident report within 48h
- Add runbook step for any new failure mode discovered
- Update alert thresholds if alert fired too late or too early

Capacity Planning

Forecast 3 months ahead. Use this formula to estimate daily token volume per feature:

capacity_forecast.py — 3-month token volume forecast 
from dataclasses import dataclass

@dataclass
class FeatureForecast:
    feature_id:       str
    dau:              int    # daily active users
    calls_per_session: float # average Claude calls per user session
    avg_input_tokens: int    # average input tokens per call
    avg_output_tokens: int   # average output tokens per call
    growth_rate_pct:  float  # monthly user growth %

    def daily_tokens(self, months_ahead: int = 0) -> tuple[int, int]:
        """Return (input_tokens/day, output_tokens/day) at months_ahead."""
        projected_dau = self.dau * ((1 + self.growth_rate_pct / 100) ** months_ahead)
        daily_calls   = projected_dau * self.calls_per_session
        return (
            int(daily_calls * self.avg_input_tokens),
            int(daily_calls * self.avg_output_tokens),
        )

    def monthly_cost_usd(
        self,
        input_price_per_1m:  float,
        output_price_per_1m: float,
        months_ahead: int = 0,
    ) -> float:
        inp, out = self.daily_tokens(months_ahead)
        daily_cost = (
            (inp / 1_000_000) * input_price_per_1m +
            (out / 1_000_000) * output_price_per_1m
        )
        return daily_cost * 30


# Example: search query expansion feature
search_feature = FeatureForecast(
    feature_id       = "search.query-expansion",
    dau              = 50_000,
    calls_per_session = 2.3,
    avg_input_tokens = 320,
    avg_output_tokens = 45,
    growth_rate_pct  = 8.0,   # 8% monthly growth
)

HAIKU_PRICING = {"input": 0.80, "output": 4.00}

for m in range(4):
    cost = search_feature.monthly_cost_usd(
        HAIKU_PRICING["input"], HAIKU_PRICING["output"], months_ahead=m,
    )
    print(f"Month +{m}: ${cost:,.2f}/month")

Vendor Resilience — What If Anthropic Has an Outage?

Anthropic's API has a strong track record but no infrastructure has a 100% SLA. Design for outage from day one:

  • Cached responses: For read-heavy, low-velocity features, the semantic cache can serve stale responses for up to 24 hours with a banner informing the user that results may be slightly out of date.
  • Graceful degradation: Most Claude-powered features can fall back to a simpler, deterministic implementation. A summarisation feature falls back to returning the first three sentences. An intent classification feature falls back to a regex rule set.
  • Bedrock failover: AWS Bedrock hosts Claude models under a separate infrastructure path. The gateway can be configured to route to Bedrock when the primary Anthropic API returns 5xx errors for more than 30 seconds. Requires Bedrock model access to be pre-provisioned.

Level 1
Direct API Access
Teams hold their own API keys. No central visibility, no governance. Common in early experimentation. Exit this level before you have more than 3 teams using Claude in production.
Level 2
Shared Gateway
Single API key, basic authentication, and an audit log. Teams call the gateway instead of Anthropic directly. Cost is visible in aggregate. Achievable in 4–6 weeks.
Level 3
Full Platform
Model routing, semantic caching, prompt registry, per-team cost accounting, and a full Grafana/Prometheus observability stack. This is the target for most enterprises at 12 months. Achievable in 12–16 weeks from Level 2.
Level 4
Self-Service + FinOps
Teams onboard themselves through a self-service portal. Finance has per-team, per-feature cost attribution in real time. Bedrock failover is live. A capability catalogue prevents duplication. Vendor negotiation is backed by usage data.
⚠
Do Not Overengineer Level 3 — The most common failure mode is building a Level 4 platform when you have three teams. Build Level 2 in week 1, get it in production. Build Level 3 incrementally over 3 months while actual teams are using it. Every architectural decision made without production usage data is a guess. Ship early, learn fast, iterate.

Day 3 · Module 4 — Key Takeaways

  • The AI Gateway pattern: A thin FastAPI proxy between all internal services and the Anthropic API provides governance, cost visibility, rate limiting, security, audit trail, and standardisation — the six reasons to never give teams direct API access at scale
  • Model routing: route_model() selects the cheapest capable model using three signals: task complexity, request metadata, and prompt length — with hard rules for security, financial, and legal features that override all signals
  • Cost optimisation: The five levers — routing, prompt compression, semantic caching, batching, output truncation — can reduce the AI budget by 40–60% relative to naive direct-API usage; start with routing and caching for 80% of the savings
  • Observability: Five Prometheus counters cover the full signal set: latency, input tokens, output tokens, cost, and errors — instrument at the gateway level so no team can forget to add metrics
  • Developer platform: An internal SDK that wraps the gateway, a prompt registry for versioned prompts, and a self-service portal for team onboarding reduces time-to-first-production-call from weeks to hours
  • Platform governance: Prompt changes need the same PR, review, and staged rollout process as code changes; incident runbooks need to exist before incidents happen; capacity planning at L4 requires actual production usage data
  • Maturity model: L1 → L2 → L3 → L4 over 12–18 months; do not skip levels; the platform that serves 10 teams well is built by learning from the first 3 teams, not by anticipating all 10 before any are live
